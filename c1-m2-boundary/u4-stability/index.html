Unit 4·
同一个问题, 两个答案?

AI 的第三个死穴: 稳定性. 对健康产品来说, 这是最容易出事的那个.

我用 AI 给用户回答"经期可以跑步吗?"

第一次说"可以, 适度慢跑"

第二次说"建议休息, 避免运动"

—— 这不是闹呢吗?? 用户信哪个?

10 步学完, 你会:
  • 理解为什么 AI 同一个问题会给出不同答案
  • 判断哪些场景能容忍不一致、哪些绝对不行
  • 掌握 4 种工程手段让 AI 输出变稳定
  • 给 Vitamin 健康问答设计一套"稳定性方案"

(1)

先看一个真实场景 — 你品, 你细品

同一个用户, 问 Vitamin 同一个问题. AI 的回答:

第 1 次
经期可以进行适度的跑步运动。建议选择慢跑，控制在 20-30 分钟以内。运动有助于缓解经期不适。如果感到不舒服，请立即停止。
第 2 次
经期建议避免跑步等剧烈运动。可以选择散步或瑜伽代替。跑步可能加重子宫下坠感，增加经血量。建议咨询医生。

问题在哪? 第一次说"可以跑", 第二次说"别跑". 两个答案都像模像样, 都有理有据. 但它们互相矛盾.

如果你是用户, 你信哪个? 如果你是 PM, 你怎么向用户解释?

这是 bug 吗? 还是 AI 本来就这样? ↓
(2)

两个常见错猜测 — 都不对

✗

猜测 A: "AI 应该像计算器, 同输入同输出"

输入 2+3 永远得 5. AI 也该这样.

为啥不对: AI 不是计算器 — 它是个概率采样器. 每次输出是从概率分布里"抽"出来的, 天然带随机性. 这是设计, 不是 bug.
✗

猜测 B: "不一致说明 AI 不靠谱, 别用了"

每次答案都不同, 那还有什么用?

为啥不对: 写诗、起名、生成文案 — 你希望它每次不同. 不一致是 feature 还是 bug, 取决于场景.
真相

随机性是设计进去的, 不是意外.

但对健康产品, 你必须把它控制住.

→ M1.1 U5 讲过 Temperature 决定"严谨 vs 放飞". 这节课我们从产品角度看: 不控制 = 出事.

那随机性到底从哪来? 为什么 T=0 也不能 100% 一致? ↓
(3)

不一致的四个技术根因 — PM 需要知道

你不需要能修这些问题, 但你需要知道它们存在, 才能和工程师对话.

🎲

1. Temperature 采样

Temperature > 0 时, 模型从概率分布里"抽签". 同样的概率表, 每次抽到的字可能不同. 这是最大的随机性来源, 也是最容易控制的.

🔢

2. 浮点运算精度

GPU 做数学时, (a+b)+c 和 a+(b+c) 可能差一点点. 这种微小差异在 1400 亿参数里传播, 最终可能导致不同的 token 被选中.

🔀

3. MoE 路由 (专家混合)

GPT-4 等模型用 Mixture of Experts 架构. 你的请求和别人的请求被打包成一批处理, 不同的"专家"可能被分配到不同的子任务, 导致结果微妙不同.

📝

4. Prompt 敏感性

措辞微调 → 概率分布移动. "经期可以跑步吗" vs "月经期间适合跑步运动吗" — 对你来说一样, 对模型来说是不同的输入.

PM 记住: 即使设了 T=0 + seed, 也不能保证 100% 一致. 因为 #2 和 #3 在 API 层面你控制不了. 所以光靠参数不够, 还需要产品层面的策略.
知道了根因, 下一步: 什么时候不一致是 OK 的? ↓
(4)

不一致: 什么时候是 Feature, 什么时候是 Bug

一条光谱, 从左到右风险递增:

OK 区
写诗、起名、生成文案、头脑风暴、聊天闲聊
灰色区
客服回复、产品推荐、邮件草稿、内容摘要
危险区
健康建议、法律意见、财务计算、医疗诊断
关键判断标准:
  1. 用户会根据答案做决定吗? 写诗不影响决定, 健康建议直接影响行为.
  2. 矛盾答案的后果多严重? 文案换个说法没事, 用药建议前后不一致可能出人命.
  3. 用户有能力自行判断吗? 程序员看 AI 写的代码能判断对错, 普通用户看健康建议不一定能.

Vitamin 的定位: 女性健康 AI → 坐在光谱的最右边. 稳定性不是"nice to have", 是"must have".

那怎么量化"有多稳定"? ↓
(5)

怎么量化稳定性 — 跑 N 次, 比一比

最简单的方法: 同一个 prompt, 跑 5 次 (或 10 次), 看输出有多像.

Consistency Score 的计算思路:
  1. 取 N 次输出 (通常 N = 5~10)
  2. 两两比较语义相似度 (可以用 embedding cosine similarity)
  3. 取平均值, 映射到 0~100% 分数

或者更简单粗暴: 人工看 5 次输出, 核心结论是否一致? 是 → 通过. 不是 → 不上线.

PM 实用版: 上线前, 拿 top 50 个高频问题, 每个跑 5 次. 标记哪些问题的答案出现了矛盾. 这些矛盾问题 = 你的"高风险清单".
测出来不稳定的, 怎么修? 4 种工程手段 ↓
(6)

4 种工程手段 — 让 AI 输出变稳定

你不用写代码, 但你得知道这些选项, 才能跟工程师开会时说到点子上.

🧊

T=0 + Seed

Temperature 设为 0 (贪心解码), 加上 seed 参数. 最简单的一招, 能消除大部分随机性. 但不能保证 100%.

💾

Response Caching

高频问题只调一次 API, 缓存答案. 用户再问同样的问题, 直接返回缓存. 100% 一致, 零 API 成本.

📋

后处理规则

AI 输出后, 过一遍规则引擎. 比如: 健康建议必须包含"请咨询医生"声明, 不允许出现"必须/一定"等绝对词.

✅

Golden Answer 审核

高风险问题由人工预审, 存一份"标准答案". AI 输出与标准答案比对, 偏差太大则触发 fallback.

面试金句:

"稳定性不是 AI 的默认值 — 需要在 API 参数 + 缓存 + 后处理 + 人工审核四层叠加来保障. 层数取决于场景的风险等级."

第 4 种 "Golden Answer" 展开讲 ↓
(7)

"Golden Answer" 模式 — 健康产品的标配

对于高风险问题, 与其让 AI 每次现场生成, 不如: 人工审核一个"标准答案", AI 只负责"匹配问题 → 返回答案".

这个模式的好处:
  • 高频问题 100% 一致 (直接返回缓存的人工审核答案)
  • 低频/长尾问题仍有 AI 灵活回答, 但加 disclaimer
  • 可以定期更新 Golden Answer (跟着医学指南走)
  • 审核成本可控: top 50 问题覆盖 80% 用户流量

(8)

动手试: Stability Tester

调 Temperature, 点"运行 5 次", 看同一个健康问题 ("经期可以跑步吗?") 的 5 次回答有多不同.

Prompt: "经期可以跑步吗?"
一致性评分 T=0.7 (默认): 语气/内容开始分化.
试试看:
  • 把 T 拉到 0 → 5 次输出完全一模一样
  • T = 1.0 → 开始出现矛盾 ("可以跑" vs "别跑")
  • T = 1.5 → 完全混乱, 有的还提到了"月球周期" ...

→ 这就是为什么 Vitamin 的健康问答必须用 T=0.

回到 Vitamin — 稳定性方案怎么设计 ↓
(9)

回到 Vitamin — 健康问答的稳定性方案

Vitamin 有 5 个身体状态 (经期、排卵期、黄体期...), 每个状态都有用户会问的健康问题. 方案:

✓ 稳定性方案 (四层防护)

  • API 参数: T=0, seed=42, max_tokens 限定
  • Top 50 高频健康问题: Golden Answer 缓存
  • 后处理: 自动添加"仅供参考, 请咨询医生"声明
  • 矛盾检测: 同一会话内前后答案一致性校验
  • 每月更新: 跟着最新医学指南刷新 Golden Answer
  • 上线前: 每个问题跑 5 次, 一致性 < 90% 的不上线

✗ 不能犯的错

  • 用默认 T=0.7 回答健康问题 (每次答案不同)
  • 让 AI 自由发挥回答用药问题 (没有 Golden Answer)
  • 不做后处理就直接展示 AI 输出 (可能出现绝对化表述)
  • 忽略长尾问题 (没预审的问题也需要 disclaimer)
  • 不做上线前的稳定性测试 (跑 5 次太简单了, 不做不行)
  • 测一次就不管了 (模型更新后必须重新测)
面试金句, 抄走:

"做健康类 AI 产品, 我会用四层策略保障输出稳定性: 参数层 (T=0 + seed), 缓存层 (高频 Golden Answer), 规则层 (后处理校验), 流程层 (上线前 N 次一致性测试). 这四层叠加, 才能对用户负责."

最后: 一句话规则 + 作业 ↓
(10)

一句话规则 + 今晚作业

一句话:

稳定性不是 AI 的默认值 ——
它是你必须用参数 + 缓存 + 规则 + 测试四层叠加争取的.

场景越高风险, 需要叠的层越多. 健康产品 = 叠满.

课后作业 (今晚 30 分钟):
  1. 打开 ChatGPT / 豆包, 问同一个健康问题 5 次 (比如"感冒可以洗澡吗"), 截图对比. 有没有出现矛盾?
  2. 列出 Vitamin 的 10 个高频健康问题, 标记每个问题的风险等级 (OK / 灰色 / 危险). 危险级别的就是需要 Golden Answer 的.
  3. 用本课的"四层防护"框架, 给 Vitamin 写一份 200 字的"输出稳定性策略" — 可以直接用在 PRD 里.
  4. 思考: Vitamin 的"情绪安抚"功能 (比如"今天心情不好") 需要高稳定性吗? 还是适度随机反而更好? 写 100 字理由.

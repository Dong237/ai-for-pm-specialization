(1)

平均值在骗你

10 个用户的响应时间:

0.5s, 0.8s, 1.0s, 1.2s, 1.5s, 1.8s, 2.0s, 2.5s, 3.0s, 12.0s

平均值 = 2.6 秒. 看起来还行?

但最后那个用户等了 12 秒! 她已经关掉 App 走了.

平均值的问题: 一个极端值就能把平均拉高或拉低. 它没法告诉你 "最差的那批用户体验有多差".

API 延迟分布是 "长尾" 的: 大部分很快, 少部分很慢 — 而且慢的那些可能慢得离谱.

什么是百分位数 ↓
(2)

P50, P95, P99 — 三个百分位数

定义

P50 (中位数): 一半用户的体验比这快, 一半比这慢.

P95: 95% 的用户体验比这快. 只有 5% 更慢.

P99: 99% 的用户体验比这快. 最差的 1%.

同样的数据:

  • 平均值: 2.6s (被极端值拉高了)
  • P50: 1.5s — "一个正常用户的真实感受"
  • P95: 3.0s — "最慢的那 5% 有多慢"
  • P99: 12.0s — "灾难级体验"

P50 看 "正常体验", P95 看 "最差体验". 两个一起看才是完整画面.

为什么 AI 特别慢 ↓
(4)

AI 延迟的四个来源

  1. Token 生成: LLM 是一个字一个字生成的. 回复越长, 越慢.
  2. 队列等待: 高峰期 API 请求排队. P95 飙升但 P50 不变 = 队列问题.
  3. 网络延迟: 到 API 服务器的往返时间, 海外 API 尤其明显.
  4. 预处理: RAG 检索、context 拼接等, 在 LLM 开始生成前的准备工作.

为什么 P95 和 P50 差很多? 因为 P95 的飙升往往来自队列等待: 大部分请求秒回, 但偶尔赶上排队就等 10 秒.

Streaming 缓解感知延迟 ↓
(5)

Streaming: 真实延迟没变, 感知延迟大降

ChatGPT 为什么 "一个字一个字蹦出来"? 因为 Streaming:

不用 Streaming: 用户等 5 秒 → 突然出来一大段文字. 感知: "卡了 5 秒".

用 Streaming: 0.3 秒后第一个字出现, 然后逐字流出. 总时间还是 5 秒, 但感知: "立刻就有反应".

Streaming 不改变真实延迟, 但改变了两个关键指标:

  • Time to First Token (TTFT): 第一个字出来的时间. 用 Streaming 后 TTFT 可以从 5s 降到 0.3s.
  • 用户感知: 用户看到 "AI 在打字" → 愿意等. 什么都不显示 → 以为卡了.
怎么降低真实延迟 ↓
(6)

降低延迟的四个手段

  1. 用 Streaming: TTFT 从 3-5s 降到 < 0.5s. 必做.
  2. 缩短 Prompt: 输入 token 越多, 预处理越慢. 精简 system prompt.
  3. 缓存: 相似问题用缓存结果, 不重新调 API. Prompt Caching 可省 60%+ 时间.
  4. 就近部署: 如果用户在中国, 用国内 API (DeepSeek / 百炼), 不用美西 API.

效果最大的通常是 1 + 3: Streaming 解决感知问题, 缓存解决成本 + 延迟双重问题.

SLO 怎么设 ↓
(7)

设定 Latency SLO

SLO (Service Level Objective) = 你对自己的承诺: "我的延迟不会超过 X".

推荐做法: 双阈值 SLO

  • P50 SLO: ≤ 2s — 保证正常用户体验流畅
  • P95 SLO: ≤ 5s — 保证最差的 5% 也不至于灾难

只设一个 SLO 是不够的. P50 好但 P95 差 = 大部分人体验好, 但有人天天被坑.

监控: P50 升了 = 系统整体变慢; P95 升了但 P50 没动 = 有一小部分请求被阻塞了 (查队列/网络).

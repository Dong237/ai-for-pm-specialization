(1)

没有 try/except 时会怎样?

你的脚本跑到第 47 条数据时, 网络突然断了:

# 没有错误处理的脚本 for i in range(100): answer = ask_ai(questions[i]) # 第 47 次, 网络断了 print(answer) # Traceback (most recent call last): # ... # openai.APIConnectionError: Connection error. # 程序崩溃, 前面 46 条的结果全丢了

问题: 没有 try/except, 一个错误就让整个脚本崩溃. 100 条数据, 跑了 46 条就全废了.

try/except 就是解决这个问题的 ↓
(2)

try/except = "试一下, 出错了怎么办"

一句话理解: try 里放可能出错的代码, except 里放出错后的应急方案.

类比

就像你去餐厅点餐:

try: "我要一个牛排" (正常操作)

except: "如果牛排没了, 给我换个鸡排" (Plan B)

没有 except = 餐厅说 "没有", 你就饿着走了. 有 except = 总能吃上饭.

# 最简单的 try/except try: answer = ask_ai("经期可以运动吗?") print(answer) except Exception as e: print(f"出错了: {e}") print("跳过这条, 继续下一条")

5 种常见 API 错误一览

错误类型HTTP 码原因频率
AuthenticationError401Key 无效/过期偶尔
RateLimitError429调用太快, 超过限频常见
APIConnectionError--网络问题偶尔
BadRequestError400参数错误/token 超限偶尔

PM 视角: 429 (限频) 是你跑批量脚本时最常遇到的. 500 (服务器错误) 不是你的锅, 等一等重试就好.

来一个一个看怎么处理 ↓
(4)

401 AuthenticationError -- Key 无效

from openai import AuthenticationError try: answer = ask_ai(question) except AuthenticationError: print("API Key 无效! 请检查:") print("1. .env 里的 Key 是否正确") print("2. Key 是否已过期/被 Revoke") print("3. 账户余额是否为 0") # 这种错误不需要重试, 直接停下来修 Key exit(1)
下一个: 最常见的 429 限频 ↓
(5)

429 RateLimitError -- 调太快了

API 有调用频率限制. 超了就返回 429. 解决方案: 等一下再重试.

import time from openai import RateLimitError def ask_ai_safe(question, max_retries=3): for attempt in range(max_retries): try: return ask_ai(question) except RateLimitError: wait = 2 ** attempt # 指数退避: 1s, 2s, 4s print(f"限频了, 等 {wait}s 后重试...") time.sleep(wait) return "[ERROR] 重试 3 次仍失败"

指数退避 (Exponential Backoff): 第一次等 1 秒, 第二次等 2 秒, 第三次等 4 秒. 越等越久, 给 API 喘息时间. 这是业界标准做法.

继续看网络错误和服务器错误 ↓
(6)

网络错误 + 服务器错误 -- 等一等重试

from openai import APIConnectionError, InternalServerError try: answer = ask_ai(question) except APIConnectionError: print("网络断了, 检查 WiFi 或 VPN") # 等 5 秒后重试 time.sleep(5) except InternalServerError: print("OpenAI 服务器出问题了, 等 10s 重试") time.sleep(10) # 不是你的锅, 过一会就好
把所有错误类型合在一起, 写一个完整的函数 ↓
(7)

完整版: 带错误处理的 ask_ai()

把所有错误处理合在一个函数里, 以后你直接用这个:

import time from openai import ( OpenAI, AuthenticationError, RateLimitError, APIConnectionError, BadRequestError, InternalServerError ) from dotenv import load_dotenv load_dotenv() client = OpenAI() def ask_ai(question, system_prompt="你是 Vitamin 健康助手.", max_retries=3): for attempt in range(max_retries): try: resp = client.chat.completions.create( model="gpt-4o-mini", messages=[ {"role": "system", "content": system_prompt}, {"role": "user", "content": question} ], temperature=0.3, max_tokens=500 ) return resp.choices[0].message.content except AuthenticationError: print("Key 无效, 停止运行.") exit(1) except RateLimitError: wait = 2 ** attempt print(f"限频, 等 {wait}s...") time.sleep(wait) except APIConnectionError: print("网络错误, 等 5s...") time.sleep(5) except (BadRequestError, InternalServerError) as e: print(f"API 错误: {e}") time.sleep(3) return "[ERROR] 重试失败"

Vitamin 案例 -- 健壮的批量脚本

Vitamin 跑 100 条用例时, 错误处理是生命线:

✓ 带错误处理的效果

  • 第 47 条遇到 429 → 等 2s 重试, 成功
  • 第 63 条网络断了 → 等 5s, 恢复, 继续
  • 第 89 条输入太长 → 跳过, 记录 "[ERROR]"
  • 100 条跑完, 97 条成功, 3 条记录为错误
  • 没有浪费前面的劳动成果

✗ 没有错误处理

  • 第 47 条出错 → 程序崩溃
  • 前 46 条结果全丢了
  • 你要从头再跑一遍
  • 可能同样的位置再次出错
  • 浪费时间 + 浪费 API 费用

一句话规则 + 今晚作业

一句话

try/except 不是写给工程师的, 是写给你的脚本的 "安全气囊".

凡是调 API 的代码, 都包在 try/except 里. 没有例外.

  2. 故意把 .env 里的 Key 改错, 跑脚本, 看是不是正确捕获了 AuthenticationError.
  3. 在循环里调 ask_ai() 10 次, 每次打印 "第 N 条成功" 或 "第 N 条失败", 观察日志.

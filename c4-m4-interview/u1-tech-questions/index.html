AI PM 面试, 技术题考什么?

好消息: 不考你写代码. 坏消息: 考你能不能用产品语言讲技术 trade-off.

根据 2026 年面试数据, AI PM 技术面最高频的 7 个问题:

  1. RAG vs Fine-tune 怎么选?
  2. Token 成本怎么控?
  3. 幻觉 (Hallucination) 怎么处理?
  4. Prompt Engineering 和写代码有什么区别?
  5. 怎么评测模型好不好?
  6. 用户等太久 (延迟) 怎么办?
  7. 数据安全怎么保障?

面试官问这些, 不是要考你会不会训模型 — 是想看你有没有和工程师对话的能力.

一个能讲清 trade-off 的 PM, 工程师愿意跟你合作; 一个说 "我不懂技术" 的 PM, 工程师会直接绕过你.

(3)

Q1: RAG 和 Fine-tune 怎么选?

这是 2026 年 AI PM 面试出现率最高的一题. 30 秒标准答案:

你的数据是否频繁更新?
↓
是 → RAG
知识库可以随时更新, 不需要重新训练模型
否 → 继续问 ↓
数据相对稳定, 可以考虑其他方案
你需要统一输出风格/格式吗?
↓
是 → Fine-tune
训练模型学会你的风格, 每次输出都稳定
否 → Prompt Engineering
预算有限就先从 Prompt 开始, 逐步升级
面试金句:

"我的原则是 Prompt → RAG → Fine-tune 逐步升级. 先用 Prompt Engineering 验证可行性, 不够就加 RAG 做知识增强, 最后才考虑 Fine-tune. 而且最佳实践是组合使用: Fine-tune 管风格, RAG 管事实."

第二题: 钱的问题 ↓
(4)

Q2: Token 成本怎么控?

面试官问这题, 是在考你有没有成本意识. AI PM 必须会算账.

面试官: "你们的 AI 功能一个月 Token 成本多少? 怎么控?"

4 个杠杆, 从易到难:

  1. Prompt 压缩 — 去掉冗余指令, 把 500 token 的 prompt 压到 200. 效果不变, 成本直降 60%.
  2. 模型分级 — 简单任务 (分类/摘要) 用 GPT-4o-mini ($0.15/1M token), 复杂任务 (推理/创作) 才用 GPT-4o ($2.5/1M token). 成本差 15 倍.
  3. Prompt Caching — 相同前缀的 prompt 走缓存, 重复调用成本降 50%.
  4. 输出控制 — 设 max_tokens 限制生成长度, 避免模型"话痨".
PM 要会算的账:

假设 Vitamin 日活 10,000, 每用户每天平均 3 次 AI 交互, 每次 ~800 token (输入+输出).
月 Token 用量 = 10,000 × 3 × 800 × 30 = 7.2 亿 token.
用 GPT-4o-mini: 7.2 亿 × $0.15/1M ≈ $108/月.
用 GPT-4o: 7.2 亿 × $2.5/1M ≈ $1,800/月.

第三题: 最让人头疼的幻觉 ↓
(5)

Q3: 幻觉怎么处理?

AI 一本正经地编 — 这是所有 AI 产品最大的风险. 面试官必问.

面试官: "你的产品里, 模型如果说错了怎么办?"

三层防线, 从底层到产品层:

  1. 底层 Grounding — 用 RAG 给模型喂真实数据源. 模型基于检索到的文档生成, 而不是凭"记忆"编造. 并要求输出引用来源.
  2. 中间层检测 — 让模型同时输出 confidence score. 低于阈值 (比如 0.7) 的回答, 走人工审核队列, 不直接展示给用户.
  3. 产品层兜底 — UI 上标注 "AI 生成, 仅供参考". 关键决策场景 (健康/金融/法律) 强制人工确认. 用户可以一键反馈 "这个回答有误".
健康类产品 (如 Vitamin) 的红线:

绝对不能让 AI 给出诊断性建议 (如 "你可能得了 X 病"). 只能给信息性建议 (如 "经期运动的一般建议是..."). 这不是技术问题, 是法律问题.

第四题: Prompt 到底是啥 ↓
(6)

Q4: Prompt Engineering 和写代码有什么区别?

很多面试官会追问: "Prompt 不就是写指令吗? 跟写代码有什么不一样?"

代码
→
确定性: 同样输入, 永远同样输出. 可以 debug.
Prompt
→
概率性: 同样输入, 每次输出可能不同. 需要评测集来验证.
面试金句:

"我把 Prompt 当作产品规格说明 (spec)来管理. 每次修改都走版本控制, 用 50-200 条评测用例跑 regression test, 效果下降就回滚. 跟代码的 CI/CD 流程一样, 只是验证方式从 unit test 变成了 eval set."

最后三题, 一起过 ↓
(7)

Q5 / Q6 / Q7 — 评测 / 延迟 / 安全

这三题出现频率稍低, 但都是加分项. 每题 20 秒答完.

Q5: 怎么评测模型?

3 步: (1) 建 50-200 条 golden truth 评测集 (2) 选指标: 分类用 F1, 生成用 ROUGE, RAG 用 RAGAS (3) 上线后持续跑 regression test, 模型更新时全量回归.

Q6: 延迟怎么降?

4 招: (1) Streaming 逐字输出, 感知延迟降 50% (2) 简单任务用小模型 <1s (3) 非实时任务走异步 (4) 高频场景预生成, 缓存结果.

Q7: 数据安全?

4 层: (1) PII 脱敏再送模型 (2) 敏感场景私有部署 (3) 遵守 GDPR/个保法, 不用用户数据训练 (4) 所有 API 调用留审计日志.

这些题在 Vitamin 项目里怎么答 ↓
(9)

回到 Vitamin — 7 题都套上你的项目

面试官追问 "你自己的项目呢?" 时, 你要能秒切到 Vitamin 案例.

Vitamin 用到的技术选型

  • RAG: 健康知识库 (经期/运动/饮食) 用 RAG 检索, 数据可以随时更新
  • Prompt Engineering: 情绪安抚对话用精调 prompt, 版本管理 + eval set
  • 模型分级: 情绪分类用 GPT-4o-mini, 复杂健康问答用 GPT-4o
  • Streaming: 对话回复逐字输出, 用户感知延迟 < 2s

Vitamin 的安全红线

  • PII 脱敏: 用户生理数据脱敏后才送入模型
  • 不做诊断: AI 只给信息性建议, 不做医疗诊断
  • 幻觉兜底: 健康建议标注 "仅供参考", 异常回答走人工审核
  • 合规: 用户数据不进训练集, 符合个人信息保护法
面试话术模板:

"在 Vitamin 项目里, 我选了 RAG + Prompt Engineering 的组合. RAG 负责健康知识检索, Prompt 负责输出风格控制. 没有用 Fine-tune, 因为我们的数据量不够 1000 条, 而且健康知识需要频繁更新, RAG 更合适."

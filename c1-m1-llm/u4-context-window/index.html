(1)

AI 的 "失忆" 现场

试过这样的场景吗 ——

第 1 条: "我叫产品经理, 在深圳, 做健康类 AI 产品."

第 5 条: "帮我分析一下用户画像."

第 12 条: "那你觉得我们产品应该怎么定位?"

第 18 条: "对了, 我第 1 条说的那个产品叫什么来着?"

AI 的回答: "抱歉, 根据我们的对话记录, 你没有提到具体的产品名..."

→ 你明明第 1 条就说了. 但 AI "忘了".

这不是 bug. 这是 AI 的物理限制 — 它的 "短期记忆" 有一个硬上限.

这个上限叫 Context Window.

Context Window 到底是什么? ↓
(2)

AI 没有 "长期记忆"

✗

猜测: AI 像人一样有记忆

它记住了我之前说的每句话, 就像人记日记.

不对: AI 没有"记忆". 每次你看到的 "上下文", 都是你自己的对话被完整塞进去重新处理的. 它不是 "想起来", 是 "重新读了一遍".
✗

猜测: 越贵的模型记忆越好

GPT-4 比 GPT-3.5 贵, 记忆肯定更大?

半对: GPT-4 的 context window 确实比 GPT-3.5 大 (128K vs 4K). 但 "大" 不等于 "好" — 中间的信息依然会被忽略.
真相

Context Window = 模型一次能 "看到" 的全部内容.

它不是记忆, 是一个固定大小的框.

每次调用, 你的 system prompt + 对话历史 + 新输入, 全部塞进这个框里. 超过框的部分, AI 完全看不到.

→ 类比: Context Window 是一张桌子. 你能同时摊开多少张纸取决于桌子大小. 放不下的纸, 就得收起来 — AI 看不到收起来的.

这个 "框" 里装的到底是什么? ↓
(3)

Context = 四块内容, 挤在一个框里

每次调用 API, 模型看到的是这四块拼在一起:

System
对话历史
用户输入
AI 输出
剩余空间
System Prompt (人设/规则)
对话历史 (之前的问答)
当前用户输入
AI 生成的输出
关键认知:
  1. System Prompt 每次都要占位: 如果你的 system prompt 有 1000 tokens, 每次对话都要 "交" 这 1000 token 的 "房租"
  2. 对话越长, 历史越大: 聊 10 轮, 历史可能已经有 3000 tokens
  3. 输出也算: AI 生成的回复也占 context 空间
  4. 总和不能超过上限: 超了就得砍
那这个上限到底是多大? ↓
(4)

不同模型, 桌子大小差别很大:

但别急着高兴 — 大窗口不代表用得好. 下一步告诉你为什么.

128K 够用了吧? 其实有个大坑 ↓
(5)

"Lost in the Middle" — 大 ≠ 好

Google 的研究发现: 即使 context 够大, 模型对中间位置的信息也 "不太上心".

模型对不同位置信息的注意力强度
← 开头 (高注意力) 中间 (低注意力) 结尾 (高注意力) →
PM 实用含义:
  1. System prompt 放在最前面 → 注意力高 ✓
  2. 用户最新的问题在最后面 → 注意力高 ✓
  3. 对话历史中间的信息 → 容易被忽略 ✗

所以 "我之前说过 X" 这种引用, AI 可能真的 "没注意到".

→ 论文: "Lost in the Middle: How Language Models Use Long Contexts" (2023, Google/Stanford)

那 context 满了之后, 怎么处理? ↓
(6)

满了之后: 三种应对策略

策略 1: 截断最早的

像聊天记录一样, 只保留最近 N 轮. 最简单, 但用户之前说的重要信息会丢失.

适合: 简单问答类产品

策略 2: 报错拒绝

超过上限直接报错 "对话太长, 请新建一个". 粗暴但安全.

适合: 早期 MVP, 不想处理复杂逻辑

策略 3: 摘要压缩

把早期对话让 AI 压缩成摘要, 用摘要替代原文. 最优雅, 但摘要可能丢细节.

适合: Vitamin 这类需要多轮追踪的产品

实际产品通常组合使用:

System prompt (固定) + 摘要 (压缩早期) + 最近 3-5 轮原文 (保留最新上下文).

ChatGPT / Claude 的 "记忆" 功能, 本质就是策略 3 的变体.

Context 越长, 钱也越贵 ↓
(7)

Context 越长 = 每次调用越贵

API 是按 输入 token 数 + 输出 token 数 收费的. 你塞进 context 的每个 token 都算钱.

一笔简单的账:

  • System prompt: ~200 tokens (每次都要付)
  • 对话历史 (5 轮): ~500 tokens
  • 用户新输入: ~50 tokens
  • AI 回复: ~150 tokens

总计: ~900 tokens / 次

如果对话到了 20 轮, 历史膨胀到 ~3000 tokens:

总计: ~3400 tokens / 次 — 是第 1 轮的 4 倍.

这意味着:

  • 对话越长, 每次调用越贵 (因为历史越来越大)
  • 用策略 3 (摘要压缩) 可以把历史压缩到固定大小, 控制成本
  • 限制对话轮数也是一种成本控制手段
PM 经验法则:

"设计产品时, 永远假设 context 是稀缺资源. 每个塞进 context 的 token, 都在烧钱."

把这个知识用到 Vitamin 上 ↓
(9)

回到 Vitamin — 多轮对话怎么管 Context

Vitamin 需要追踪用户 5 种身体状态, 记住历史. 但 context 有限.

✓ 可以这么设计

  • System prompt 精简: 控制在 200 tokens 以内
  • 摘要压缩: 每 5 轮, 让 AI 压缩历史到 ~100 tokens
  • 状态卡片: 用结构化 JSON 保存用户 5 状态, 固定 50 tokens
  • 最近 3 轮原文: 保持最新上下文的完整性
  • 选 128K 模型: 用 DeepSeek V3, 空间够大

✗ 别这么做

  • 无限保留所有历史: 20 轮后 context 炸了
  • 把文档塞进 context: 一份说明书 5000 tokens, 直接占满
  • 忽略 lost-in-middle: 重要信息放在中间, AI 会忽略
  • 不限制对话轮数: 成本随轮数线性增长
面试金句, 抄走:

"Context Window 是多轮对话的天花板. 我设计 Vitamin 时, 用 '摘要 + 最近 3 轮 + 结构化状态卡' 的策略管理 context, 既保证连贯性, 又控制成本. 每次调用控制在 500 tokens 以内."

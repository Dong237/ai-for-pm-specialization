中文为什么更 "贵"?

同样意思, 中文 token 数 ≈ 英文的 1.5 倍. 这不是语言的问题, 是 tokenizer 的问题.

上节课我发现 "维他命" = 3 tokens, "Vitamin" = 1 token.

如果 Vitamin 是中文产品, 每次调 API 都比英文产品贵?

有没有办法解决?

—— 今天弄清楚.

10 步学完, 你会:
  • 知道中文 token 数为什么通常比英文多
  • 理解 UTF-8 编码对 tokenizer 的影响
  • 能对比不同模型对中文的 token 效率
  • 给 Vitamin 选一个 "中文友好" 的模型
走起 →
(1)

一封让 PM 冒汗的账单

假设你给 Vitamin 写了一段健康建议 prompt:

英文版: "You should drink more water and get enough sleep during your period."
~14 tokens (GPT-4)

中文版: "经期期间你应该多喝水，保证充足的睡眠。"
~18 tokens (GPT-4)

同样的意思, 中文多了 29% 的 token.

而且这还是一句话. 如果是一个完整的 system prompt + 多轮对话, 差距会更大.

PM 的直觉:

"等一下... Vitamin 是中文产品. 如果每次对话都多花 30-50% 的 token, 一个月下来, 我的 API 成本直接涨了 30-50%?"

答: 对. 但好消息是, 选对模型可以大幅缩小这个差距.

先搞清楚: 为什么中文 token 多? ↓
(2)

两个猜测 — 一对一错

△

猜测 A: 中文字多, 所以 token 多

中文一个字信息量大, 字数其实少于英文.

不完全对: "经期应该多喝水" = 7 个字. 英文 "Drink more water during period" = 6 个词. 中文字数并不多 — 问题不在信息量.
✗

猜测 B: 中文语言更复杂

中文语法更难, 所以 AI 需要更多 token 理解?

完全不对: Token 只是切分方式, 跟语法难度无关. 日文甚至比中文更 "贵" — 因为混了汉字 + 假名 + 罗马字.
真相

中文 token 多, 是因为 tokenizer 是英文数据训出来的.

训练数据偏谁, 谁就被压缩得更好.

GPT-4 的训练语料约 90% 是英文. 英文的常见组合被充分合并成大积木, 中文的组合合并不够.

→ 好消息: 中国公司的模型 (DeepSeek, Qwen) 用了更多中文数据训练 tokenizer, 差距小很多.

来看根因: UTF-8 编码的影响 ↓
(3)

根因: 一个中文字 = 3 个字节

BPE tokenizer 的输入不是 "字符", 而是 字节 (byte). 这就是差距的物理根源.

A →
0x41
1 byte
中 →
0xE4 0xB8 0xAD
3 bytes
好 →
0xE5 0xA5 0xBD
3 bytes
BPE 的起点是字节. 所以:
  1. 英文 "hello" = 5 bytes → BPE 起步轻松, 很快合并成 1 token
  2. 中文 "你好" = 6 bytes → BPE 起步要处理更多字节
  3. 如果训练数据里 "你好" 出现得够多, BPE 会把这 6 bytes 合并成 1 token
  4. 但如果不够多 (因为训练数据 90% 英文), 就只能拆成 "你" + "好" 两个 token

→ 这就是为什么 "你好" 在 GPT-4 里是 1 token (够常见), 但 "维他命" 是 3 token (不够常见).

来看具体数据: 5 组等义句子的对比 ↓
(4)

5 组等义句子, 看差距

同样意思的句子, 中英文 token 数差多少? 用 GPT-4 的 tokenizer 来切:

场景 英文 tokens 中文 tokens 倍率
问候 6 8 1.3x
健康建议 14 18 1.3x
产品介绍 15 22 1.5x
技术描述 15 20 1.3x
长段落 38 56 1.5x

规律:

  • 短文本: 差距较小 (~1.3x), 因为常见词已被合并
  • 长文本: 差距拉大 (~1.5x), 罕见组合被逐字切
  • 平均: 中文约 1.3-1.5 倍 于等义英文
但这只是 GPT-4 的情况. 中国模型呢? ↓
(5)

不同模型, 中文效率大不同

GPT-4 的 tokenizer 偏英文. 但中国公司的模型, 专门为中文优化了 tokenizer:

模型 词表大小 "健康建议"句 tokens 中文效率
GPT-4 ~100K 18 基准
DeepSeek V3 ~128K 11 提升 39%
Qwen (通义) ~152K 10 提升 44%

→ 同一句中文, Qwen 只需要 GPT-4 一半多的 token. 差距巨大.

为什么中国模型更高效?

  • 更大的词表: 128K-152K vs 100K, 能装更多中文常见组合
  • 更多中文训练数据: 中文组合被充分合并成大积木
  • 专门优化: 阿里 / DeepSeek 把中文压缩效率作为设计目标
这对产品成本意味着什么? ↓
(6)

选对 tokenizer = 省钱

Token 效率 + 单价, 双重影响你的账单:

GPT-4o
中文 token 效率低 + 单价高
¥3,000/月
1000 DAU · 5 次/天 · 500 tokens/次
DeepSeek V3
中文效率高 + 单价低
¥90/月
同样用量, 便宜 33 倍
成本差距来自两个维度:
  1. Token 效率: 同一句中文, DeepSeek 只需 GPT-4 的 ~60% token → 省 40%
  2. 单价: DeepSeek V3 ≈ ¥2/百万 token, GPT-4o ≈ ¥60/百万 token → 省 30 倍
  3. 叠加效果: 0.6 × (1/30) ≈ 总成本仅为 GPT-4o 的 3%

→ 对中文产品来说, tokenizer 效率是选型的重要维度. 但别忘了, 质量也很重要.

但中文不是在所有场景都更贵 — 有个反直觉 ↓
(7)

一个反直觉: 中文不是永远更贵

中英文 token 差距不是固定的, 跟内容类型有关:

内容类型 中/英 倍率 (GPT-4) 原因
短问候 ~1.0-1.3x "你好"、"谢谢" 等超常见词已被合并
日常对话 ~1.3x 常见词多, 差距不大
专业文档 ~1.5-2.0x 专业术语少见, 被逐字切
代码 / 数字 ~1.0x 都是 ASCII, 没有差异
JSON 结构 ~1.0x 键名是英文, 只有值是中文

实用技巧:

  • System prompt 中的指令部分可以用英文写 → 省 token
  • JSON 输出格式的键名用英文 → 省 token
  • 但用户看到的内容 (回复、建议) 必须是中文 → 这部分没法省
一个 Vitamin 的 prompt 优化例子:

优化前 (全中文):
你是一个女性健康助手。请根据用户的身体状态给出建议。
→ ~16 tokens

优化后 (指令用英文):
You are a women's health assistant. Respond in Chinese based on user body state.
→ ~18 tokens

→ 这个例子差不多. 但如果 system prompt 有几百字, 差距就明显了. 通常混合方式效果最好.

(9)

回到 Vitamin — 选模型的 "中文税" 计算

Vitamin 是中文产品. 选模型时, tokenizer 的中文效率是关键考量维度.

✓ 中文友好模型

  • DeepSeek V3: 中文 token 效率比 GPT-4 高 ~39%
  • Qwen (通义): 中文效率最优, 比 GPT-4 高 ~44%
  • GLM (智谱): 中文优化, 效率接近 DeepSeek
  • 单价也更低 → token 省 + 价格省 = 双重优势

✗ 中文 "贵" 的模型

  • GPT-4 / GPT-4o: 英文优化, 中文效率低
  • Claude: 中文支持好但 tokenizer 偏英文
  • Llama: Meta 出品, 中文效率最低
  • 如果必须用, 要在成本预算里加 "中文税"
面试金句, 抄走:

"选模型不只看'谁最聪明', 还要看 tokenizer 的中文压缩效率. 同一个中文 prompt, 在 Qwen 上只需要 GPT-4 一半的 token. 对中文产品来说, 这直接决定成本和用户体验."

→ 面试官问 "为什么选这个模型", 这句话就是加分项.

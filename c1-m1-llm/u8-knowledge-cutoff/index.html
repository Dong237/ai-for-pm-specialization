Unit 8·
知识截止
Knowledge Cutoff

为什么 AI 不知道最近的事, 以及怎么 "补救".

我问 ChatGPT: "谁拿了 2026 奥斯卡最佳影片?"

它要么说 "我不知道", 要么... 编一个.

—— 为什么? 怎么解决?

10 步学完, 你会:
  • 理解知识截止日 (Knowledge Cutoff) 是什么
  • 知道不同模型的截止日各不相同
  • 掌握两大解法: RAG 和 Web 搜索
  • 给 Vitamin 设计一套 "知识保鲜" 方案
走起 →
(1)

试一试: "谁拿了 2026 奥斯卡?"

打开任何一个 AI 聊天工具, 问这个问题:

你的 prompt →
"谁拿了 2026 年奥斯卡最佳影片?"
A

老实型回答

"抱歉, 我的知识截止到 2024 年 4 月, 无法回答这个问题."

至少它承认了 — 但用户体验不好.
B

幻觉型回答

"2026 年奥斯卡最佳影片是《xxx》, 由导演 yyy 执导..."

自信满满, 但完全是编的. 上节课学的 Hallucination.

两种回答都不理想. 但问题的根源是同一个: 模型的训练数据有一个截止日期.

什么是训练数据的截止日? ↓
(2)

训练数据有一条日期边界

LLM 的知识来自训练数据 — 而训练数据的收集 有一个截止日期.

核心概念

Knowledge Cutoff (知识截止日)

模型最后一次 "看" 到新数据的日期.

截止日之前的事 → 模型可能知道 (因为训练数据里有).

截止日之后的事 → 模型完全不知道 (因为训练数据里没有).

就像一本百科全书, 印刷之后就不会自动更新了.

类比:

LLM 就像一个在某个日期定格的超级读者. 它在截止日之前疯狂阅读了互联网上几乎所有的文字. 但截止日之后, 它就 "闭关" 了, 外面发生的事一概不知.

不同模型的截止日一样吗? ↓
(3)

不同模型, 不同截止日

每个模型训练时间不同, 所以截止日也不同:

模型知识截止日意味着...
GPT-4o2024 年 4 月不知道 2024 年 5 月之后的事
Claude 3.5 Sonnet2025 年 4 月不知道 2025 年 5 月之后的事
DeepSeek V32025 年 3 月不知道 2025 年 4 月之后的事
Gemini 1.5 Pro2024 年 11 月不知道 2024 年 12 月之后的事
Llama 32023 年 12 月不知道 2024 年之后的事
PM 必须知道的:
  • 截止日越新 → 模型知道的越多 (但也意味着训练成本更高)
  • 同一家公司不同版本的模型, 截止日也不同 (GPT-3.5 vs GPT-4 vs GPT-4o)
  • 截止日 ≠ 发布日 — 模型可能在 2024 年发布, 但训练数据只到 2023 年
"不知道" 到底意味着什么? ↓
(4)

"不知道" ≠ "笨" — 是 "从未见过"

这个区分很重要. 模型不知道 2026 奥斯卡, 不是因为它理解能力不行.

模型能做到 (不受截止影响)

  • 推理、分析、总结 (能力不过期)
  • 写代码、翻译、改写文风
  • 回答截止日之前的事实问题
  • 基于你提供的新信息进行分析

模型做不到 (截止后的盲区)

  • 知道截止日之后的新闻
  • 知道新发布的产品/论文/政策
  • 知道最新的股价、天气、赛事结果
  • 知道自己不知道 (可能会编造)

记住:

Knowledge Cutoff 影响的是 "知识",
不影响 "能力".
模型的推理能力不会过期.

对做产品有什么具体影响? ↓
(5)

对产品的影响: 过期的知识是危险的

对于像 Vitamin 这样的 健康产品, 过期的知识可能造成伤害:

危险场景:
  • 医学指南更新: 2025 年某个药物被发现有新的副作用, 但模型的数据只到 2024 → 它还会推荐这个药
  • 新药上市: 2025 年 FDA 批准了一种新的女性健康药物, 但模型完全不知道它的存在
  • 研究反转: 2024 年的研究说 "A 有效", 2025 年的研究说 "A 无效" → 模型只知道 "A 有效"
  • 法规变化: 新的隐私法规要求不同的健康数据处理方式, 但模型不知道
PM 决策:

如果你的产品涉及时效性强的领域 (医疗、法律、金融、新闻), 你不能直接依赖模型的内置知识. 你需要外挂知识更新机制.

这就是接下来要讲的 RAG 和 Web Search.

第一个解法: RAG ↓
(6)

解法 1: RAG — 把新鲜文档塞进去

RAG (Retrieval-Augmented Generation) = 检索增强生成. 核心思想很简单:

RAG 原理

先搜, 再答.

用户问问题 → 先从你的知识库里搜出相关文档 → 把文档塞进 prompt → 让模型基于这些文档回答.

模型不需要 "知道" 答案, 只需要 "读懂" 你给的材料.

→
检索知识库
(向量搜索)
→
相关文档
塞进 Prompt
→
模型基于
文档回答
RAG 的优势:
  • 知识可控: 你决定知识库里放什么内容
  • 实时更新: 往知识库加新文档就行, 不用重新训练模型
  • 可溯源: 能告诉用户 "这个回答基于哪篇文档"
  • 成本低: 比重新训练模型便宜几个数量级
RAG 的局限:
  • 知识库需要人工维护 — 谁来添加新文档? 多久更新一次?
  • 检索不到 → 还是回答不了
  • 知识库太大 → 检索慢, 召回率下降
  • 只覆盖你准备好的领域, 不覆盖全网信息
另一个解法: Web 搜索 ↓
(7)

解法 2: Web 搜索 — 让模型上网

另一个思路: 让模型在回答之前先搜一下互联网.

Web 搜索 vs RAG 对比:
维度RAGWeb 搜索
信息来源你的私有知识库公开互联网
新鲜度取决于你更新频率实时
可控性高 (你决定放什么)低 (搜到什么用什么)
准确性高 (来源可控)不确定 (网上信息良莠不齐)
适用场景专业领域 (医疗/法律)通用信息 (新闻/热点)
实现复杂度中等 (需要搭知识库)低 (调用搜索 API)
最佳实践: RAG + Web 搜索 = 双保险

很多产品两个都用: 专业知识用 RAG (自建知识库, 确保准确), 通用信息用 Web 搜索 (保证新鲜).

例如: Vitamin 用 RAG 存储经过审核的医学指南, 用 Web 搜索补充最新的健康新闻.

回到 Vitamin ↓
(9)

回到 Vitamin — 医学知识保鲜方案

Vitamin 是女性健康 AI 应用, 做 5 州身体状态追踪. 医学知识的时效性是生死攸关的.

Vitamin 的知识截止问题:
功能知识时效要求解决方案
经期健康建议需要最新的医学指南RAG: 每月更新医学文献库
用药查询需要最新的 FDA 批准信息RAG: 每周同步 FDA 数据
健康新闻需要实时信息Web 搜索: 检索最新报道
身体状态分析基于用户数据, 不依赖外部知识不需要补救 (能力不过期)
日常关怀对话不涉及时效性不需要补救 (通用能力)
面试金句, 抄走:

"Vitamin 涉及女性健康, 知识过期可能造成伤害. 我会用 RAG 搭建经过医学团队审核的知识库, 每月更新医学指南; 同时接入 Web 搜索补充通用健康资讯. 两套方案互补: RAG 保证准确性, Web 搜索保证新鲜度."

RAG 知识库设计要点:
  • 来源审核: 只收录经过医学团队审核的内容 (不是随便网上抓)
  • 更新频率: 核心医学指南每月更新, FDA 数据每周同步
  • 版本管理: 旧版文档不删除, 标记为 "已过期", 方便追溯
  • 回答溯源: 每条回答标注来源文档和更新日期, 让用户知道信息有多 "新"
一句话 + 作业 ↓
(10)

一句话规则 + 今晚作业

一句话:

模型的知识有 "保质期".
RAG 补专业知识, Web 搜索补通用信息.

知识截止不是 bug — 是架构特性. PM 的工作是设计补救方案.

课后作业 (今晚 30 分钟):
  1. 打开 ChatGPT, 问它 3 个 "它的截止日之后" 的事实问题, 记录它的回答方式 (承认不知道? 编造? 用了搜索?)
  2. 给 Vitamin 设计一个 RAG 知识库方案: 包含哪些数据源, 更新频率, 审核流程.
  3. 对比 ChatGPT (有 Web 搜索) 和原始 GPT-4 API (无搜索) 回答同一个时事问题的差异, 写 150 字分析.
  4. 写 100 字: 向你的 CEO 解释 "为什么我们不能直接用模型的内置知识给用户做健康建议, 必须加 RAG".

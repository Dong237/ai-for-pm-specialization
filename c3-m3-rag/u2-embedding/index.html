(1)

电脑不认字, 只认数字

你写"苹果"两个字, 电脑看到的是一串编码 (比如 Unicode). 但编码只是ID, 不包含任何"意思".

"苹" = U+82F9, "果" = U+679C

这些数字告诉电脑怎么显示这个字, 但不告诉它这是一种水果.

所以"苹果" 和 "橘子" 在 Unicode 里的距离, 跟"苹果" 和 "卡车" 的距离一样远. 电脑完全不知道苹果和橘子更相似.

怎么让电脑理解"意思"?
答案就是 Embedding — 一种把"意思"编码进数字的方法 ↓
(2)

Embedding: 把"意思"变成坐标

核心概念

Embedding 就是:

给每个词/句子一个"坐标"

意思相近的词, 坐标靠近. 意思不同的词, 坐标远离.

就像地图上, 北京和天津离得近, 北京和纽约离得远. Embedding 是"语义地图"上的坐标.

一个简单的例子. 如果我们只用 2 个数字来表示一个词:

苹果

[0.82, 0.15]
第一个数 = "水果程度", 第二个数 = "交通工具程度"

橘子

[0.85, 0.12]

卡车

[0.05, 0.91]

看到了吗? 苹果和橘子的数字很接近, 而苹果和卡车的数字差很远.

实际的 Embedding 不止 2 个数字 — 通常是 768 个甚至更多 ↓
(3)

768 维的"意思坐标"

实际的 Embedding 模型 (比如 OpenAI 的 text-embedding-3-small) 给每段文本一个 768 维 的向量. 就是 768 个数字.

为什么要这么多维?

2 维只能区分"是不是水果" 和 "是不是交通工具". 但语言的意思比这复杂得多 — 你还需要区分"颜色", "情绪", "时间", "健康", "动作" 等无数维度.

768 维 = 768 个"意思维度", 足以捕捉人类语言中大部分的语义关系.

类比: GPS 坐标用 2 个数字 (经度, 纬度) 定位地球上的位置. Embedding 用 768 个数字定位"语义空间"里的位置.

  • GPS: 2 维物理空间 → 定位地理位置
  • Embedding: 768 维语义空间 → 定位"意思"
那这些向量是怎么"学"出来的? ↓
(4)

向量怎么"学"出来的 — 训练直觉

Embedding 不是人手写的, 是模型从大量文本中自动学出来的.

训练原理 (简化版)
  1. 读大量句子: "我吃了一个苹果" "她买了三斤橘子"
  2. 发现: "苹果" 和 "橘子" 经常出现在类似的上下文中 (吃、买、水果店...)
  3. 所以: 模型给它们分配了相近的向量
  4. "苹果" 和 "卡车" 很少出现在同一种上下文 → 向量远离
→ 上下文越相似的词, 向量越接近

一句话: Embedding 模型通过"看哪些词经常出现在一起" 来学习语义. 这就是为什么它能捕捉"意思" — 不是因为它"理解", 而是因为它见过太多例子.

向量有了, 怎么量"距离"? ↓
(5)

向量距离 = 语义相似度

RAG 检索的核心就是: 算问题向量和每个文档 Chunk 向量之间的距离. 距离越近, 越相关.

苹果 ↔ 橘子
0.92
经期 ↔ 排卵
0.88
苹果 ↔ 手机
0.45
苹果 ↔ 卡车
0.12

Cosine Similarity (余弦相似度) 是最常用的距离度量:

  • 1.0 = 完全相同方向 (意思一样)
  • 0.0 = 完全无关
  • RAG 检索时, 取 cosine similarity 最高的 top-K 个 chunk

你不需要记公式 — 只需要知道: 越接近 1, 越相关.

市面上有哪些 Embedding 模型? PM 怎么选? ↓
(6)

Embedding 模型选型 — PM 需要知道的

不同的 Embedding 模型, 向量质量不一样, 价格也不一样.

OpenAI
→
text-embedding-3-small: 1536 维, $0.02/百万 token. 性价比高, 适合大多数场景.
Cohere
→
embed-multilingual-v3: 1024 维, 多语言支持好. 适合中英混合知识库.
开源
→
bge-large-zh / m3e-base: 中文专优, 可以本地部署, 无 API 费用.
阿里
→
通义千问 Embedding: 百炼平台直接用, 中文效果好, 国内合规.

PM 选型原则:

  1. 中文为主 → 选中文优化模型 (bge / m3e / 通义)
  2. 需要多语言 → Cohere 或 OpenAI
  3. 数据敏感 → 选开源模型本地部署
  4. 预算有限 → 开源免费, API 按量付费
Embedding 和 RAG 的关系到底是什么? ↓
(7)

Embedding 在 RAG 中的角色

Embedding 是 RAG 的"搜索引擎". 没有 Embedding, RAG 就没法搜.

离线阶段 每个文档 Chunk → Embedding 模型 → 768 维向量 → 存入 Vector DB
在线阶段 用户问题 → 同一个 Embedding 模型 → 768 维向量 → 和 Vector DB 里的向量比距离 → Top-K

关键陷阱: 离线和在线必须用同一个 Embedding 模型!

如果建库时用 OpenAI 模型, 查询时用 Cohere 模型 → 向量空间不同 → 距离算出来没意义 → 检索全乱.

就像你用北京坐标系定位了所有餐馆, 却用东京坐标系去搜 — 当然搜不到.

到了最有趣的部分 — 亲手玩一下向量空间 ↓
(8)

动手: Embedding 空间探索器

下面是一个 2D 的"语义空间". 每个圆点是一个词. 点击查看最近邻居, 拖动改变位置.

点击任意词语圆点, 查看它的最近邻居. 拖动圆点可以移动位置.

观察:

  • 同类的词 (水果、交通工具、健康) 天然聚在一起

回到 Vitamin ↓
(9)

Vitamin 的 Embedding 选择

Vitamin 的知识库是中文医学内容为主. Embedding 模型的选择直接影响检索质量.

推荐方案

  • 首选: bge-large-zh-v1.5 — 中文 MTEB 榜单第一梯队, 免费开源
  • 备选: 通义千问 Embedding — 阿里云生态, 中文效果好
  • 向量维度: 1024 维, 足以覆盖医学术语的语义细节
  • 医学术语覆盖: "经期"、"排卵"、"黄体期" 都能正确编码

需要注意

  • 英文模型 (如 OpenAI) 对中文医学术语效果偏弱
  • 换模型 = 重新向量化整个知识库 (几小时成本)
  • 模型版本更新也可能需要重建索引
  • 测试: 用真实用户问题测检索准确率, 再定模型

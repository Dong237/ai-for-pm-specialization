(1)

为什么不能把整篇文档直接 Embed?

一篇医学文档可能有 5000 字, 覆盖 10 个不同话题. 如果整篇 Embed 成一个向量:

✗

问题 1: 向量太"稀释"

5000 字的内容压成一个向量, 每个话题的信息被稀释. 就像把 10 种颜色混在一起变成灰色 — 什么颜色都不像了.

✗

问题 2: 塞不进 LLM

LLM 的 Context Window 有限 (4K~128K tokens). 如果每次检索返回 3 篇 5000 字文档 = 15000 字, 占满了窗口, 没空间给 prompt 和回答.

所以: 把长文档切成小段 (Chunk), 每段 200~500 字. 每段有自己的向量, 检索时能精确匹配到最相关的那一小段.

类比: 整本教科书 vs 索引卡片. 你搜"经期饮食"时, 找到的是那一张相关的卡片, 不是整本书.

那切多大最好? ↓
(2)

Chunk 大小: 不大不小, 刚刚好

核心

Chunk 太大 → 向量被稀释, 检索不精准

Chunk 太小 → 缺乏上下文, 意思不完整

最佳: 256 ~ 512 tokens

中文大约 400~800 字. 行业共识是从 512 tokens 开始试.

256
短 Chunk
适合 FAQ、简短回答
精确匹配, 但上下文少
512
标准 Chunk
最推荐的起始值
平衡精度和上下文
1024
长 Chunk
适合复杂分析型问题
上下文充足, 但不够精确
大小定了, 还有一个问题: 边界怎么处理? ↓
(3)

边界问题: 一句话被切成两半

假设你按 50 个字一刀切. 原文是:

原文:
经期应避免食用过于寒凉的食物,如冰淇淋、冰镇饮料等,|← 第50个字, 这里切! 因为寒凉食物可能会加重痛经症状.

Chunk 1: "经期应避免食用过于寒凉的食物,如冰淇淋、冰镇饮料等,"

Chunk 2: "因为寒凉食物可能会加重痛经症状."

Chunk 2 失去了上文 — 用户问"经期能吃冰吗?", 如果只检索到 Chunk 2, 它说"因为寒凉食物加重痛经", 但没说什么寒凉食物!

解决方案: Overlap (重叠) — 让相邻 Chunk 有一段重叠内容.

Overlap 具体怎么工作? ↓
(4)

Overlap: 给边界加"缓冲带"

Overlap 就是让相邻的 Chunk 共享一部分文字. 比如 Chunk Size = 50 字, Overlap = 10 字:

Chunk 1: 字 1~50

Chunk 2: 字 41~90 (前 10 个字和 Chunk 1 重叠)

Chunk 3: 字 81~130 (前 10 个字和 Chunk 2 重叠)

重叠的部分就是"缓冲带" — 确保被切断的句子在下一个 Chunk 里也出现.

行业最佳实践: Overlap = Chunk Size 的 10%~20%

  • Chunk = 512 tokens → Overlap = 50~100 tokens
  • NVIDIA 测试发现 15% overlap 效果最好
  • 太多 overlap → Chunk 数量暴增, 存储成本上升, 检索冗余
  • 太少 overlap → 边界信息丢失

有趣的发现: 2026 年的研究显示, 在某些检索场景中, overlap 的收益可能接近零. 所以最好的做法是: 有 overlap 和没 overlap 都试一试, 看你的具体场景哪个好.

除了大小和重叠, 还有一个维度: 在哪里切? ↓
(5)

在哪里切: 按字数 vs 按段落/句子

最简单的切法是"每 500 字切一刀". 但更聪明的切法是尊重文本结构.

固定大小
→
每 N 个字/token 切一刀. 简单粗暴, 可能切断句子.
Recursive
→
先按段落切, 段落太长再按句子切, 再太长按逗号切. LangChain 默认方式, 最推荐.
语义切分
→
用 Embedding 检测"话题转折点", 在转折处切. 精确但慢. → U4 详讲.
页面级
→
PDF 按页切. 适合排版固定的文档 (医学指南、法规). NVIDIA 测试中准确率最高.

PM 决策: 从 Recursive Character Splitting 开始, 设 chunk_size=512, overlap=50~100. 跑一轮检索测试. 如果效果不够好, 再试语义切分或调参.

来看看不同 Chunk Size 的效果差异 ↓
(6)

大 Chunk vs 小 Chunk: 效果对比

同一个问题 "经期能吃冰吗?", 用不同 Chunk Size 检索, 结果天差地别.

小 Chunk (200 字)

  • 检索到: "经期应避免寒凉食物, 如冰淇淋..."
  • 精准命中 — 但缺乏前因后果
  • Chunk 数量多 → 存储大
  • 适合: FAQ, 简短事实性问题

大 Chunk (1000 字)

  • 检索到: 整篇"经期饮食"章节
  • 上下文丰富 — 但包含很多无关内容
  • Chunk 数量少 → 存储小
  • 适合: 复杂分析型问题

关键发现 (NVIDIA 2024):

  • 事实性问题 (谁? 什么时候?) → 小 Chunk (256 tokens) 效果好
  • 分析性问题 (为什么? 怎么做?) → 大 Chunk (1024 tokens) 效果好
  • 混合场景 → 512 tokens 是最佳平衡点
还有一个常见问题: 短文档需要切吗? ↓
(7)

什么时候不需要 Chunking?

不是所有文档都需要切. 有些时候, 不切反而更好.

不需要切的场景:

  • FAQ 列表 — 每个 Q&A 本身就很短 (50~200 字)
  • 产品描述 — 每个产品一段, 天然是独立的
  • 客服工单 — 每个工单就是一个完整的上下文

必须切的场景:

  • 长篇医学指南 (几千到几万字)
  • 法律法规文档
  • 技术文档手册

规则: 文档 > 500 字且覆盖多个话题 → 切. 文档短且单一主题 → 不切.

(9)

Vitamin 的 Chunking 策略

Vitamin 的知识库有多种文档类型, 需要不同的 Chunking 策略.

医学指南 PDF
→
按页 + Recursive 切, 512 tokens, 15% overlap. 保留章节标题作为 metadata.
FAQ 列表
→
每个 Q&A 作为一个 chunk, 不切. 天然是独立的.
中医养生文
→
按段落切, 400 tokens. 中医文本往往一段一个方子, 天然分段.
营养学手册
→
每种食物/营养素一个 chunk. 表格类内容特殊处理.

Vitamin Chunking 建议:

  • 默认: Recursive Character Splitting, 512 tokens, overlap 80 tokens
  • 每个 chunk 保留 metadata: 来源文档名 + 章节标题 + 页码
  • FAQ 和短文档: 不切, 整条存
  • 上线后监控: 看哪些 chunk 被检索到但用户不满意 → 调参

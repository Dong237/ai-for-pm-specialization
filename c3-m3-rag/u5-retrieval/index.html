向量搜索的"噪音"问题

你问 "经期能吃冰吗?", 向量搜索 Top-7 里可能出现:

#1 经期饮食指南 (相关 ✓)

#2 夏季饮品推荐 (不相关 — 只是都提到了"冰")

#3 中医养生 (相关 ✓)

#4 冰淇淋制作方法 (不相关 — 只是"冰"这个词的表面匹配)

#5 经期运动建议 (部分相关)

向量搜索按"距离"排序 — 但距离近不一定真的"语义相关". 它容易被表面词汇骗.

问题清楚了. 怎么解决? ↓
(2)

两阶段检索: 先广后精

核心思路

粗筛快, 精排准

Phase 1: 用快速的 Bi-Encoder 搜出 Top-50 候选 (粗筛, 重recall)

Phase 2: 用慢但准的 Cross-Encoder 精排到 Top-5 (精排, 重precision)

就像招聘: 先 HR 粗筛简历 (快但可能漏), 再业务面试精选 (慢但准).

Bi-Encoder 和 Cross-Encoder 有什么区别? ↓
(3)

Bi-Encoder vs Cross-Encoder

Bi-Encoder (Phase 1 · 粗筛)

  • 原理: 问题和文档分别编码成向量, 算距离
  • 速度: 极快 — 文档向量可以预计算, 查询只需一次编码
  • 精度: 中等 — 把整个文档意思压进一个向量, 信息有损失
  • 用途: 从百万文档中快速筛出 Top-50

Cross-Encoder (Phase 2 · 精排)

  • 原理: 问题+文档拼在一起, 整体打分 (0~1)
  • 速度: 慢 — 每对(问题,文档)都要单独跑一次模型
  • 精度: 高 — 能看到问题和文档的交互, 理解更深
  • 用途: 对 Top-50 精排到 Top-5

为什么不直接用 Cross-Encoder? 因为太慢了. 知识库有 100 万个 chunk, 每个都要和问题配对跑一次 Cross-Encoder = 100 万次推理 = 不可接受的延迟. 所以先用 Bi-Encoder 筛到 50 个, 再 Cross-Encoder 处理 50 个 — 只需要 50 次推理.

关键发现: Top-K 越小, Reranking 越重要. 因为你给 LLM 的 chunk 越少, 每个 chunk 的质量就越关键. Reranker 确保这 3~5 个 chunk 真的是最相关的.

除了 Reranking, 还有什么方法提升检索? ↓
(5)

Hybrid Search: 向量 + 关键词 双保险

纯向量搜索会漏掉精确关键词匹配. 纯关键词搜索不懂语义. 混合搜索 = 两者结合.

向量搜索
→
擅长: 语义理解 ("吃冰" ≈ "寒凉食物"). 弱点: 可能漏掉精确词.
关键词 (BM25)
→
擅长: 精确匹配 ("叶酸" = "叶酸"). 弱点: 不懂同义词.
Hybrid
→
两个列表合并, 取并集. RAG 质量提升 15~30% (行业共识).

最佳实践: Hybrid Search (Top-50) → Reranking (Top-5) → LLM. 这是 2025-2026 年最被推荐的 RAG 检索管线.

那 Top-K 这个 K 设多少好? ↓
(6)

K 设多少: 粗筛 K vs 精排 N

两个 K 不一样: 粗筛的 K (50~100) 和最终给 LLM 的 N (3~5).

推荐设置:

  • 粗筛 K = 50~100 — 足够大, 确保不遗漏相关文档
  • 精排 N = 3~5 — 给 LLM 的 chunk 数量. 太多增加噪音, 太少可能缺信息
  • K 太小 → 可能漏掉相关文档 (recall 下降)
  • K 太大 → Reranker 需要处理更多, 延迟增加

Vitamin 建议:

  • 医学问答: 粗筛 K=50, 精排到 N=5 (准确优先)
  • 客服 FAQ: 粗筛 K=20, 精排到 N=3 (速度优先)
  • 健康周报: 粗筛 K=30, 精排到 N=5 (信息完整优先)
Query Expansion: 有时候问题本身不够好 ↓
(7)

Query Expansion: 改写问题, 搜得更全

用户的问题往往不够"专业". 比如 "经期能吃冰吗?" — 知识库里可能用的是 "寒凉食物" 而不是 "冰".

Query Expansion 策略:

  1. 同义词扩展: "吃冰" → "吃冰" + "寒凉食物" + "冷饮"
  2. HyDE: 先让 LLM 生成一个"假回答", 用假回答的向量去搜 (因为假回答里会包含专业术语)
  3. 多角度改写: 把一个问题改写成 3 种说法, 各搜一遍, 合并结果

Query Expansion 提升 recall 的效果往往比调 chunk size 还大.

(9)

Vitamin 的检索管线设计

为 Vitamin 搭建一个完整的检索管线:

Vitamin 检索架构:

  1. Hybrid Search: 向量搜索 (bge-large-zh) + BM25 关键词搜索
  2. 粗筛 Top-K=50: 两路合并, 取并集
  3. Reranking: Cross-Encoder 精排到 Top-5
  4. Augment: Top-5 chunks + 用户问题 → LLM prompt

特殊处理:

  • 医学问答: 加入 Query Expansion (同义词 + 专业术语)
  • 安全过滤: 对涉及用药/诊断的问题, 额外检查是否有"不确定"标记
  • Metadata 过滤: 按文档类型、时间筛选 (优先最新指南)

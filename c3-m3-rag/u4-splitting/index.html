语义 vs 固定切分

按字数切 vs 按意思切 — 哪种更聪明?

上节课学了"按字数切" — 简单但会切断句子...

有没有更聪明的切法, 能自动在"话题切换"的地方断开?

—— 今天对比两种切法, 看差距.

(1)

固定切分的尴尬: 切在了最不该切的地方

还记得上节课的例子吗? 一篇经期饮食文档, 按 120 字一切:

Chunk 3 (固定切分):

"...生姜红糖水，是很多女性经期的首选饮品。第三，要注意补充铁质。经期失血较多，容"

这个 chunk 混了两个话题: "温热食物" 的尾巴 + "补铁" 的开头. 如果用户问"经期喝什么好?", 这个 chunk 可能被检索到, 但里面一半内容是关于铁的 — 噪音!

语义切分就是来解决这个问题的 ↓
(2)

语义切分: 找到"话题转折点"

原理

语义切分的核心思路:

在"意思变了"的地方切

把每个句子 Embed 成向量, 比较相邻句子的相似度. 相似度突然下降 = 话题变了 = 在这里切.

具体步骤:

  1. 把文档拆成一个个句子
  2. 每个句子 → Embedding 向量
  3. 计算相邻句子向量的 Cosine Similarity
  4. Similarity 低于阈值 → 在这里切一刀
  5. 结果: 每个 Chunk 都是一个完整的话题
这个"阈值"怎么选? ↓
(3)

阈值: 语义切分的调参核心

阈值太高 → chunk 太大 (不够切). 阈值太低 → chunk 太碎 (切太多).

三种阈值选择方法:

  • 百分位 当相邻句子的相似度差异超过某个百分位 (如 P75) 时切 — 自适应, 推荐
  • 标准差 差异超过均值 + N 个标准差时切 — 适合话题分布均匀的文档
  • 固定值 直接设一个 cosine similarity 阈值 (如 0.5) — 简单但不灵活

大部分工具 (LlamaIndex, LangChain) 默认用百分位法, 开箱即用.

说了半天原理, 来看看实际效果差多少 ↓
(4)

效果对比: 精确度提升了多少?

Chroma (向量数据库厂商) 2025 年的测试数据:

固定切分
→
Recall@10 = 85~90% (400 tokens, Recursive)
语义切分
→
Recall@10 = 91~92% — 提升 2~3%
成本代价
→
语义切分需要对每个句子做 Embedding — 计算量大很多

PM 视角: 2~3% 的提升值不值得? 取决于场景.

  • 医疗问答 (错一个可能出事) → 值, 多花钱换准确率
  • 客服 FAQ → 不值, 固定切分已经够好
还有一个问题: Springer 的研究说语义切分不一定更好? ↓
(5)

反直觉: 语义切分不一定赢

Springer Nature (学术出版商) 2025 的研究发现:

LlamaIndex SemanticSplitter 的效果, 居然略低于固定切分的 baseline.

原因: 语义切分的阈值很难调. 调得不好, 切出来的 chunk 要么太大, 要么太碎, 反而不如简单的固定切分 + overlap.

这告诉 PM 什么?

  • "高级"不等于"更好" — 先用简单方法, 不行再升级
  • Recursive Character Splitting 是经过验证的默认选择
  • 语义切分只在你有精力调参、且文档话题跨度大时才值得
  • 最终衡量标准是检索准确率, 不是切分方法的"高级程度"
实际项目中, 大多数团队怎么做? ↓
(6)

行业最佳实践: 渐进式升级

成熟的 RAG 团队一般这样做:

Phase 1: 先跑通 (1~2 周)
  1. 用 Recursive Character Splitting, 512 tokens, 15% overlap
  2. 跑一批测试 query, 看检索准确率
  3. 如果 Recall > 85%, 够用了, 先上线
→ 快速验证, 不纠结切分方法
↓ 如果效果不够好 ↓
Phase 2: 针对性优化 (持续)
  1. 分析: 哪些 query 检索不准? 是 chunk 太大还是切断了?
  2. 对问题最多的文档类型试语义切分
  3. A/B 测试: 固定 vs 语义, 看指标
  4. 可能还需要: 混合策略 (不同文档类型用不同方法)
→ 数据驱动, 逐步提升
还有一些新兴切分方法值得知道 ↓
(7)

前沿: 2026 年的新切分方法

Chunking 领域还在快速发展. 几个值得关注的新方向:

Late Chunking
→
先把整篇文档 Embed, 再切分向量. 保留了全局上下文.
Hierarchical
→
分层切分: 大 chunk 提供概要, 小 chunk 提供细节. 像书的章 → 节 → 段.
Adaptive
→
根据文档内容动态调整 chunk size. 密集段落切小, 松散段落切大.
LLM-based
→
用 LLM 来判断切分点. 效果好但贵 — 适合高价值文档.

PM 不需要追这些前沿方法. 先把 Recursive + 语义切分用好. 新方法的收益通常在 1~5%, 但复杂度增加很多.

回到 Vitamin ↓
(9)

Vitamin 切分策略选型

不同文档类型, 适合不同的切分方法.

医学指南
→
语义切分 话题丰富, 值得精确切分. 每个话题一个 chunk.
FAQ
→
不切 每条 Q&A 天然是一个 chunk.
中药方剂
→
固定 + 段落 每个方剂一段, 结构规整, Recursive 够用.
用户反馈
→
不切 每条反馈通常 < 200 字, 整条存.

Vitamin 混合策略: 默认 Recursive (512 tokens, 15% overlap), 医学长文档加语义切分, FAQ 和短文档不切.

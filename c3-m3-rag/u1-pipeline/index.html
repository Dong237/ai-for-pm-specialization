(1)

LLM 的致命弱点: "不知道它不知道的事"

你问 ChatGPT "你们公司昨天发了什么公告?" — 它会自信地编一个. 因为它训练时没见过你的内部文档.

场景 1

"经期能吃冰吗?"

LLM 可能说"完全没问题" — 但你们的医学顾问说"建议少吃". 谁对?
场景 2

"Vitamin 会员怎么退款?"

LLM 编了一个退款流程 — 跟你们实际政策完全不一样.
场景 3

"这个月哪些食物对备孕好?"

LLM 给出泛泛建议, 没有引用任何医学指南.

核心问题: LLM 的知识是训练时"冻住"的. 它不知道你的产品文档、最新医学指南、用户数据. 它只能靠"记忆"回答, 记忆不够就编.

那怎么办? 有一个简单到不像话的方案 ↓
(2)

RAG 的核心思路: 开卷考试

想象一下考试. LLM 平时都是闭卷考试 — 全靠脑子里的记忆. RAG 做的事就是: 让它变成开卷考试.

📕

闭卷考试 (纯 LLM)

LLM 只用训练时学到的"记忆"回答. 记忆里没有的东西, 它就编.

  • 优点: 快, 不需要额外基础设施
  • 缺点: 会编 (Hallucination), 不能引用来源, 知识会过时
📖

开卷考试 (RAG)

回答之前先搜知识库, 把相关资料塞进 prompt. LLM 基于资料回答.

  • 优点: 能引用来源, 知识随时更新, 幻觉大幅减少
  • 缺点: 需要搭知识库, 检索质量是瓶颈
一句话

RAG = Retrieval-Augmented Generation

先搜, 再答.

Retrieval(检索) + Augmented(增强) + Generation(生成). 就是给 LLM 加一个"资料搜索"步骤.

这三个字 R-A-G 分别对应什么? 拆开看 ↓
(3)

三步走: Embed → Retrieve → Augment + Generate

RAG 的核心就三步. 记住这三步, 你就理解了 RAG 的 80%.

Step 1 · Embed 把知识库里的文档和用户问题都转成数字向量 (Embedding). 这样才能"比较"它们的相似度.
Step 2 · Retrieve 用向量相似度, 在知识库里找到最相关的几个文档片段. 这一步是 RAG 的核心 — 也是最容易出错的地方.
Step 3 · Augment + Generate 把检索到的片段塞进 LLM 的 prompt, 和用户问题一起发给模型. LLM 基于这些"参考资料"生成回答.

类比: 就像你在百度上搜一个医学问题, 点进前 3 个结果看了一遍, 然后用自己的话总结给朋友听. RAG 做的就是这件事 — 只不过搜索和总结都是自动的.

光说不够 — 走一遍具体例子 ↓
(4)

走一遍: "经期能吃冰吗?" 的 RAG 旅程

假设 Vitamin 已经搭好了一个医学知识库. 用户问了这个问题, 看看 RAG 怎么处理.

准备工作 (离线, 只做一次):

  1. 把所有医学文档 (PDF / Word / Markdown) 切成小段 (Chunks)
  2. 每个 Chunk 用 Embedding 模型转成向量
  3. 向量存进向量数据库 (Vector DB), 等着被搜索

→ 这就像图书馆先给每本书编好索引, 方便将来查找.

用户提问时 (实时, 每次都做):

  1. Embed: "经期能吃冰吗?" → 变成一个 768 维向量
  2. Retrieve: 拿这个向量去知识库搜, 找到 3 个最相关的 Chunk
  3. Augment: 把 3 个 Chunk + 用户问题 打包成一个 prompt
  4. Generate: prompt 发给 LLM, LLM 读完资料后回答
说了这么多, 来动手试试这个流程 ↓
(5)

RAG 的两个阶段: 离线建库 vs 在线查询

RAG 分两个时间线. 搞清楚"什么时候做什么", 你就不会搞混.

阶段 1 · 离线建库 (一次性, 定期更新)
  1. 收集文档: PDF、Word、网页、数据库里的内容
  2. 切分 (Chunking): 把长文档切成 200~500 token 的小段
  3. 向量化 (Embedding): 每段文本 → 一个数字向量
  4. 存储: 向量存进 Vector Database (Pinecone / Weaviate / Milvus)
→ 一个可搜索的"知识向量库"
↓ 用户来提问时 ↓
阶段 2 · 在线查询 (每次用户问, 跑一次)
  1. 用户问题 → Embedding 向量
  2. 向量在知识库里搜索 → Top-K 个最相关 Chunk
  3. Chunk + 问题 → 拼成 prompt
  4. prompt → LLM → 生成回答 + 引用来源
→ 有依据的回答 + 来源链接
为什么不直接 Fine-tune 模型? RAG 比 Fine-tuning 好在哪? ↓
(6)

RAG vs Fine-tuning — PM 必须知道的选型

你可能会问: "为什么不直接把文档喂给模型训练?" 因为成本和灵活性差太多了.

更新速度
→
RAG: 加一篇文档到知识库, 几分钟生效. Fine-tune: 重新训练, 几小时到几天.
成本
→
RAG: 向量库存储费 + API 调用费, 每月几百到几千. Fine-tune: GPU 训练费, 一次几千到几万.
可追溯
→
RAG: 每个回答都能标注"来自哪篇文档". Fine-tune: 知识混在权重里, 无法追溯.
幻觉控制
→
RAG: 检索到的资料 = 事实锚点, 幻觉减少. Fine-tune: 模型可能"记错"训练数据.

PM 决策规则:

  • 知识需要频繁更新 → RAG
  • 需要引用来源 → RAG
  • 需要模型学会新的输出风格/格式 → Fine-tune
  • 最佳方案: RAG + 轻量 Fine-tune 结合
RAG 听着很美, 但它最容易出错的环节是哪里? ↓
(7)

RAG 失败的 73% 原因: 检索不准

RAG 不是万能的. 2026 年的行业数据显示: 当 RAG 回答出错时, 73% 的情况是检索环节出了问题 — 不是 LLM 的锅.

检索问题 73%
其他 27%

检索问题: 搜到了不相关的文档, 或者漏掉了关键文档

其他: LLM 理解错误、prompt 设计不当等

这意味着什么?

作为 PM, 你花 80% 精力应该放在优化检索质量, 而不是挑选更贵的 LLM. 接下来 7 个 Unit (U2-U8) 全部都是在教你怎么把检索做好.

  1. Embedding — 向量质量决定搜索质量
  2. Chunking — 切片大小影响检索粒度
  3. Splitting — 语义切分 vs 固定切分
  4. Retrieval + Reranking — 精选最相关的结果
  5. Citation — 让回答可追溯
  6. Evaluation — 量化检索和生成质量

(9)

Vitamin 的 RAG 应用场景

Vitamin 是女性健康 AI 应用, 有大量医学文档和健康指南. 这些正是 RAG 的最佳素材.

适合用 RAG 的场景

  • 用户问"备孕期间能吃叶酸吗?" → 检索医学指南回答
  • "月经周期不规律怎么办?" → 检索妇科文档 + 标注来源
  • "这个中药方子有什么副作用?" → 检索中药数据库
  • 客服问"会员怎么退款?" → 检索内部政策文档
  • 健康周报生成 → 检索用户历史数据 + 健康知识

不适合 RAG 的场景

  • 单纯的情绪安抚对话 → 不需要检索, LLM 直接聊
  • 计算 BMI / 排卵日 → 用规则引擎, 不需要 RAG
  • 实时天气/新闻查询 → 需要 API 调用, 不是知识库
  • 用户隐私数据分析 → 合规限制, 不能存进通用知识库

Vitamin RAG 知识库可能包含:

  • 妇科医学指南 (经期、孕期、更年期等)
  • 营养学手册 (食物成分、禁忌)
  • 中医养生文档 (体质辨识、食疗方案)
  • 产品 FAQ 和客服手册
  • 用户反馈和常见问题汇总

→ U8 会详细设计这个知识库结构.

Prompt Caching
省钱大杀器

你的 system prompt 每次都一样, 但每次都按原价收费? Prompt Caching 让它只算一次.

我的 system prompt 有 300 token, 每次调用都带着它...

一天 5000 次调用, 光 system prompt 就花了 300×5000 = 150 万 token!

—— Prompt Caching: 让这 150 万 token 只收 2% 的钱.

(1)

每次调用都带同样的 system prompt — 好浪费

Vitamin 的每次 API 调用大概长这样:

System Prompt (固定, ~300 tokens)
"你是 Vitamin 健康助手. 你擅长女性健康话题... 回复风格: 温柔简洁..."
对话历史 (半固定, ~200 tokens)
用户之前说过的话 + AI 之前的回复
当前用户输入 (变化, ~50 tokens)
"经期可以跑步吗?"

问题: 那 300 token 的 system prompt, 每次都一模一样, 但每次都按原价收费!

一天 5000 次调用 × 300 token = 150 万 token 的浪费.

Prompt Caching 就是来解决这个浪费的 ↓
(2)

Prompt Caching 是怎么工作的

原理

LLM 处理 prompt 时, 会为每个 token 计算 KV (Key-Value) 向量.

Prompt Caching 把这些 KV 向量存起来. 下次同样的前缀来了, 直接复用, 不重新计算.

→ 你不需要做任何配置. 发送请求时, API 自动检测前缀是否匹配, 匹配就用缓存, 不匹配就正常计算.

没有缓存

每次请求 → 重新算 system prompt 的 KV → 全价

300 token × 原价
有缓存

前缀匹配 → 复用已有 KV → 缓存价

300 token × 缓存价 (低至原价 2%)

关键: 缓存不影响模型输出质量. 它只是跳过了重复的计算, 结果完全一样.

缓存只对 prompt 开头有效 — 这是关键 ↓
(4)

前缀匹配: 缓存只从开头算

Prompt Caching 用的是 前缀匹配 — 只有 prompt 从头开始完全一致的部分才会被缓存.

✓ 前缀相同 → 缓存命中
请求 A: [System Prompt] + [用户问题 1]
请求 B: [System Prompt] + [用户问题 2]
→ System Prompt 部分命中缓存!
✗ 前缀不同 → 缓存失效
请求 A: [System Prompt A] + [问题]
请求 B: [System Prompt B] + [问题]
→ 前缀不同, 缓存完全失效!

PM 要点: 让 system prompt 在所有请求中保持 完全一致. 不要在 system prompt 里塞动态内容 (时间戳 / 用户名等). 把动态内容放在 user message 里.

怎么设计缓存友好的 prompt? ↓
(5)

缓存友好的 Prompt 设计原则

黄金法则: 稳定在前, 变化在后

  1. System Prompt (最稳定) → 放最前面 → 100% 缓存
  2. 工具定义 / Few-shot examples (稳定) → 紧跟 system → 大部分缓存
  3. 对话历史 (半稳定) → 中间 → 部分缓存
  4. 当前用户输入 (每次变) → 放最后 → 不缓存
✗

坏设计

"你是助手. 现在是 2026-05-15 14:23. 用户叫小明. 请帮她..."

→ 时间戳 + 用户名在 system prompt 里, 每次都不同, 缓存失效!

✓

好设计

System: "你是 Vitamin 健康助手..."
User: "时间: 2026-05-15. 我叫小明. 经期可以跑步吗?"

→ System prompt 稳定, 动态信息放 user message. 缓存命中!

对话历史怎么处理? ↓
(6)

多轮对话的缓存: 越来越长 = 越来越便宜

多轮对话时, 每轮新增的 token 是 "增量", 之前的全部可以缓存.

第 1 轮: System (300) + User 1 (50) = 350 token → 全部 miss

第 2 轮: System (300) + User 1 (50) + AI 1 (200) + User 2 (50) = 600 token

→ 前 350 token 缓存命中, 只有 250 token miss

第 3 轮: 前 600 token 缓存, 只有新增的 ~250 token miss

→ 对话越长, 缓存比例越高, 单轮边际成本越低!

但注意: 如果你每轮都重新组装 prompt (比如截断历史), 缓存可能被破坏. 尽量保持前缀稳定.

怎么知道缓存有没有生效? ↓
(7)

监控缓存: cache_hit_tokens vs cache_miss_tokens

DeepSeek / Anthropic 的 API 返回里有专门的字段, 告诉你缓存情况.

"usage": {
  "prompt_cache_hit_tokens": 298,
  "prompt_cache_miss_tokens": 52,
  "completion_tokens": 180
}

PM 该关注的指标:

缓存命中率 = hit / (hit + miss) = 298 / 350 = 85%

→ 目标: 70%+ 就算不错, 90%+ 就非常好.

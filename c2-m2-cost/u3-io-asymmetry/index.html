为什么输出更贵? 原因在 GPU 的计算方式 ↓
(2)

Prefill vs Decoding — 技术原因

Prefill (处理输入)

把你的全部输入一次性并行送进 GPU. 大批量并行计算, GPU 利用率高, 成本低.

= 便宜 (Input Price)
Decoding (生成输出)

一个 token 一个 token 串行生成. 每生成一个字都要等上一个字. GPU 利用率低, 成本高.

= 贵 (Output Price)

类比: 输入 = 批发 (一卡车同时运), 输出 = 零售 (一件一件发). 零售当然比批发贵.

这意味着什么? 控制输出量 = 省钱 ↓
(3)

输出占了你账单的大头

一条典型的 Vitamin 对话: 输入 400 tokens (system + user), 输出 300 tokens (回复). 算算看:

以 Claude Sonnet 4.6 ($3/$15) 为例:

  • 输入费: 400 ÷ 1M × $3 = $0.0012
  • 输出费: 300 ÷ 1M × $15 = $0.0045
  • 总计: $0.0057

输出只占总 token 数的 43%, 但占了总费用的 79%!

控制输出量 =
控制你 80% 的账单.

怎么控制输出量? 4 种手段 ↓
(4)

控制输出量的 4 种手段

1

设 max_tokens

API 参数直接限制输出上限. 超出就截断.

2

Prompt 指令约束

"请用 3 句话总结", "回复不超过 100 字", "只返回 JSON".

3

结构化输出格式

用 JSON / 固定模板, 避免模型自由发挥写长文.

4

分步骤拆解

不要一次让模型做 5 件事. 拆成 5 次短输出, 总量可能更少.

先看第一种 — max_tokens ↓
(5)

max_tokens: 给输出上一道锁

所有 API 都支持 max_tokens 参数 — 设了它, 模型生成到上限就自动停止.

Vitamin 推荐设置:

  • 情绪分类 max_tokens = 20 (只需返回一个标签)
  • 简短回复 max_tokens = 150 (日常对话)
  • 健康建议 max_tokens = 300 (需要一些解释)
  • 详细报告 max_tokens = 800 (周报/月报)

注意: max_tokens 设太小会导致回复被截断, 用户体验差. 设太大又浪费钱. 需要针对每个功能调优.

第二种手段: Prompt 指令约束 ↓
(6)

Prompt 约束 + 结构化输出

在 system prompt 里加一句话, 就能把输出量砍掉 40-60%.

Before (无约束)
好的, 关于经期跑步这个问题, 我来详细为你解答. 首先, 经期是否可以跑步取决于个人的身体状况和经期的阶段. 一般来说, 轻度到中度的运动在经期是安全的, 甚至可能有益... (后面还有 200 字)
输出: ~320 tokens · 费用 (Sonnet): $0.0048
After (加约束)
经期轻度跑步通常安全. 建议: 1) 减速减量 2) 注意身体信号 3) 经量大或痛经严重时休息. 如持续不适请咨询医生.
输出: ~80 tokens · 费用 (Sonnet): $0.0012 · 省 75%!

加的那句 prompt:

"请用 2-3 句话简洁回答. 给出具体建议, 不要寒暄和铺垫."

→ 多了 ~15 个输入 token, 但省了 ~240 个输出 token. 净省 $0.0036/次.

看看结构化输出能省多少 ↓
(7)

结构化输出: JSON 比自由文本省 50%

如果你不需要给用户看回复 (比如后台分类/打标), 直接让模型返回 JSON — 比自由文本短 50%+.

自由文本
根据用户的描述, 她的情绪状态可以归类为 "焦虑", 主要表现在对月经不规律的担忧, 以及对是否需要就医的不确定性.
~65 tokens
JSON 输出
{"emotion": "anxious", "confidence": 0.85, "trigger": "irregular_period"}
~22 tokens · 省 66%!

PM 规则: 后台逻辑 (分类 / 打标 / 路由判断) → 用 JSON. 给用户看的回复 → 用自然语言, 但加长度约束.

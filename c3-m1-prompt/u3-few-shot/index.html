Unit 3·
给 AI 看几个例子, 它就聪明了?

Few-shot 是最直觉的 Prompt 技巧 — 但例子不是越多越好.

同事说"多给几个例子效果就好了"

那到底给几个? 10 个够不够?

每多一个例子, 成本也多一点吧?

—— 3 个, 通常就是最优解.

10 步学完, 你会:
  • 理解 Zero-shot / One-shot / Few-shot 的区别
  • 掌握 "3 个例子" 的性价比拐点
  • 知道例子的质量比数量更重要
  • 亲手调滑块看准确率曲线和成本变化
走起 →
(1)

你在工作中一直在用 Few-shot, 只是不知道

想象你招了一个新实习生, 需要他帮你分类用户反馈:

方案 A (Zero-shot): "帮我把用户反馈分成正面/负面/中性."

方案 B (Few-shot): "帮我分类. 比如:

  • 'APP 太好用了!' → 正面
  • '闪退了三次' → 负面
  • '还行吧' → 中性

现在帮我分这条: '最近更新后速度变快了'"

B 方案给了 3 个例子, 实习生立刻就知道你想要什么格式、什么标准. AI 也一样 — 例子是最好的说明书.

专业术语叫什么? ↓
(2)

三个级别: Zero / One / Few

Zero-shot 没有例子 — 只靠指令. 简单任务够用, 复杂任务准确率低.

One-shot 给 1 个例子 — AI 知道你要什么格式了, 但可能过拟合这一个例子.

Few-shot 给 2-5 个例子 — 性价比最高的区间. 覆盖多种情况, 准确率显著提升.

关键数据

GPT-4 的实测数据 (OpenAI 官方):

0 例 → 84.5%   5 例 → 91.5%

→ 就加 5 个例子, 分类准确率提了 7 个百分点. 但从 5 个到 10 个, 只提了不到 1 个点.

递减效应具体长什么样? ↓
(3)

边际效用递减: 3 个之后就平了

Brown et al. (2020) 在 GPT-3 论文里发现的规律:

准确率曲线的形状:
  1. 0 → 1 个: 跳跃式提升 (+16%). AI 第一次知道你要什么.
  2. 1 → 3 个: 稳步提升 (+13%). 覆盖更多场景, 纠正偏差.
  3. 3 → 5 个: 提升放缓 (+3%). 边际效用开始递减.
  4. 5 → 10 个: 几乎持平 (+1%). 多出来的例子 = 多花的钱, 没多少回报.

结论:

3 个例子是性价比拐点.
2 个有点少, 5 个有点多, 3 个刚刚好.

但也有例外情况 ↓
(4)

例外: 什么时候需要更多例子?

需要 5+ 个例子的场景:
  • 多分类任务 (比如 10 个分类, 每个至少 1 个例子)
  • 格式非常特殊 (比如特定 JSON schema, 医学术语)
  • 模型较小 / 较弱 (大模型 3 个够, 小模型可能需要 5 个)
例子太多的副作用 (Over-prompting):
  • 占用 Context Window — 留给用户输入的空间变少
  • 某些小模型反而变差 — 被例子"带偏"了
  • 成本线性增长 — 每个例子 ~40 token, 10 个就是 400 token
比数量更重要的是: 例子的质量 ↓
(5)

质量 > 数量: 3 个好例子胜过 10 个差例子

好例子的 3 个标准:
  1. 覆盖多样性: 3 个例子覆盖 3 种不同的情况, 而不是 3 个相似的.
  2. 包含边界案例: 至少 1 个"不太明显"的例子, 比如"还行吧"这种模糊输入.
  3. 格式一致: 所有例子的输入输出格式完全一致, AI 才能学到模式.
✗

差: 3 个例子都是"正面"

"太好了!" → 正面
"很赞!" → 正面
"好棒!" → 正面

AI 只见过正面, 遇到负面不知道怎么分.
✓

好: 3 个例子覆盖 3 种类型

"太好了!" → 正面
"闪退了" → 负面
"还行吧" → 中性

每种类型一个, 覆盖最大.
例子的顺序也有讲究 ↓
(6)

例子的顺序影响结果

研究发现, 同样的 3 个例子, 换一个排列顺序, 准确率可能差 10%+.

排序建议:
  1. 最典型的放前面: 先展示最常见的情况
  2. 最难的放最后: 复杂边界案例放最后, 给 AI "加深印象"
  3. 避免连续同类: 不要连续放 3 个正面, 交叉排列
Vitamin 示例排序建议:

经期(最常见) → 安全期(对比) → 经前期(边界, 容易误分)

→ 先教 AI 什么是"典型", 再教它什么是"容易搞混的".

Few-shot 在 Prompt 中该放哪里? ↓
(7)

Few-shot 在 4 段式中的位置

回忆 U1 的 4 段式 + U2 的 System/User 分配:

完整结构:
/* System Prompt */ ## Role 你是 Vitamin 的健康顾问... ## Task 根据用户输入, 分类身体状态... ## Examples 输入: "今天肚子胀" → 经前期 · 日常建议 输入: "好累啊" → 经期 · 关怀建议 输入: "精力超好" → 排卵期 · 运动建议 ## Format JSON: {"state":"...","mood":"...","type":"..."} /* User Prompt */ 用户输入: "最近睡得不太好, 有点烦"

Examples 放在 Task 和 Format 之间, 属于 System Prompt, 因为例子不会因用户不同而变化, 可以缓存.

Vitamin 场景怎么用 Few-shot ↓
(9)

Vitamin 实战: 5 状态分类的 Few-shot 策略

Vitamin 有 5 个身体状态. 最小方案: 每种状态 1 个例子 = 5 个 Few-shot. 但我们可以优化到 3 个.

推荐: 3 个精选例子

  • 经期(最常见, 必须有)
  • 排卵期(与经期反差大, 帮助对比)
  • 经前期(最容易误分, 边界案例)

不推荐: 10 个例子

  • 占 ~400 token, 每次调用多花钱
  • Context Window 被占用, 用户历史信息放不下
  • 准确率只比 3 个多 ~2%, 不值得
PM 决策框架:

先用 3 个 Few-shot 上线 → 监控分类准确率 → 如果某个状态准确率低 → 针对性增加那个状态的例子, 而不是无脑加数量.

总结 + 作业 ↓
(10)

一句话规则 + 今日作业

一句话

3 个好例子 > 10 个烂例子.
覆盖多样性, 包含边界, 格式一致.

Few-shot 的性价比拐点在 3 个. 超过 5 个要有充分理由.

课后作业 (30 分钟):
  1. 为 Vitamin 的"症状分类"任务写 3 个 Few-shot 示例 (覆盖经期/排卵期/经前期).
  2. 在 ChatGPT 里测试: 同一个分类任务, 分别用 0-shot、1-shot、3-shot 试, 记录准确率.
  3. 找一个你工作中的分类/提取任务, 写 3 个高质量 Few-shot 示例.
  4. 写 100 字: 给产品团队解释"为什么我们只需要 3 个例子, 不需要 10 个".

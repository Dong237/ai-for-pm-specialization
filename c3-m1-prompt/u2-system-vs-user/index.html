(1)

API 调用里的两个"信封"

当你用 API 调用 Claude 或 GPT, 请求体里有两个关键字段:

System Prompt

比喻: 像公司的"员工手册" — 入职时发一次, 之后每次上班都默认遵守.

特点:

  • 设置一次, 多次对话都不变
  • AI 把它当作"最高优先级指令"
  • 用户看不到这段内容
  • 可以被缓存, 节省成本

User Prompt

比喻: 像用户每天发来的"工单" — 每次不同, 包含这次的具体需求.

特点:

  • 每次对话/请求都不同
  • 包含用户的具体输入和上下文
  • 用户直接可见
  • 每次都要重新处理, 不能缓存
那 4 段式的 RTCF 分别该放哪个信封? ↓
(2)

分配原则: "变不变?" 一个问题就够了

核心原则

问自己一个问题:

"这段内容, 下一个用户来了还是一样的吗?"

是 → 放 System. 不是 → 放 User.

4 段式的典型分配:
  • System Role — 角色不会因用户不同而变
  • System Task — 核心任务描述通常不变
  • User Context — 每个用户的状态/输入都不同
  • System Format — 输出格式通常固定

→ 4 段里有 3 段是不变的, 只有 Context 每次变化. 这意味着75% 的 Prompt 可以被缓存!

为什么缓存这么重要? 看一笔账 ↓
(3)

Prompt Caching: 同一段话, 只付一次钱

Prompt Caching 是 Anthropic 和 DeepSeek 提供的省钱机制:

原理:

当你的 System Prompt 不变时, API 只在第一次完整处理它, 之后的请求直接从缓存读取. 缓存命中时, 这部分 token 的费用打折 90%.

不用缓存 (全放 User)

$500/月
1000 DAU × 5次/天 × 800 token System
每次都重新处理

用缓存 (分 System/User)

$95/月
System 部分缓存命中 90% 折扣
只有 User 部分每次付全价

省了 81%. 对于 Vitamin 这种高频调用的产品, 一年能省几万块.

但不是所有内容都适合放 System — 有陷阱 ↓
(4)

常见错误: 把变化的内容塞进 System

✗

错误: 把用户信息放进 System

"把用户的经期天数放 System 不是更方便?"

后果: 每个用户的经期天数不同, System 每次都变 → 缓存永远命中不了 → 白白浪费钱.
✗

错误: 把规则放进 User

"反正放哪里效果都一样."

后果: System 级指令的优先级更高, AI 更不容易违反. 放 User 里, 规则被用户输入"冲淡"的风险更大.
还有一些边界情况 ↓
(5)

边界情况: Few-shot 和安全规则放哪里?

Few-shot 示例 → System

如果你的示例是通用的(不依赖具体用户), 放 System 更好 — 可以缓存, 而且 AI 会把示例当作"更权威"的参考.

安全规则 → 必须放 System

"不要给出医疗诊断"、"不要回答政治话题" — 这些安全红线必须在 System 里. 因为:

  1. System 级指令优先级更高
  2. 用户无法看到或篡改 System Prompt
  3. 可以缓存, 不增加额外成本
动态 Context → User

用户输入、当前时间、用户历史记录、当前身体状态 — 这些每次都不同的信息, 放 User.

(7)

多轮对话的 System 不变原则

在多轮对话中, System Prompt 在第一轮设置, 之后每一轮都自动携带, 不需要重复发送.

对话流程:

第 1 轮 System: 角色+任务+格式+规则 | User: "我今天头疼"

第 2 轮 System: (自动延续) | User: "需要吃什么?"

第 3 轮 System: (自动延续) | User: "可以运动吗?"

→ System 只传一次, 后续都缓存. 这就是为什么放对位置能省这么多钱.

(9)

Vitamin 实战: System + User 完整模板

System Prompt (不变, 可缓存):

## Role 你是 Vitamin App 的健康顾问"小 V". 专注女性周期健康. 说话温暖、不说教、像闺蜜. ## Task 根据用户当前身体状态和描述, 给出当天的个性化健康建议. ## Rules - 不要给出医疗诊断或用药建议 - 不要回答与健康无关的话题 - 如果用户描述的症状严重, 建议就医 ## Format JSON: {"greeting":"...","tips":["..."],"warning":"..."}

User Prompt (每次变化):

当前状态: 经期第 3 天 用户输入: "今天肚子有点疼, 能跑步吗?" 过往记录: 周期 28 天, 经期通常 5 天

这样分的好处

  • System 部分 ~200 token 可缓存, 一年省数万元
  • 安全规则在 System 级别, 不会被用户输入绕过
  • 新增功能只需改 User 模板, System 保持不动

要注意的

  • System 改了 → 所有用户的缓存都失效, 谨慎修改
  • System 太长 → 缓存占用内存大, 控制在合理长度
  • 部分模型对 System Prompt 的遵从度不同, 需要测试

Prompt 也要版本管理

改了一个字, 效果可能天差地别. 把 Prompt 当代码管理, 是生产级 AI 产品的标配.

上周改了一句 Prompt, 今天用户投诉变多了...

改了什么? 谁改的? 能回滚吗?

—— 如果没有版本管理, 这些问题无解.

10 步学完, 你会:
  • 理解为什么 Prompt 需要像代码一样管理
  • 掌握 Semantic Versioning 在 Prompt 上的应用
  • 知道如何设计 A/B 测试验证 Prompt 改动
  • 亲手在版本时间线里查看 diff 和指标
走起 →
(1)

一个真实的事故: "改了一个字, 炸了"

某 AI 产品的 PM 把 Prompt 里的"建议"改成了"应该". 用户反馈语气变得"很强硬". 但没人记录这个改动, 花了 3 天才定位问题.

Prompt 不像代码:

代码改错了, 编译器报错. Prompt 改错了, 没有报错, 只有用户投诉. 而且很难定位"哪里改错了".

为什么传统的代码管理不够? ↓
(2)

Prompt 管理 vs 代码管理: 有什么不同?

代码: 改了 → 跑测试 → 确定性结果. 同样的输入, 永远同样的输出.
Prompt: 改了 → 跑测试 → 非确定性结果. 同样的输入, 可能不同的输出. 需要统计评估, 不能只看一次.
核心

Prompt 管理 = 代码管理 + 统计评估

→ 除了版本号和 diff, 还需要 A/B 测试和指标对比.

Semantic Versioning 怎么用在 Prompt 上? ↓
(3)

Semantic Versioning: Major.Minor.Patch

版本号含义Prompt 场景
Major (1.0.0)破坏性变更输出格式改了 / 角色重新定义 / 整体重写
Minor (1.1.0)新增功能加了 Few-shot 示例 / 加了新的安全规则
Patch (1.1.1)小修小补改了一个错别字 / 调整了一句话的措辞
Vitamin 示例:

v1.0.0 — 初版上线, 4 段式基础 Prompt

v1.1.0 — 加了 3 个 Few-shot 示例 (Minor: 新功能)

v1.1.1 — "建议"改成"你可以试试" (Patch: 措辞调整)

v2.0.0 — 输出格式从纯文本改成 JSON (Major: 破坏性变更)

版本管理的流程 ↓
(4)

流程: 编写 → 测试 → 灰度 → 上线

  1. 编写: 在测试环境修改 Prompt, 标记为 Draft
  2. 测试: 用 20-50 个测试用例跑一遍, 看准确率和格式正确率
  3. 灰度: 5-10% 真实用户先用新版本, 监控指标
  4. 上线: 指标达标后, 逐步扩大到 100%
A/B 测试怎么做? ↓
(5)

A/B 测试: 新旧 Prompt 对决

A/B 测试 Prompt 的方法:
  1. 保持旧版本 (A 组, 90% 流量)
  2. 新版本只发给 10% 用户 (B 组)
  3. 对比两组的核心指标: 接受率、重新生成率、用户满意度
  4. B 组指标更好 → 逐步扩大到 100%
  5. B 组指标更差 → 回滚, 分析原因
Vitamin 的关键指标:

接受率 > 60% (用户没点"换一个") · 拒答率 < 5% · P95 延迟 < 5 秒

变更日志怎么写? ↓
(6)

变更日志: 每次改动都要记录

## Prompt Changelog — Vitamin 经期状态 ### v1.2.0 (2026-05-10) - [Add] 加了 3 个 Few-shot 示例, 覆盖疲劳/疼痛/情绪 - [Change] 输出格式改为 JSON - [Metric] 接受率 65% → 72% (+7%) ### v1.1.1 (2026-05-05) - [Fix] "你应该" → "你可以试试", 语气更温和 - [Metric] 用户满意度 4.1 → 4.3 ### v1.1.0 (2026-04-28) - [Add] 加了 Negative Instructions: 不给医疗诊断 - [Metric] 拒答率从 0% 升到 3% (预期内)
回滚机制 ↓
(7)

回滚: 新版本出问题, 一键回退

何时回滚:
  • 接受率下降超过 5%
  • 用户投诉突然增多
  • 出现安全违规(给了医疗诊断)
  • 延迟显著增加
回滚前提:

必须保留每个版本的完整 Prompt 文本. 如果你只保存了最新版, 回滚就无从谈起.

→ 最简单的方案: 把 Prompt 存在 Git 里, 每次修改都是一个 commit.

Vitamin 怎么管理 Prompt 版本 ↓
(9)

Vitamin 实战: Prompt 管理制度

要做的

  • 每个 Prompt 有版本号和变更日志
  • 修改前先写 A/B 测试计划
  • Major 变更需要评审 (PM + 开发)
  • 保留所有历史版本, 支持一键回滚

避免的

  • 直接在生产环境改 Prompt
  • 不记录就修改 ("我就改了一个字")
  • 没有测试就全量上线
  • 只保留最新版本, 无法回滚
总结 + 作业 ↓
(10)

一句话规则 + 今日作业

一句话

Prompt 是产品资产, 不是一次性文字.
版本号 + 变更日志 + A/B 测试 = 生产级管理.

改 Prompt 之前先问: 版本号是什么? 怎么测试? 能回滚吗?

  1. 为 Vitamin 的经期 Prompt 写一份变更日志 (至少 3 个版本, 每个标明 Major/Minor/Patch).
  2. 设计一个 A/B 测试方案: 把 Vitamin 的问候语从"你好"改成"嗨~", 怎么测试效果?
  3. 在你常用的 AI 工具里, 故意改一个 Prompt 的小细节, 记录输出变化.
  4. 写 100 字: 给团队解释"为什么不能直接在生产环境改 Prompt".

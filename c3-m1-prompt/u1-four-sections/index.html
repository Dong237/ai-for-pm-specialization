(1)

一句话 Prompt 的尴尬

先看一个真实场景. 你想让 AI 给 Vitamin 用户一条健康建议:

你的 Prompt
给用户一条健康建议
AI 的回复
多喝水,每天至少8杯。保持充足的睡眠,每晚7-8小时。适当运动,每周至少150分钟中等强度活动...

看起来没错, 但问题是 —— 它不知道用户是谁、处于什么状态、你要什么格式. 这条建议放给任何人都一样, 没有任何个性化.

Anthropic 的工程师发现, 90% 的 Prompt 质量问题来自"缺信息" ↓
(2)

不是 AI 不行, 是你没给够信息

✗

错误直觉 A: AI 应该自己理解我要什么

"它那么聪明, 一句话应该就懂了吧?"

为啥不对: AI 没有你的业务背景, 没有你的用户画像, 也不知道你想要什么格式. 它只能按"最通用"的方式回答.
✗

错误直觉 B: 写很长的一大段就行

"那我把所有信息都堆上去总行了吧?"

为啥不对: 一大段没有结构的文字, AI 会抓不住重点. 就像你收到一封 2000 字没有分段的邮件 —— 你也会遗漏信息.
真相

好 Prompt 不是"写得多", 是每段信息放对位置.

4 段式: Role · Task · Context · Format

→ Anthropic 官方推荐的 Prompt 结构. 像填表一样, 一段一段填, 质量自然上去.

四段分别是什么? 一个一个看 ↓
(3)

四段拆解 — 每段解决一个问题

R

Role (角色)

"你是谁?" — 给 AI 一个专业身份, 让它从特定视角回答. 比如"你是一位有 10 年临床经验的女性健康顾问".

T

Task (任务)

"你要做什么?" — 明确说要执行的动作. 不是"聊聊健康", 而是"根据用户描述的身体状态, 给出健康建议".

C

Context (上下文)

"在什么背景下?" — 提供业务场景、用户信息、限制条件. 比如"用户处于经期第 2 天, 感觉疲劳".

F

Format (输出格式)

"你要怎么回答?" — 指定输出的结构. 比如"分 3 点, 每点一句话, 最后加一条警告信号".

记忆口诀: RTCF — 就像写邮件: 收件人(Role) + 主题(Task) + 正文(Context) + 附件格式(Format).

Anthropic 在官方 Prompt Engineering Guide 里反复强调: 好的 Prompt 需要明确的角色、任务、上下文和输出格式.

光说不练不行, 来看一个 before / after 对比 ↓

Before / After — 同一个需求, 天壤之别

Before: 一句话 Prompt
给经期的用户一些健康建议
After: 4 段式 Prompt
## Role 你是一位专业的女性健康顾问,有10年临床经验,擅长用通俗易懂的语言解释健康概念。 ## Task 根据用户描述的身体状态,给出个性化的健康建议和注意事项。 ## Context 用户正在使用 Vitamin App,她当前处于经期第2天,感觉腹部不适和疲劳。她平时有跑步习惯。 ## Output Format 1. 状态解读(1-2句) 2. 建议措施(3条,每条一句话) 3. 需要注意的信号(1条)
Before 质量
25%
After 质量
92%

After 版本的 AI 回复会: 以"女性健康顾问"的口吻回答, 针对"经期第 2 天 + 疲劳"给出具体建议, 考虑跑步习惯, 并且严格按 3 段格式输出. 同一个模型, 纯靠 Prompt 结构.

四段的优先级不一样 — 有些可以省, 有些不能 ↓
(5)

四段的优先级: Task > Format > Context > Role

如果只能写一段, 写哪段? Anthropic 建议:

  1. Task — 绝不能省. 不说要做什么, AI 完全没方向.
  2. Format — 高优先. 没有格式要求, 输出长度和结构不可控.
  3. Context — 中优先. 简单任务可以不给, 复杂任务必须给.
  4. Role — 锦上添花. 大部分场景加了更好, 但不加也能跑.

换句话说:

规律

Task 是骨架, Format 是控制, Context 是燃料, Role 是滤镜.

→ 写 Prompt 的时候, 先把 Task 和 Format 写清楚, 再补 Context 和 Role.

每段有什么写法技巧? 挨个细讲 ↓
(6)

Role 段: 不只是"你是 XX 专家"

Role 的作用不只是换个称呼 — 它会改变 AI 的回答风格、用词水平和专业深度.

弱 Role
你是一个健康助手。
强 Role
你是一位有10年临床经验的女性健康顾问。你的沟通风格是温暖但专业的,像闺蜜兼医生。你总是先确认用户的感受,再给建议。
Role 写作公式:

身份 + 年限/资质 + 沟通风格 + 行为倾向

比"你是 XX"多给 3 层信息: 多有经验、怎么说话、习惯性行为.

Task 段怎么写 ↓
(7)

Task + Format: 你的"需求文档"

Task 要回答"做什么", Format 要回答"怎么给我".

Task 的 3 个常见错误:
  1. 太模糊: "帮我分析一下" → 分析什么? 分析到什么程度?
  2. 太多步: 一次塞 5 个任务, AI 会丢步骤. 一个 Prompt = 一个核心任务.
  3. 没有动词: "关于用户反馈" → 总结? 分类? 提取? 动词决定行为.
Format 常用模板:
  • 分 N 点, 每点一句话 — 适合建议类
  • JSON 格式: {"field": "value"} — 适合需要程序解析的场景
  • 表格: | 列1 | 列2 | — 适合对比/评估
  • 先结论后展开, 控制在 X 字以内 — 适合摘要类

(9)

Vitamin 案例 — 4 段式怎么落地

Vitamin 有 5 个身体状态(经期、排卵期、黄体期、经前期、安全期). 每个状态都需要不同的 Prompt. 看看经期状态的 4 段式:

Role: 你是 Vitamin App 的健康顾问"小 V", 专注女性周期健康, 说话温暖、不说教、像闺蜜.

Task: 根据用户当前的身体状态和描述, 给出当天的个性化健康建议.

Context: 用户 ID: {user_id}, 当前状态: 经期第 {day} 天, 用户输入: "{user_input}", 过往记录: 上个周期 {cycle_length} 天.

Format: JSON 格式: {"greeting": "...", "analysis": "...", "tips": ["tip1","tip2","tip3"], "warning_sign": "..."}

4 段式带来的好处

  • 不同开发者写的 Prompt 质量一致 — 有框架
  • 新功能上线只需换 Context 和 Format — 复用 Role 和 Task
  • Bug 定位更快 — 哪段出问题改哪段
  • 产品评审时能直接读 Prompt — 不是一坨文字

4 段式不能解决的

  • 不能保证 AI 100% 按格式输出 (需要 U5 XML Tags)
  • 不能防止 AI 编造信息 (需要 U6 Negative Instructions)
  • 不能让 AI "想清楚再答" (需要 U4 CoT)
  • 不能管理多个版本 (需要 U7 版本管理)

一句话

Role(你是谁) + Task(做什么) + Context(背景) + Format(怎么给)

(1)

复杂问题: AI 直接答经常错

看一个 Vitamin 场景. 用户说: "最近总是头疼, 想吃甜的, 情绪不太稳." AI 需要判断她处于哪个身体状态.

直接让 AI 答:

"根据你的描述, 你可能处于经期."

→ 错了. 头疼+嗜甜+情绪不稳 = 经前期(PMS), 不是经期. AI 跳过了分析, 直接猜了一个最常见的答案.

问题不是 AI 不知道, 而是它没有"想"就直接"答"了.

怎么让它"想"? ↓
(2)

Chain-of-Thought: 让 AI 展示推理过程

核心概念

Chain-of-Thought (CoT) = 让模型先写推理步骤, 再给最终答案.

"请一步一步分析, 然后给出结论."

→ 就这一句话, Wei et al. (2022) 在 Google 的论文中证明: 大模型的数学推理准确率提升了 10-20%.

为什么有效?

LLM 是"一个 token 一个 token 往后猜"的(回忆 Module 1). 当你让它先输出推理步骤, 这些步骤就变成了后续 token 的额外上下文. 等于给它自己"打了个草稿", 再基于草稿给答案.

有哪几种 CoT 写法? ↓
(3)

三种 CoT 写法, 从简单到精细

Level 1: Zero-shot CoT

在 Prompt 末尾加一句: "Let's think step by step." 或 "请一步一步分析."

→ 最简单, 大部分场景够用. 适合 PM 日常.

Level 2: Manual CoT

在 Few-shot 示例中, 手动写出推理步骤作为示例, 让 AI 学着来:

问: 头疼+嗜甜+情绪波动 → 什么状态? 思考: 头疼可能是经前或经期. 嗜甜与孕酮相关, 是经前特征. 情绪波动在经前3-7天常见. 三者叠加 → 经前期. 答: 经前期

Level 3: Structured CoT

用 XML Tags (U5 会详细讲) 把推理和答案分开:

<thinking>逐步分析...</thinking> <answer>经前期</answer>

→ 最精确, 方便程序提取答案, 忽略推理过程.

哪些任务适合 CoT? ↓
(4)

适合 CoT 的任务 vs 不适合的

适合 CoT

  • 多步推理 (症状分析 → 状态判断)
  • 数学/逻辑问题
  • 需要权衡多因素的决策
  • 需要解释"为什么"的场景
  • 分类有多个可能结果的情况

不适合 CoT

  • 简单的一问一答 (浪费 token)
  • 纯创意写作 (CoT 反而让文风变死板)
  • 实时性要求极高 (CoT 增加延迟)
  • 翻译任务 (直接翻就行)
  • 简单分类 (情绪正/负/中, 不需要推理)
CoT 的代价是什么? ↓
(5)

CoT 的代价: 更多 token + 更高延迟

使用 CoT 的成本:
  1. Token 增加: 推理过程本身也是 output token, 可能增加 50-200 token
  2. 延迟增加: 多输出的 token = 多等待的时间 (每 100 token ≈ 0.5 秒)
  3. 费用增加: output token 通常比 input 贵 2-5 倍

不用 CoT

72%
快, 便宜, 但复杂问题容易错

用 CoT

91%
慢 0.5-1 秒, 贵 20%, 但准确

Vitamin 建议: 健康状态判断(高风险)用 CoT, 日常闲聊(低风险)不用.

一个实用技巧: 隐藏推理过程 ↓
(6)

用户不需要看到"思考过程"

CoT 的推理步骤是给 AI 用的, 用户只需要看到最终答案.

解决方案: 分段输出, 程序提取
Prompt: "请先在 <thinking> 标签中分析, 然后在 <answer> 标签中给出面向用户的回复." AI 输出: <thinking> 用户描述头疼和嗜甜, 结合经前3-7天特征... </thinking> <answer> 你好! 根据你的描述, 你现在可能处于经前期. 建议多补充镁元素, 用坚果代替甜食哦~ </answer>

→ 程序只提取 <answer> 里的内容展示给用户, <thinking> 部分留给日志.

更高级的 CoT 变体 ↓
(7)

CoT 进阶: Self-Consistency + Tree of Thoughts

Self-Consistency (自洽性):

同一个问题让 AI 用 CoT 回答 3-5 次, 取多数一致的答案. 就像让 3 个医生会诊, 多数意见更可靠.

→ 代价是 3-5 倍调用次数, 但对高风险决策(如健康建议)值得.

Tree of Thoughts (ToT):

让 AI 同时探索多条推理路径, 自我评估每条路径的可行性, 然后选最优的. 像下棋时提前想 3 步.

→ 太复杂, PM 了解概念即可, 实际用 Level 1-2 的 CoT 就够了.

让我分析用户的症状: 1. 头疼 → 可能出现在经前期或经期 2. 想吃甜的 → 典型的经前期(PMS)症状, 与孕酮水平变化有关 3. 情绪不稳定 → 经前期常见, 通常在月经前 3-7 天 综合判断: 三个症状同时出现, 最符合经前期(PMS)的特征模式.

Vitamin 怎么用 CoT ↓
(9)

Vitamin 实战: 哪些功能加 CoT

加 CoT 的功能 (高风险, 需要推理):
  • 身体状态判断 — 需要综合多个症状推理
  • 健康建议生成 — 需要结合状态+历史给出个性化建议
  • 异常信号识别 — 需要判断是否要建议就医
不加 CoT 的功能 (低风险, 简单):
  • 日常打招呼 — "早上好" 不需要推理
  • 情绪分类 — 简单的正/负/中分类, Few-shot 就够
  • 文案生成 — 创意任务, CoT 反而让文风变死板

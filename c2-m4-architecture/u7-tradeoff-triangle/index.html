(1)

"不可能三角" — 项目管理的经典理论

项目管理有句老话: "快、好、便宜 — 只能选两个."

AI 架构也一样:

效果好 + 成本低 → 但复杂度高
效果好 + 简单 → 但很贵
成本低 + 简单 → 但效果有限

PM 的工作不是追求 "三个都要", 而是决定 "牺牲哪一个".

3 个维度分别是什么? ↓
(2)

三角的 3 个维度

维度 1

效果 (Effectiveness)

输出质量、准确率、用户满意度. 效果越好, 产品越有价值.
维度 2

成本 (Cost)

API 调用费 + 搭建费 + 维护人力. 成本越低, 商业模型越健康.
维度 3

复杂度 (Complexity)

搭建难度 + 维护难度 + 调试难度. 复杂度越低, 团队越能快速迭代.
每种架构在三角里的位置 ↓
(3)

Prompt-only: 成本低 + 简单, 牺牲效果

Prompt-only 优化了: 成本极低, 复杂度极低. 几小时上线, 月费几百块.

牺牲了: 效果受限于模型自带能力. 不知道的知识会编, 风格控制有限.

适合: MVP 阶段, 快速验证, 不需要极致效果的场景.

RAG 呢? ↓
(4)

RAG: 效果好 + 成本可控, 牺牲一点复杂度

RAG 优化了: 效果显著提升 (幻觉↓85%), 成本比 Fine-tune 低得多.

牺牲了: 复杂度上升 — 需要搭建向量数据库、维护文档、优化检索策略.

适合: 需要专有知识 + 溯源的产品. 性价比最高的中间选项.

Fine-tune 和 Multi-Agent 呢? ↓
(5)

Fine-tune 和 Multi-Agent: 效果最好, 但代价最大

FT

Fine-tune

效果: 风格/术语完美匹配

代价: 训练费 + 数据标注 + 推理成本上涨 + 需要重新训练
MA

Multi-Agent

效果: 能处理最复杂的多步骤任务

代价: 成本翻倍 + 延迟叠加 + 编排和调试极其复杂

PM 该怎么决策? ↓
(7)

PM 的决策框架: "我们牺牲什么?"

不要问 "能不能三个都要", 要问 "我们产品的阶段, 应该牺牲哪一个?"

不同阶段的取舍策略:

  • MVP 阶段 牺牲效果, 保成本和简单 → Prompt-only
  • 验证后 愿意增加复杂度, 换效果提升 → RAG
  • PMF 后 愿意花钱换极致效果 → Fine-tune
  • 规模化 效果是竞争壁垒, 不惜代价 → Multi-Agent

回到 Vitamin ↓
(9)

Vitamin 的取舍决策

Vitamin 是初创健康产品, 当前阶段的优先级:

Vitamin 策略

现阶段: 牺牲效果上限, 保成本和简单

核心功能用 Prompt-only 快速上线, 医学问答用 RAG 保准确.

Fine-tune 和 Multi-Agent 留到 PMF 后, 有数据有预算再做.

→ 这不是 "偷懒", 这是 "阶段性最优解".

面试金句:

"AI 架构的三角取舍是: 效果、成本、复杂度只能优化两个. Vitamin 现阶段选择牺牲效果上限, 保成本和简单 — 因为 MVP 的核心目标是验证需求, 不是追求极致效果. PMF 后, 再按需升级."

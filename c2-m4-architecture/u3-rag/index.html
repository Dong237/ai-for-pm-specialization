RAG 适用场景

当模型 "不知道" 的时候, 给它配个图书馆.

用户问: "经期能吃布洛芬吗?"

模型瞎编了一个答案, 还特别自信!

如果是医疗产品, 这要出事的...

—— RAG 就是解决这个问题的.

(1)

人类也是 "先查再答"

你去看病, 医生不是凭记忆给你开药 — 她会翻医学指南、查药物手册、看你的检查报告. 先检索, 再判断.

RAG 的灵感就是这个:

让 LLM 在回答之前, 先去 "图书馆" 查一下相关资料.

RAG = Retrieval-Augmented Generation = 检索增强生成

对比:

Prompt-only = 纯靠记忆答题
RAG = 开卷考试

开卷考试的准确率, 当然比纯记忆高得多.

RAG 的 4 步 pipeline 是怎么工作的? ↓
(2)

RAG 的 4 步 Pipeline

整个 RAG 系统就 4 步. 记住这 4 个词: Embed → Retrieve → Augment → Generate.

准备阶段 (一次性):
文档 → 切块 → Embedding → 存入向量数据库

使用阶段 (每次查询):
用户提问 → 问题 Embedding → 检索相似文档 → 塞进 Prompt → LLM 生成

PM 认知: RAG 系统有两个独立的 pipeline — "准备" 和 "使用". 准备阶段可以定期跑 (比如每天凌晨更新知识库), 使用阶段是实时的.

一步步拆开看 ↓
(3)

Step 1: 文档切块 + Embedding

Step 1: Embed

把你的文档变成模型能 "搜索" 的格式.

  1. 切块 — 把长文档切成 200-500 字的小块. 太长检索不精准, 太短缺上下文.
  2. Embedding — 每个块通过 Embedding 模型变成一组数字 (向量). 语义相似的块, 向量也相似.
  3. 存储 — 向量存入向量数据库 (Pinecone / Milvus / pgvector).

类比: 就像图书馆管理员把每本书做了索引卡, 写上关键词. 以后要找什么, 先查索引卡, 不用翻遍所有书.

用户提问时发生什么? ↓
(4)

Step 2: 检索 (Retrieve)

Step 2: Retrieve

用户提问 → 变成向量 → 找最相似的文档块.

  1. 用户问: "经期能跑步吗?"
  2. 这句话通过同一个 Embedding 模型变成向量
  3. 在向量数据库里计算余弦相似度, 找到最相关的 Top 3 文档块
  4. 比如: "经期运动指南"、"剧烈运动注意事项"、"瑜伽散步推荐"

检索质量 = RAG 质量. 如果检索到的文档不相关, 后面 LLM 再厉害也白搭. 这就是为什么 "切块策略" 和 "Embedding 模型选择" 是 RAG 的核心优化点.

检索到了, 然后呢? ↓
(5)

Step 3: 增强 (Augment)

Step 3: Augment

把检索到的文档块塞进 Prompt, 作为 "参考资料".

System: 你是 Vitamin 健康顾问. 根据以下参考资料回答用户问题. 如果参考资料中没有相关信息, 说 "我不确定". <context> [检索到的文档块1] [检索到的文档块2] [检索到的文档块3] </context> User: 经期能跑步吗?

关键: "如果参考资料中没有相关信息, 说 '我不确定'" — 这句话非常重要. 它让模型在没有依据时选择拒答, 而不是编造. 这就是 RAG 降低幻觉的核心机制.

最后一步: 生成回答 ↓
(6)

Step 4: 生成 (Generate)

Step 4: Generate

LLM 基于参考资料生成回答, 并标注来源.

模型输出:

"经期可以适度运动, 但建议避免高强度跑步. 推荐散步、瑜伽等轻量运动. 如果月经量较大或有痛经, 建议以休息为主."

来源: 《女性经期运动指南》, 《运动医学建议》

对比一下:

没有 RAG 时: 模型可能说 "经期可以正常跑步, 没有影响" — 错误且危险.
有 RAG 时: 模型基于权威资料回答, 并给出引用 — 准确且可验证.

RAG 的 3 大核心优势 ↓
(7)

RAG 的 3 大核心优势

🔄
知识可更新
换文档就行, 不用重新训练模型. 医学指南更新了? 换个文件, 几分钟搞定.
📑
可溯源
回答附带引用来源. 用户/医生可以验证 "这个建议出自哪份指南". 合规必需.
🛡️
降低幻觉
研究表明 RAG 可将幻觉率降低高达 85%. 有据可查比纯靠记忆靠谱得多.

什么时候必须用 RAG?

  • 医疗产品 — 必须引用权威指南, 不能瞎编
  • 法律产品 — 必须引用法条, 出错有法律责任
  • 企业知识库 — 公司内部文档, 模型不可能知道
  • 客服系统 — 产品 FAQ 经常更新, 不能重新训练

Vitamin 的 RAG 方案

Vitamin 是女性健康产品. 涉及医学知识的功能, 必须用 RAG.

Vitamin 知识库应该包含:

  • 权威医学指南 (经期健康、营养建议、运动指南)
  • 药物交互数据 (布洛芬禁忌、常见中药注意事项)
  • Vitamin 产品 FAQ (功能说明、使用方法)
  • 季节性健康提醒 (换季注意事项, 定期更新)

面试金句:

"Vitamin 的医学知识问答必须用 RAG, 原因有三: 一, 医学指南会更新, RAG 换文档就行; 二, 用户需要看到建议来源, RAG 支持溯源; 三, 健康产品零容忍幻觉, RAG 将幻觉率降低 85%."

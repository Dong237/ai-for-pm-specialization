(3)

结论: Vitamin 是混合架构

核心结论

Vitamin 不是 "选一种架构",
而是 "每个功能选最合适的架构".

这就是 2026 年的最佳实践 — 混合架构 (Hybrid Architecture).

Vitamin 架构摘要:

  • Prompt-only 情绪分类 + 健康提醒 + 文案生成
  • RAG 医学知识问答 + 药物交互查询
  • Fine-tune 品牌语调统一 (PMF 后)
  • Multi-Agent 身体翻译器完整版 (规模化后)
为什么不直接 Fine-tune 一切? ↓
(4)

为什么不 Fine-tune 一切?

面试官可能会问: "为什么不直接 Fine-tune 一个 Vitamin 专用模型, 什么都能干?"

3 个原因:

  1. 知识更新 — 医学指南每季度更新. Fine-tune 要重新训练, RAG 只需换文档.
  2. 溯源需求 — 健康产品必须标注建议来源. Fine-tune 做不到引用, RAG 天然支持.
  3. 成本节奏 — 现阶段没有 PMF, 花 5 万训模型可能需求一变就白花了. 等稳定了再做.

每个阶段的三角取舍

阶段优化了牺牲了理由
Phase 1成本 + 简单效果上限验证需求 > 追求效果
Phase 2效果 + 成本复杂度用户要准确的医学信息
Phase 3效果成本 + 复杂度品牌差异化是竞争力
Phase 4效果 (极致)成本 + 复杂度 (极高)规模化后的用户体验壁垒

关键: 每个阶段都有明确的 "我们牺牲了什么". 这不是偷懒, 是 阶段性最优解.

完整论证 ↓
(7)

完整论证: 为什么选 RAG + Multi-Agent

身体翻译器为什么不能只用 Fine-tune?

  1. Fine-tune 教 "怎么说", 不教 "知道什么" — 医学知识需要 RAG 实时检索, Fine-tune 搞不定
  2. Fine-tune 没有溯源能力 — 健康建议必须标注来源, RAG 天然支持
  3. Fine-tune 是单步的 — 身体翻译器有 4 步 (分析→检索→生成→审核), 需要 Multi-Agent 编排
  4. Fine-tune 更新代价高 — 医学指南更新要重新训练, RAG 换文档即可

最终架构: RAG + Multi-Agent 的组合

RAG 负责 "知识层" (检索医学指南), Multi-Agent 负责 "编排层" (协调 4 个 Agent). Fine-tune 作为 "润色层" (PMF 后给生成 Agent 微调品牌语调). 三者不是替代关系, 是互补关系.

面试怎么说? ↓
(9)

面试回答模板

面试官问 "Vitamin 的架构怎么选的", 这样答:

面试完整版回答 (2 分钟):

"Vitamin 是混合架构, 按功能分层选型.

第一层, 情绪分类、文案生成这类标准化任务, 用 Prompt-only — 成本最低, 效果够用.

第二层, 医学知识问答必须用 RAG — 原因三点: 知识需要更新, 回答需要溯源, 幻觉在健康领域零容忍.

第三层, Fine-tune 放在 PMF 后 — 现在 Prompt + Few-shot 能控制 80% 语调, 等品牌风格确定、数据积累够了再做.

第四层, 身体翻译器终版用 Multi-Agent — 因为有 4 步 (分析→检索→生成→审核), 每步需要不同专长, 单 Agent 搞不定.

整体策略: 从简到繁, 按需升级. MVP 先用最简方案上线, 验证需求后再逐步加 RAG、Fine-tune、Multi-Agent."

四种 AI 架构概览

同一个 AI 功能, 竟然有 4 种完全不同的搭法?

PRD 写完了, 成本算完了, 指标定完了.

老板问: "技术上到底怎么实现?"

是直接调 API? 还是要建知识库? 还是要训模型?

—— 今天 10 步, 把 4 种架构看清楚.

10 步学完, 你会:
  • 画出 4 种 AI 架构的复杂度阶梯
  • 一眼判断某个 AI 功能该用哪种架构
  • 知道为什么 "越高级越好" 是错的
  • 给 Vitamin 的每个功能匹配对应架构
走起 →
(1)

同一个功能, 4 种搭法

假设你要做一个 "帮用户总结文章" 的功能. 看下面 4 种实现方式:

Level 1
Prompt-only
用户输入文章 → System Prompt: "请总结" → LLM → 输出摘要

一个 API 调用搞定. 几行代码.

Level 2
RAG
用户提问 → 检索知识库 → 拼装上下文 → LLM → 带来源的回答

多了一个 "检索" 步骤, 但回答有据可查.

Level 3
Fine-tune
准备训练数据 → 微调模型 → 部署新模型 → 用户使用

改了模型本身. 风格/术语 "内化" 到权重里.

Level 4
Multi-Agent
用户输入 → Agent A: 分析 → Agent B: 检索 → Agent C: 生成 → Agent D: 审核 → 输出

多个 Agent 分工合作, 各有专长.

哪个更好? 不是 "越高级越好" — 下一步说清楚 ↓
(2)

最常见的错误: "越高级越好"

✗

错觉: 我要用最先进的架构

"Multi-Agent 听起来好厉害, 我们产品一定要用!"

为啥不对: 过度工程是 AI 产品最常见的死因. 花 3 个月搭 Multi-Agent, 结果一个好的 Prompt 就能解决.
✗

错觉: Fine-tune 一定比 RAG 好

"训练过的模型肯定更懂我的业务!"

为啥不对: Fine-tune 教的是 "风格", 不是 "事实". 要最新知识, RAG 才是正解. 搞混了等于烧钱.
真相

AI 架构选型的唯一原则:

从最简单的开始, 只在遇到瓶颈时升级.

→ 80% 的初创 AI 产品, Prompt-only 就够了. 真正需要 Multi-Agent 的, 可能不到 5%.

那每种架构到底怎么工作? 从最简单的开始 ↓
(3)

Level 1: Prompt-only — 最简单

一个 API 调用 + 一段精心设计的 System Prompt. 就这样.

Prompt-only
怎么工作
你写好 System Prompt
↓
用户每次提问, 连同 Prompt 一起发给 LLM
↓
LLM 返回回答

适合什么场景:

  • 任务相对标准化 (分类 / 改写 / 翻译)
  • 不需要外部知识 (模型自带的就够)
  • 规模小, 不需要极致性能
  • MVP 阶段 — 先验证需求再说

什么时候不够了?

  • 用户问了模型训练时不知道的事 → 需要 RAG
  • Prompt 太长, 超出 Context Window → 需要 RAG 或 Fine-tune
  • 模型风格不对, 怎么调 Prompt 都不像 → 需要 Fine-tune
当模型 "不知道" 你的知识时 → 升级到 RAG ↓
(4)

Level 2: RAG — 给模型配个 "图书馆"

Retrieval-Augmented Generation. 不改模型, 改输入 — 用检索补充模型不知道的知识.

RAG Pipeline
4 步流程
Step 1: 把文档切块 + 向量化, 存入向量数据库
↓
Step 2: 用户提问 → 把问题也向量化
↓
Step 3: 在数据库里找最相似的文档块 (检索)
↓
Step 4: 把检索结果塞进 Prompt → LLM 生成回答

RAG 的核心优势:

  • 知识可更新 — 换文档就行, 不用重新训练模型
  • 可溯源 — 回答附带引用来源, 用户能验证
  • 成本可控 — 不用训练, 只多了检索这一步
  • 降低幻觉 — 有据可查, 比纯靠模型记忆靠谱

一个数据: 据研究, 优化的 RAG 系统可以将幻觉率降低 85%. 这就是为什么医疗、法律场景几乎必用 RAG.

RAG 解决了 "知识" 问题. 但如果是 "风格" 问题呢? ↓
(5)

Level 3: Fine-tune — 改模型本身

不是给模型 "看参考资料" (那是 RAG), 而是让模型 "上课学习", 把新知识/风格写进权重.

Fine-tune
流程
准备 500-5000 条标注数据 (input → ideal output)
↓
在基础模型上训练 (几小时到几天)
↓
得到一个 "你的专属模型"
↓
部署 → 用户使用

关键认知: Fine-tune 改的是 "怎么说", 不是 "知道什么".

Fine-tune 让模型学会一种风格、一套术语、一种判断倾向. 但不适合教它 "今天的新闻" — 那是 RAG 的活.

行业共识: "把变化的知识放 RAG, 把稳定的风格放 Fine-tune, 别搞反了."

当任务太复杂, 一个 Agent 搞不定时 ↓
(6)

Level 4: Multi-Agent — 团队作战

一个人干不完, 就组个团队. Multi-Agent 的逻辑跟组建团队一样 — 每个 Agent 是一个 "专家角色".

Multi-Agent
典型模式: Orchestrator + Workers
用户请求 → Orchestrator (指挥官)
↓
┌── Agent A: 信息收集 (搜索/检索)
├── Agent B: 分析推理 (专业判断)
├── Agent C: 内容生成 (写作/回复)
└── Agent D: 质量审核 (验证/修正)
↓
Orchestrator 整合 → 最终输出

Multi-Agent 的代价:

  • 每个 Agent 都是一次或多次 API 调用 — 成本翻倍
  • Agent 间传递信息可能丢失或曲解 — 调试困难
  • 响应时间 = 所有 Agent 延迟之和 — 用户等更久
  • 2026 年预测: 40% 企业应用会嵌入 Agent — 但大部分是 single-agent

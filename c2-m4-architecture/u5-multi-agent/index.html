为什么需要 "多个 Agent"?

想象你要写一份研究报告. 你不会让一个人同时搜资料、分析数据、写文章、校对事实. 你会组个团队:

研究员
→
搜集资料和数据
分析师
→
分析数据, 提取洞察
写手
→
写成报告
审核
→
事实核查 + 质量把关

Multi-Agent 就是这个逻辑: 每个 Agent 是一个 "专家角色", 各干自己擅长的, 由一个 "指挥官" (Orchestrator) 协调.

核心模式: Orchestrator + Workers ↓
(2)

核心模式: Orchestrator-Worker

2026 年最主流的 Multi-Agent 架构模式:

Orchestrator-Worker
用户请求 → Orchestrator (指挥官)

┌── Worker A: 信息收集
├── Worker B: 分析判断
├── Worker C: 内容生成
└── Worker D: 质量审核

Orchestrator 整合各 Worker 结果 → 最终输出

关键区别:

  • Orchestrator 不干活, 只做 "决策 + 协调"
  • 每个 Worker 有自己的 Prompt / 工具 / 甚至不同的模型
  • Worker 之间可以串行 (A→B→C) 也可以并行 (A+B 同时跑)
还有哪些常见模式? ↓
(3)

3 种常见的 Multi-Agent 模式

Orchestrator-Worker
一个指挥官动态分配任务. 最灵活, 适合复杂多变的场景.
指挥官 → [A, B, C, D] → 整合
Sequential Pipeline
Agent 按固定顺序串行. 简单可靠, 适合流程固定的任务.
A → B → C → D → 输出
Handoff (接力)
Agent 之间显式 "交接". OpenAI Agents SDK 的核心模式.
A --handoff-→ B --handoff-→ C
什么任务适合 Multi-Agent? ↓
(4)

什么任务需要 Multi-Agent

✓ 适合 Multi-Agent

  • 多步骤, 步骤间有不同 "专长"
  • 需要调用不同工具 (搜索/数据库/API)
  • 需要审核/反馈循环 (生成 → 审核 → 改)
  • 不同步骤最好用不同模型 (贵的做决策, 便宜的做执行)

✗ 不需要 Multi-Agent

  • 任务简单, 一个 Prompt 就能搞定
  • 步骤少于 3 步
  • 对延迟敏感 (每多一个 Agent 多几秒)
  • 团队没有能力维护复杂的 Agent 编排
Multi-Agent 的代价是什么? ↓
(5)

Multi-Agent 的代价: 成本 × 延迟 × 复杂度

代价 1

成本翻倍

5 个 Agent = 5 次以上 API 调用. 用 Plan-and-Execute 模式可以用便宜模型做执行, 降低 90% 成本.
代价 2

延迟叠加

串行 Agent: 总延迟 = 各 Agent 延迟之和. 5 个 Agent 每个 2 秒 = 用户等 10 秒.
代价 3

调试困难

Agent 之间的信息传递可能丢失或曲解. 66% 开发者说 AI 方案 "差一点但不对".

PM 认知: Multi-Agent 是最强但也最贵的架构. 用之前先问: "这个任务真的需要多个 Agent 吗? 单个 Agent + 工具调用够不够?"

关键协议: MCP ↓
(7)

MCP: Agent 的 "USB 接口"

2026 年 Agent 领域最重要的协议: MCP (Model Context Protocol).

类比:

USB 让所有设备用同一种接口连接电脑.
MCP 让所有 Agent 用同一种协议连接工具.

MCP 解决了什么:

  • 之前: 每个工具都需要单独写集成代码. 10 个工具 = 10 套代码.
  • 现在: 工具只要支持 MCP, Agent 就能直接连. 即插即用.
  • Anthropic 发起, Google (A2A) 和 IBM (ACP) 也在推类似标准.

Vitamin 身体翻译器: 为什么需要 Multi-Agent

身体翻译器的完整流程有 4 个步骤, 每步需要不同能力:

症状分析 Agent
→
分析 "我头疼, 有点恶心" → 提取关键症状, 分类到经期状态
知识检索 Agent
→
在医学知识库 (RAG) 中找相关指南
建议生成 Agent
→
用 Vitamin 风格写出个性化建议
安全审核 Agent
→
过滤 "去买 XX 药" 等医疗风险内容

面试金句:

"Vitamin 终版身体翻译器用 Multi-Agent 架构, 因为任务有 4 步, 每步需要不同专长: 症状分析需要 NLP, 知识检索需要 RAG, 建议生成需要品牌风格, 安全审核需要规则引擎. 单 Agent 搞不定这个复杂度."

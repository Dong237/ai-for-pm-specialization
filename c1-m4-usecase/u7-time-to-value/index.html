Time-to-value 估算

选好了 Build / Buy / API, 但老板下一句一定是: "多久能上线?"

(1)

PM 最怕的那个问题: "多久能上线?"

面试、周会、OKR 评审 — 无论哪个场景, 这个问题都逃不掉.

场景 1

面试官问

"你打算用 API 做这个功能, 大概多久能让用户用上?"

场景 2

老板在周会上问

"这个 AI 功能能赶上 Q3 发布吗? 时间表给我."

场景 3

工程师反问你

"你说两周, 但 prompt 调优你算了吗? 测试数据准备了吗?"

AI 项目时间估计为啥特别难?

传统软件: 功能明确 → 写代码 → 测试 → 上线. 流程成熟, 可预测.

AI 项目: 多了一个 "模型效果不确定" 的变量. 你不知道 prompt 要调几轮, 不知道 edge case 有多少, 不知道合规审查要多久.

但好消息是 — 有框架可以估. 先看三条路径各自的时间线 ↓
(2)

两个常见误判 — 都会坑你

✗

误判 A: "API 接一下, 两天搞定"

觉得调 API 就像调个天气接口一样简单.

为啥不对: API 返回的是 "概率性输出", 你需要 prompt engineering (1-3天), edge case 处理 (2-3天), 输出格式解析 (1天), 还有安全审查. 不是 "接上就能用".
✗

误判 B: "反正是 no-code, 不用开发"

觉得 Coze/Dify 拖拖拽拽, 一下午就好.

为啥不对: No-code 工具确实快, 但"能跑" ≠ "能上线". 你还需要: 测试各种用户输入 (2天), 对接现有系统 (1-2天), 写操作文档 (1天).
真相

AI 项目的交付时间 =

技术开发 × 不确定性乘数

Productboard 的 PM 团队分享过: "我们在 2 周内做完了 UI 和后端, 然后花了 2 个月调 AI 质量." 这个乘数通常是 1.5x 到 2.5x.

具体到三条路径, 各自要多久? ↓
(3)

三条路径的时间地图

从 U6 你已经选好了路径. 现在给每条路标上真实时间.

🧩

No-code (Coze / Dify)

1-2 周 MVP
设计 2d
搭建 3d
测试 2d
上线 1d

适合: 内部工具、原型验证、简单对话流

🔌

API + 代码 (Python / Node)

2-4 周 MVP
设计 3d
API 对接 5d
UI 开发 5d
测试 3d
部署 2d

适合: 大多数生产级 AI 功能、需要深度定制的场景

🔬

Fine-tune + 自部署

2-6 个月
数据收集 2-4w
训练 2-4w
评估 1-2w
部署 1-2w
监控 持续

适合: 核心竞争力功能、数据隐私要求极高的场景

这些是 "技术开发" 的时间. 但还有很多隐藏时间 ↓
(4)

每个阶段里到底在干啥?

把每条路径的阶段拆开, 你就知道时间花在了哪里.

📋 设计阶段 (所有路径共有)

  • 需求文档: 输入什么 → AI 做什么 → 输出什么
  • Prompt 草稿: 写初版 system prompt + 几个 few-shot 示例
  • 数据准备: 收集 10-50 个真实用户输入做测试集
  • 安全边界: 定义 AI 不该说什么 (尤其健康类产品)

🔧 开发阶段 (差异最大)

No-code

在 Coze/Dify 拖拽 workflow, 配置 prompt + 变量, 接知识库.

耗时: 1-3 天

API

写 API 调用代码, 处理流式输出, 做错误重试, 解析返回格式, 构建前端 UI.

耗时: 5-10 天

Fine-tune

清洗训练数据, 标注, 选基础模型, 配置训练参数, 跑训练, 评估, 迭代.

耗时: 4-8 周

🧪 测试 + 部署阶段

  • 功能测试: 正常输入能跑通
  • 边界测试: 奇怪输入不会崩 (空字符串、超长文本、注入攻击)
  • 质量测试: AI 输出是否达标 (这步最耗时!)
  • 合规审查: 尤其是健康/金融类 — 可能单独需要 1-2 周
  • 灰度发布: 先给 5% 用户试, 观察 1 周
好了, 时间看起来够清楚了? 不, 还有 5 个 "隐藏时间黑洞" ↓
(5)

5 个隐藏时间黑洞 — 每个都坑过人

这些时间不在任何 "技术开发" 排期里, 但每一个都能让你延期.

1

数据准备 +3-10 天

你以为数据 "已经有了" — 实际上: 格式不对, 有脏数据, 缺少标注, 需要脱敏处理. 这个环节永远比预期长.

2

Prompt 调优迭代 +2-7 天

第一版 prompt 写完, 效果只有 60%. 接下来是反复测试 → 修改 → 测试的循环. 从 60% 到 90% 的路比 0% 到 60% 更长.

3

合规审查 +5-15 天

健康类产品尤其严格: AI 生成的内容不能构成医疗建议, 需要加免责声明, 敏感词过滤, 可能需要法务审批. Vitamin 必须过这关.

4

团队学习曲线 +3-5 天

如果团队第一次用 LLM API / no-code 工具, 需要时间学习. 文档阅读, 踩坑, 问 StackOverflow... 首个 AI 功能永远最慢.

5

Edge Case 长尾 +2-5 天

AI 在 90% 的正常输入上表现好, 但那 10% 的奇怪输入会让你抓狂: 方言、错别字、中英混输、emoji... 每修一个坑又冒出两个.

Productboard PM 的真实分享:

"我们用 2 周做完了 UI 和后端. 然后花了 2 个月 调 AI 质量. 我们没有在估计中加上'质量迭代'这个环节, 这是最大的教训."

所以 "估计时间" 到底应该怎么修正? ↓
(6)

现实乘数: 把你的估计 ×1.5 到 ×2.5

软件工程有个经典规律: 开发者永远低估时间, 且低估的倍数惊人地一致.

路径乐观估计×1.5 (正常)×2.0 (常见)×2.5 (首次)
No-code 1 周1.5 周2 周2.5 周
API 2 周3 周4 周5 周
Fine-tune 2 月3 月4 月5 月
怎么选乘数?
  • ×1.5 — 团队做过类似 AI 项目, 数据已准备好, 合规流程清楚
  • ×2.0 — 团队有经验但第一次做这类功能, 数据需要处理
  • ×2.5 — 第一个 AI 功能, 团队刚学 LLM, 合规没走过
2024 vs 2026: 工具进化让速度变快了
Coze/Dify 2024: 不存在 / 刚出生 → 2026: 成熟, 模板丰富
API 工具链 2024: 手写, 缺文档 → 2026: SDK 完善, Cursor 辅助写代码
Fine-tune 2024: 需要 GPU 集群, $1000+ → 2026: LoRA 在消费 GPU 上跑, <$2

→ 所以 2026 年的乘数可以比 2024 年低一档, 但永远不要用 ×1.0.

(9)

Vitamin 首发功能时间线

假设我们选了 Vitamin 的 #1 功能: 症状关键词提取 (用 API 路径).

症状关键词提取 · API 路径 · 详细排期

设计 3d
  • Day 1: 需求文档 — 用户输入症状文本 → 提取关键词 → 结构化输出
  • Day 2: 初版 prompt + 20 条测试用例
  • Day 3: 安全边界定义 (不做诊断, 只做提取)
API 对接 5d
  • Day 4-5: 搭 API 调用框架, 处理流式输出
  • Day 6-7: Prompt 迭代 (从 60% 准确率 → 85%+)
  • Day 8: 输出格式解析 + 错误重试机制
测试 4d
  • Day 9-10: 正常输入 + 边界输入测试
  • Day 11: 合规审查提交 (健康类内容)
  • Day 12: 修复测试发现的问题
部署 2d
  • Day 13: 灰度发布 5% 用户
  • Day 14: 观察指标, 准备全量发布
Vitamin 的特殊风险:
  • 🔴 合规审查 (高风险): 健康类 App, AI 输出不能有医疗诊断暗示. 可能需要额外 1-2 周法务审批.
  • 🟡 数据质量 (中风险): 中文健康描述多样性高, "头疼" "头痛" "脑壳疼" 都是同一个症状.
  • 🟢 API 对接 (低风险): 症状提取是成熟的 NLP 任务, prompt 调优有大量参考.
面试金句, 抄走:

"我估计这个功能 API 路径需要 2 周开发 + ×1.5 的现实乘数 = 3 周. 最大风险是合规审查, 我的 contingency plan 是提前提交审查 + 先内部灰度."

→ 这种 "数字 + 乘数 + 风险 + Plan B" 的回答, 比 "大概两周吧" 强 10 倍.

(10)

一句话规则 + 今晚作业

一句话:

乐观估计 × 2 = 你应该给老板的时间.
再加一个 contingency plan = 靠谱 PM.

AI 项目的不确定性不是 bug, 是 feature. 你的价值就是把不确定性变成可管理的风险.

课后作业 (30 分钟):
  1. 用上面的 Timeline Estimator, 为 Vitamin 的 "个性化健康建议" 功能选择最适合的路径, 并生成完整时间线.
  2. 找出该功能 time-to-value 中最大的风险, 写出你的 contingency plan (100 字).
  3. 比较三条路径的时间线, 写 150 字分析: 为什么你选了这条路径而不是另外两条?
  4. 上 Coze (coze.cn) 或 Dify, 花 15 分钟搭一个最简的 "症状关键词提取" workflow, 体会 no-code 的速度.

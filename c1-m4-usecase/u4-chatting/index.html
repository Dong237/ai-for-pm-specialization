Chatting 类用例

Q&A / Support / Companion — 都叫"聊天", 但架构、风险、产品设计完全不同.

上一课学了 Writing — 让 AI 生成内容, 那是单次的.

但 Vitamin 要跟用户"聊天", 多轮对话. 这比 Reading 和 Writing 都复杂.

"客服问答" 跟 "情感陪伴", 虽然都是聊天, 但产品逻辑应该很不一样吧?

—— 对. 完全不一样. 今天拆清楚.

10 步学完, 你会:
  • 区分 Q&A / Support / Companion 三种对话模式
  • 知道每种模式需要的记忆、语气、Fallback 策略有什么不同
  • 用 Chat Pattern Visualizer 亲眼看到三种对话的运行方式
  • 判断 Vitamin 的每个对话功能该用哪种模式
走起 →
(1)

三个"聊天"场景, 一个比一个复杂

想象 Vitamin 用户打开 App, 发了三条消息. 都是"跟 AI 聊天", 但仔细看 ——

📚 Q&A

"经期能吃布洛芬吗?"

用户要一个事实答案, 最好有来源. 问完就走.

🩺 Support

"我这两天肚子痛, 怎么办?"

用户有一个待解决的问题, AI 需要追问、排查、给建议.

💬 Companion

"今天心情不太好, 想聊聊"

用户需要情感陪伴, 没有明确目标. 要的是被"听到".

从表面看, 三个都是"用户发消息 → AI 回消息". 但产品经理如果把它们当同一种功能来设计, 一定会踩坑.

为什么会踩坑? 因为大多数人对"聊天"有一个错误假设 ↓
(2)

错误直觉: "接一个 LLM 就能聊"

✗

错觉: 所有聊天都一样

"接一个大模型 API, 加个对话框, 用户发什么它回什么就行了."

为啥不对: Q&A 需要检索知识库才能给出准确答案 (不然它就编). Support 需要决策树引导用户一步步排查. Companion 需要人格设定 + 长期记忆才有温度. 一个裸 LLM, 三个都做不好.
✗

错觉: 越像人越好

"AI 聊天越自然、越有感情越好, 用户越喜欢."

为啥不对: Q&A 场景里用户要的是精准和快, 不需要 AI 嘘寒问暖. Support 场景里 AI 太热情会显得不专业. 只有 Companion 才需要"像人一样". 场景决定语气.
真相

Chatting 不是一种模式, 是三种完全不同的产品形态:

Q&A · Support · Companion

→ 三种模式的数据流向、记忆需求、风险等级都不同. Reading 和 Writing 是 one-shot (单次), Chatting 加了循环 (多轮). 这个循环怎么转, 就是这三种模式的核心区别.

先看最简单的: Q&A ↓
(3)

模式 1: Q&A — 一问一答, 线性流

用户提一个事实性问题, AI 回答并附上来源. 像查字典, 但比字典聪明.

Q&A 数据流: 直线型

用户提问
→
检索知识库
(RAG)
→
生成答案
+ 引用来源
→
用户满意
对话结束

关键词: RAG (Retrieval-Augmented Generation)

AI 不是凭"记忆"回答, 而是先从你准备好的知识库里检索相关内容, 再用 LLM 把检索到的内容组织成自然语言答案.

→ 没有 RAG 的 Q&A = 让 AI 瞎编. 有 RAG 的 Q&A = AI + 字典.

Vitamin 例子:

"经期能吃布洛芬吗?" → AI 检索 Vitamin 的健康知识库 → 找到《妇产科学》相关段落 → 生成答案并标注来源.

如果知识库里没有? → AI 回答"我目前没有关于这个问题的权威资料, 建议咨询医生."

Q&A 是最简单的模式. 下一个复杂一级: Support ↓
(4)

模式 2: Support — 引导式排查, 分支流

用户有一个待解决的问题, 但信息不够. AI 需要主动追问, 像医生问诊一样, 逐步缩小范围.

Support 数据流: 分支型

用户描述问题
→
AI 追问
缩小范围
↓ 根据回答分支 ↓
情况 A
→ 建议 A
情况 B
→ 建议 B
无法判断
→ 转人工/就医

关键词: Decision Tree (决策树)

Support 不是让 LLM 自由发挥, 而是把 AI 约束在一棵预设的决策树上. 每次追问都是在走树的分支, 最终到达一个确定的"叶子节点" (建议).

→ 如果让 AI 自由回答健康问题, 它可能说"你可能怀孕了" —— 这在产品里是灾难性的.

Vitamin 例子:

"肚子痛怎么办?" → AI 追问: 位置? 时间? 经期阶段? → 判断: 经前痉挛 → 给出热敷 + 休息建议. 如果排查后无法判断 → 明确说"建议线下就医".

Support 的风险比 Q&A 高, 因为涉及健康判断. Fallback (兜底) 策略必须清晰.

最复杂的来了: Companion ↓
(5)

模式 3: Companion — 情感陪伴, 循环流

用户不是来"问问题"或"解决问题"的, 而是来被听到的. 没有明确终点, 对话可以一直转下去.

Companion 数据流: 循环型

用户表达情绪
→
AI 共情回应
→
用户继续倾诉
→
AI 回忆 + 鼓励
↻ 循环 — 没有固定终点, 直到用户觉得好了

关键词: Persona + Long-term Memory (人设 + 长期记忆)

Companion 需要两个 Q&A 和 Support 不需要的东西:

  1. 人设 — AI 不是"助手", 是"朋友". 有固定的说话风格、价值观、称呼方式.
  2. 长期记忆 — 记住用户上周说过什么、喜欢什么甜品、工作压力大的频率. 这需要跨会话的记忆存储.
风险警告:

Companion 是三种模式里风险最高的:

  • 情感依赖 — 用户可能把 AI 当成真正的朋友, 取代真人社交
  • 边界模糊 — 用户可能说出自杀倾向, AI 需要识别并引导到专业资源
  • 隐私敏感 — 长期记忆存储了大量个人情感数据, 合规风险极高

→ 这就是为什么很多健康 App 不敢做 Companion: 不是技术做不到, 是产品风险太大.

三种模式摆齐了, 来做一个全面对比 ↓
(6)

三种模式, 一张表看清差异

PM 做产品决策, 需要从 6 个维度对比这三种模式:

维度 Q&A 问答 Support 支持 Companion 陪伴
对话流向 线性 (一问一答) 分支 (追问→判断) 循环 (无固定终点)
记忆需求 无 / 极短期 本次会话内 跨会话长期记忆
语气 专业、客观 关切、有条理 温暖、像朋友
核心技术 RAG 知识检索 决策树 + Prompt 人设 + 记忆存储
Fallback 策略 "不知道" + 推荐来源 无法判断 → 转人工 继续倾听 / 推荐专业帮助
风险等级 低 中 高
Context 消耗 ~800 tokens ~2000 tokens ~3000+ tokens
PM 决策要点:

从左到右, 复杂度递增、风险递增、成本递增. 所以产品经理的策略应该是: 先做 Q&A (低风险验证), 再做 Support (中风险, 价值高), 最后考虑 Companion (高风险, 需充分准备).

光看表不够, 来看底层架构到底差在哪 ↓
(7)

底层架构: 三种模式各需要什么"零件"

PM 不需要自己写代码, 但要知道每种模式需要什么基础设施, 才能准确评估工期和成本.

Q&A 需要的零件

  • 必须 知识库 (文档/FAQ)
  • 必须 Embedding + 向量检索
  • 必须 Prompt: "基于以下内容回答"
  • 可选 引用溯源 (标注出处)

工期估计: 1-2 周 (用 Coze/Dify 更快)

Support 需要的零件

  • 必须 决策树 / 对话流设计
  • 必须 会话状态管理
  • 必须 Fallback → 人工兜底
  • 可选 用户历史数据接入
  • 可选 症状分类模型

工期估计: 3-4 周 (含医学审核)

Companion 需要的零件

  • 必须 人格设定 (Persona Prompt)
  • 必须 长期记忆存储 (DB)
  • 必须 情绪识别 / 风险检测
  • 必须 危机 Fallback (自杀干预)
  • 可选 个性化推荐

工期估计: 6-8 周 (含伦理审核)

注意 Context Window 的消耗差异:

Q&A 每次只需塞入问题 + 检索结果 (~800 tokens). Companion 需要塞入人设 + 长期记忆摘要 + 近期对话 (~3000+ tokens). 这意味着 Companion 的 API 调用成本是 Q&A 的 3-4 倍.

→ 回想 M1 学的 Context Window 和 Token 定价 — 这些知识现在派上用场了.

面试金句:

"不是所有对话都该用同一种模式. Q&A 靠知识库, Support 靠决策树, Companion 靠人设和记忆. 我会按风险等级排优先级: 先上 Q&A, 再上 Support, Companion 放最后."

一句话规则 + 作业

一句话:

对话 ≠ 对话.
先分清 Q&A / Support / Companion,
再定架构、语气、Fallback.

看到"要做个聊天功能", 永远先问: 用户要的是事实、解决方案、还是陪伴?

课后作业 (30 分钟):
  1. 打开豆包 / ChatGPT, 分别模拟一个 Q&A、一个 Support、一个 Companion 场景, 观察同一个 AI 在三种场景下的表现差异. 哪种它做得最好? 哪种最差?
  2. 给 Vitamin 的"健康知识问答"功能写 5 个 Q&A 场景, 标出每个问题应该检索哪个知识来源.
  3. 画一个 Vitamin 的 Support 决策树: 用户说"不舒服" → 你会设计 AI 追问哪 3 个问题? 最终导向哪些建议?
  4. 思考: 如果 Vitamin 做 Companion 功能, 你会设计什么样的"危机检测"机制? 写 200 字.

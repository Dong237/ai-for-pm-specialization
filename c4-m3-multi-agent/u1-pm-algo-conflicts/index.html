(1)

PM 和算法工程师, 天生就"语言不通"

你说"效果不好", 算法听到的是"你在质疑我的模型". 你说"能不能快点", 算法听到的是"你不懂训练要多久". 这不是态度问题 — 是两个专业的语言系统完全不同.

PM 的世界

  • 用户反馈 / NPS / 留存率
  • "快速上线验证"
  • "效果好不好" = 用户满不满意
  • 周级迭代节奏
VS

算法的世界

  • AUC / F1 / Precision / Recall
  • "数据质量决定上限"
  • "效果好不好" = 离线指标够不够
  • 月级训练周期
不是谁对谁错, 是双方在用不同的尺子量同一件事 ↓
(2)

错误直觉: "算法不配合" vs "PM 不懂技术"

✗

PM 的错觉: 算法团队不配合

"他们总说做不了, 是不想做吧?"

为啥不对: 算法说"做不了"通常有技术原因: 数据不够, 模型能力边界, 合规约束. 不是态度问题.
✗

算法的错觉: PM 不懂技术还瞎指挥

"你连 Precision 和 Recall 都分不清, 怎么提需求?"

为啥不对: PM 不需要会写模型, 但需要理解能力边界. 同样, 算法也要理解业务指标. 双向翻译才是解法.
真相

PM-算法冲突的本质:

目标错位 + 语言不通 + 节奏不同

不是人的问题, 是结构性的认知鸿沟. 好在有方法论可以弥合.

接下来逐个拆解 5 大冲突. 冲突 1 ↓
(3)

冲突 1: "效果不好"

这是最常见的冲突. PM 说效果不好, 算法说指标很好. 两个人说的"效果"根本不是一回事.

PM 说的"效果不好": 用户投诉多了 / 点击率下降 / NPS 掉了

算法说的"效果很好": 离线 AUC 0.85 / F1 提升了 3 个点 / 消融实验显著

→ 一个看线上用户行为, 一个看离线实验数据. 根本不在一个维度上.

解法: 对齐指标体系
  1. 建立"北极星指标": PM 和算法共同定义一个业务+技术都认可的核心指标
  2. 打通线上线下: 离线 AUC 只是筛选条件, 线上 A/B 测试才是金标准
  3. 定期 review: 每周看同一份数据 dashboard, 用同一套语言讨论
冲突 2: 速度问题 ↓
(4)

冲突 2: "模型迭代慢"

产品要周级迭代, 模型训练要月级周期. 两个时钟天然不同步.

PM 的焦虑: 竞品两周出一版, 我们的模型三个月没更新

算法的现实: 标注 1 万条数据要 2 周, 训练一轮要 3 天, 评测+调参又 1 周

→ 模型不是代码, 不能"今天写完明天上线".

解法: 双轨并行
  1. 快轨 (规则/Prompt): 用规则引擎或 Prompt 工程快速响应业务需求, 先上线验证
  2. 慢轨 (模型): 算法团队并行准备数据、训练、评测, 等模型就绪再替换快轨方案
  3. 共识排期: PM 提需求时附上优先级 + 可接受的等待时间, 算法回复可行性 + 预计时间
冲突 3: 数据问题 ↓
(5)

冲突 3: "数据要不到"

AI 产品的核心燃料是数据. 但数据往往拿不到 — 不是不给, 是不能给.

常见场景:

  • 用户隐私数据: 合规部门不批
  • 跨部门数据: 别的 BU 不愿意共享
  • 标注数据: 标注成本高, 预算不够
  • 历史数据: 格式混乱, 质量差, 无法直接使用
解法: 数据策略前置
  1. 合规先行: 在需求阶段就拉合规团队入场, 确认数据使用边界
  2. 脱敏方案: 匿名化 + 差分隐私 + 用户明确授权
  3. 数据飞轮: 从产品使用中持续收集数据, 而非一次性要大量数据
  4. 合成数据: 用 LLM 生成模拟数据做初期训练, 真实数据做微调
冲突 4: 能力边界 ↓
(6)

冲突 4: "这个需求没法做"

PM 想象中的 AI 无所不能. 现实中的 AI 有很硬的边界.

"没法做" 的三种含义:

  1. 技术不可行: 当前模型能力做不到 (如精确医疗诊断)
  2. 数据不支持: 没有足够的训练数据来支撑这个功能
  3. 法律不允许: 涉及合规风险 (如自动开处方)

→ 这三种"没法做"的解法完全不同. PM 要追问"哪种没法做".

解法: 需求降级 + 边界探索
  1. 追问具体原因: "是模型能力不够, 还是数据不够, 还是合规不允许?"
  2. 降级需求: 从"诊断" → "风险提示", 从"自动决策" → "辅助建议"
  3. 分步实现: 先做 80% 可行的部分, 剩下 20% 留给下个版本
冲突 5: 效果衰减 ↓
(7)

冲突 5: "上线后效果变差了"

模型上线第一周效果 90 分, 第三周变成 70 分. 这不是 bug, 是数据漂移 (Data Drift).

为什么会变差?

  • 用户行为变了: 训练数据是冬天的, 现在是夏天
  • 用户群变了: 初期用户 vs 大规模用户, 分布不同
  • 外部环境变了: 政策变化, 竞品变化, 市场变化
  • 反馈循环: 模型的输出影响了用户行为, 用户行为又影响了模型输入
解法: 监控 + 重训 + 预期管理
  1. 上线监控: 部署实时指标看板, 关键指标异常自动告警
  2. 定期重训: 每 2-4 周用最新数据重新训练模型
  3. 预期管理: 在 PRD 中写明"模型需要持续维护", 预留运维人力
  4. A/B 灰度: 新模型先灰度 10% 用户, 确认效果再全量

Vitamin 团队的 5 个协作场景

把 5 大冲突映射到你的 Vitamin 项目. 提前预防 > 事后救火.

场景 1 · "效果不好": Vitamin 的情绪分类准确率 85%, 但用户觉得"AI 不懂我". → 对齐: 把"用户满意度评分"加入模型优化目标.

场景 2 · "迭代慢": 营养知识库更新了, 模型还是旧知识. → 双轨: RAG 检索先上, 模型微调并行.

场景 3 · "数据要不到": 用户的经期数据是核心隐私. → 脱敏: 本地计算 + 聚合统计, 不上传原始数据.

场景 4 · "做不了": "AI 能不能判断用户是否怀孕?" → 降级: 只做"提醒用户去医院检查", 不做诊断.

场景 5 · "变差了": 夏天用户的饮食习惯变了, 营养建议不准了. → 监控: 按季度重训营养推荐模型.

面试金句:

"AI 产品上线不是终点, 而是起点. 好的 AI PM 在 PRD 里就写好模型监控、重训排期和数据飞轮方案, 而不是上线后才发现效果变差."

Agent 工作交接

A Agent 完成任务后, 怎么把上下文传给 B Agent? 信息丢失 = Multi-Agent 最大坑.

周期 Agent 说"你在黄体期", 然后交给营养 Agent...

但营养 Agent 不知道用户花生过敏!

推荐了花生酱三明治... 用户: ???

—— 交接设计, 决定 Multi-Agent 的成败.

  • 亲眼看到信息丢失如何导致回复质量暴跌

走起 →
(1)

交接 = 医生转科室时的病历

你从骨科转到康复科. 如果医生不传病历, 康复科医生对你的情况一无所知, 要从头问一遍. 更糟: 可能给你做了和骨科冲突的治疗.

Multi-Agent 的交接也一样: Agent A 知道的上下文, Agent B 一个字都不知道 — 除非你主动传过去.

每个 Agent 是独立的 LLM 调用, 它们之间不会自动共享任何信息.

交接信息丢失有多严重? ↓
(2)

信息丢失的 3 个等级

完整交接 Agent B 收到所有上下文: 用户状态、对话摘要、前一个 Agent 的结论、约束条件 → 回复精准个性化

部分丢失 关键字段缺失 (如过敏信息、前次建议) → 回复不够个性化, 可能冲突

严重丢失 几乎没有上下文传递 → Agent B 相当于跟新用户对话, 完全丧失连续性

3 种交接模式 ↓
(3)

3 种交接模式

全量传递

把完整对话历史传给下一个 Agent.

优: 不丢信息. 缺: 上下文很长, 成本高, 可能超 context window.

摘要传递

Agent A 输出一段摘要给 Agent B.

优: 精简. 缺: 摘要可能遗漏关键细节.

结构化 Context Object

预定义字段的 JSON 对象, 必填字段不可省略.

优: 标准化, 不易遗漏. 推荐方案.

什么是 Context Object? ↓
(4)

Context Object: 结构化的交接协议

{
  "user_id": "u_12345",
  "cycle_phase": "黄体期",
  "cycle_day": 22,
  "current_mood": "有点烦躁",
  "diet_today": ["燕麦", "牛奶"],
  "allergies": ["花生"],
  "previous_advice": "多补充镁",
  "conversation_summary": "用户因PMS不适来咨询,
    周期Agent已确认黄体期状态",
  "handoff_reason": "用户问了饮食建议,
    需要营养Agent处理",
  "timestamp": "2026-05-15T14:30:00"
}

关键: allergies 是必填字段. 如果这个字段丢了, 营养 Agent 可能推荐过敏食物 — 这是产品事故.

哪些字段是必须传的? ↓
(5)

必填字段 vs 可选字段

必填 (丢了就出事):
  • user_id — 不知道是谁, 无法个性化
  • allergies/禁忌 — 安全相关, 绝对不能丢
  • conversation_summary — 上下文连续性
  • current_mood — 决定回复语气
  • handoff_reason — Agent B 知道为什么接到这个任务
可选 (丢了体验降级但不出事):
  • diet_today — 没有的话不影响安全, 只是建议不够精准
  • cycle_day — 知道 phase 就够了, day 是锦上添花
  • previous_advice — 没有的话可能重复建议, 但不会出错
共享 Memory 怎么做? ↓
(6)

Shared Memory: 所有 Agent 共享的记忆

除了交接时传递 Context Object, 更好的方案是所有 Agent 共享一个 Memory 层.

Shared Memory 的实现:
  • 用户画像: 固定信息 (过敏、偏好、年龄段) 存在数据库, 每个 Agent 都能读取
  • 会话状态: 当前对话的上下文, 用 Redis 或内存缓存存储
  • 历史摘要: 过去 7 天的对话摘要, 定期压缩更新

Shared Memory 让 Agent 之间的"交接"变成"共同记忆" — 不需要传来传去, 大家都看同一份数据.

交接失败怎么处理? ↓
(7)

交接失败的兜底策略

再完美的设计也会有交接失败的时候. 兜底比完美更重要.

兜底策略:
  1. 必填字段缺失 → 不交接: 缺少 allergies 或 user_id, 拒绝交接, 打日志, 用当前 Agent 兜底回答
  2. 可选字段缺失 → 降级回答: Agent B 回答时标注"由于信息不完整, 建议仅供参考"
  3. 回退机制: 如果 Agent B 判断上下文不够, 可以主动问用户 "我需要确认一下, 你有没有食物过敏?"

Vitamin 交接设计 ↓
(9)

Vitamin 的交接协议

Vitamin Context Object 规范:
  • 必填 5 字段: user_id, allergies, current_mood, conversation_summary, handoff_reason
  • 可选 5 字段: cycle_phase, cycle_day, diet_today, previous_advice, timestamp
  • Shared Memory: 用户画像 (Redis) + 当前会话状态 (内存)
  • 交接协议: 必填字段缺失 → 拒绝交接 → 日志告警 → 当前 Agent 兜底
面试金句:

"Multi-Agent 最大的坑不是路由, 而是交接. 我给 Vitamin 设计了结构化的 Context Object, 5 个必填字段确保安全信息(如过敏)永远不丢失. 同时用 Shared Memory 让所有 Agent 共享用户画像, 减少交接开销."

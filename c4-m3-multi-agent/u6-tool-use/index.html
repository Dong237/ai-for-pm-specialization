Tool Use

让 LLM 调用外部工具 — 查天气、查日历、查数据库. Coze "插件" 的本质就是这个.

LLM 自己不能查天气、查数据库...

那 ChatGPT 怎么能搜网页、画图、跑代码的?

答案: Function Calling — LLM 学会了"叫外卖".

—— 看看这个魔法是怎么工作的.

走起 →
(1)

LLM 的核心局限: 只能"说", 不能"做"

LLM 本质上只做一件事: 输入文本 → 输出文本. 它不能访问网络, 不能查数据库, 不能设置闹钟. 它是一个纯文本生成器.

没有 Tool Use 的 LLM:

用户: "深圳今天几度?" → LLM: "大概 25-30 度吧" (编的, 因为它不能查天气)

用户: "帮我设一个明天的提醒" → LLM: "好的, 我帮你设好了" (假的, 它根本没有这个能力)

Function Calling 怎么解决这个问题? ↓
(2)

Function Calling = LLM 学会了"叫外卖"

核心概念

Tool Use 的本质:

LLM 不执行工具,
它只是决定调用哪个工具 + 传什么参数

真正执行的是你的后端代码. LLM 只是"点菜", 厨房(后端)才做菜.

具体流程长什么样? ↓
(3)

第 1 步: 告诉 LLM "你有哪些工具可以用"

在调用 LLM 之前, 你需要定义工具列表 (Tool Schema). 用 JSON 描述每个工具的名字、功能、参数.

{
  "tools": [{
    "name": "get_weather",
    "description": "获取指定城市的实时天气",
    "parameters": {
      "city": {"type": "string", "description": "城市名"},
      "date": {"type": "string", "description": "日期"}
    }
  }]
}

这就是 Coze 里"创建插件"时你填的那些字段 — 名称、描述、参数. 本质是在给 LLM 一份"工具菜单".

LLM 怎么决定要不要调用工具? ↓
(4)

第 2 步: LLM 决定 — "这个问题需要工具吗?"

LLM 看到用户问题 + 工具列表, 它会判断: 自己能答就直接答, 需要外部数据就生成 function_call.

"你好" → LLM 直接回答 (不需要工具)

"深圳今天几度?" → LLM 生成 function_call (需要实时数据)

LLM 的决策基于: 问题是否需要它不知道的信息 + 工具列表里有没有能回答这个问题的工具.

function_call 长什么样? ↓
(5)

第 3 步: LLM 输出结构化的调用指令

LLM 不会输出自然语言, 而是输出一段结构化的 JSON — 这是它和工具之间的"合同".

{
  "function_call": {
    "name": "get_weather",
    "arguments": {
      "city": "深圳",
      "date": "2026-05-15"
    }
  }
}

注意: LLM 是生成了这段 JSON, 不是执行了它. 你的后端代码拿到这段 JSON, 解析参数, 去调真正的天气 API.

工具返回数据之后呢? ↓
(6)

第 4 步: 工具返回 → LLM 整合回答

工具返回了真实数据, 你把数据塞回 LLM 的 context, LLM 用自然语言把数据"翻译"给用户.

// 工具返回:
{"temp": 28, "condition": "多云", "aqi": 45}

// LLM 整合后回答:
"今天深圳28度, 多云, 空气质量优!
 适合户外运动, 但记得涂防晒~"

看到了吗? LLM 做了两件事: (1) 用真实数据回答, 不编造; (2) 加上了人性化的语气和建议.

这和 Coze 插件有什么关系? ↓
(7)

Coze 插件 = Function Calling 的可视化包装

Coze/Dify/扣子 里的"插件", 本质就是 Function Calling 的封装:

  • 插件名称 = function name (get_weather)
  • 插件描述 = function description
  • 输入参数 = function parameters (city, date)
  • API 地址 = 你的后端服务 URL
  • 返回值 = function result (JSON 数据)

平台只是帮你省去了写代码的步骤, 底层逻辑完全一样.

PM 认知升级: 当你在 Coze 里"配置插件"时, 你实际上是在给 LLM 写 Tool Schema. 理解这个, 你就知道为什么有些插件"不被调用" — 是因为你的 description 写得不够好, LLM 判断不了什么时候该用它.

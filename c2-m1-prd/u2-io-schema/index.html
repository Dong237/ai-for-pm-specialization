Unit 2·
什么进去, 什么出来

算法工程师拿到你的 PRD, 第一个问题: "Input 是什么? Output 是什么? 什么格式?" 答不上来, 他没法开工.

算法工程师说: "你先定义好 Input/Output, 我才能开始写模型接口."

Input 不就是用户的问题吗? 还要传什么?

Output 是一段文字? 还是 JSON? 怎么约定?

—— 这节课搞定.

10 步学完, 你会:
  • 知道 AI 功能的 Input Schema 包含哪些核心字段
  • 会选择合适的 Output 格式 (JSON / Markdown / 纯文本)
  • 给每个字段加约束 (类型、必填、长度限制)
  • 用 IO Schema 和算法工程师高效对齐需求
走起 →
(1)

算法工程师的第一个问题

你写好了一个 AI 功能的 PRD, 交给算法同学. 他看完后说:

算法工程师:
"这个功能, 我需要调 LLM 的 API. 你告诉我: 什么数据传进去, 我给你返回什么?"

这个问题就是在问 IO Schema — Input/Output 的数据结构定义. 就像一份 合同: PM 定义 "我需要什么", 算法承诺 "我给你什么".

传统功能的 "接口" 通常由后端工程师定义. 但 AI 功能的 IO 涉及模型能力边界, 所以 PM 必须参与定义. 你不需要写代码, 但必须说清楚 "业务上需要哪些字段".

那 "说用户问题" 不够吗? ↓
(2)

"输入用户问题, 输出 AI 回答" — 这不够

✗

模糊版 PRD: "用户输入问题, AI 给出健康建议"

看起来合理, 其实啥都没说.

算法会问: "问题" 是纯文本还是选择题? 需要传用户历史数据吗? "建议" 是一段话还是分点? 最长多少字? 需要返回置信度吗?
✗

模糊版 PRD: "AI 生成每日总结"

"总结" 包含什么? 多长? 什么格式?

算法会问: 总结需要什么输入数据? 只看今天还是看历史? 输出是纯文本还是 JSON (前端好解析)? 包含标签吗? 有风险提示字段吗?
真相

AI 功能的 Input 远不止 "用户问题".

Input = 用户输入 + 上下文 + 参数 + 约束

Output 也不只是 "一段文字". 它需要明确的格式、字段、约束.

Input Schema 到底包含什么? ↓
(3)

Input Schema 的 5 类核心字段

一个 AI 功能的输入, 通常包含这 5 类信息:

INPUT SCHEMA
{ // 1. 用户直接输入 "user_query": "我最近总是觉得很累, 怎么回事?", // 2. 上下文数据 (系统自动拼接) "user_id": "u_12345", "body_state": "疲惫期", "metrics": [ {"type": "sleep", "value": 5.5, "unit": "hours"}, {"type": "steps", "value": 2100}, {"type": "heart_rate", "value": 82} ], // 3. 历史数据范围 "history_days": 7, // 4. 输出控制参数 "max_tokens": 300, "language": "zh-CN", // 5. 系统指令 (System Prompt) "system_prompt": "你是 Vitamin 健康助手..." }

关键认知: 用户只输入了一句话 "我最近总是觉得很累", 但传给模型的 Input 是一个包含 8 个字段的 JSON. 其中 7 个是系统自动补充的上下文. 这就是 PM 需要在 PRD 里定义的东西.

Output 呢? ↓
(4)

Output Schema: 4 种常用格式

模型的输出, 根据下游消费者不同, 选择不同格式:

JSON (结构化)

前端需要解析、分块展示时用. 最推荐.
{"summary":"...", "tags":[]}

Markdown

需要富文本渲染 (标题/列表/加粗) 时用.
## 总结\n- 睡眠不足...

纯文本

简单场景, 直接展示一段话.
你今天的状态还不错...

结构化对象

复杂场景, 包含多个子对象.
{"analysis":{}, "plan":{}}

PM 决策点: 选格式不是技术决策, 是产品决策. 问自己: "前端怎么用这个输出?" 如果需要分块展示 → JSON. 如果直接显示 → 纯文本. 如果有格式要求 → Markdown.

看一个完整例子 ↓
(5)

完整示例: Vitamin 每日总结的 IO

把 Input 和 Output 放在一起看, 这就是写进 PRD 的 IO Schema:

OUTPUT SCHEMA
{ // 主体内容 "summary_text": "你今天睡了 5.5 小时, 比平时少了 1.5 小时. 心率偏高 (82bpm), 建议今天多休息, 避免高强度运动.", // 置信度 (兜底决策用) "confidence": 0.91, // 标签 (前端展示用) "tags": ["睡眠不足", "心率偏高", "建议休息"], // 健康建议 (分条展示) "suggestions": [ "今晚争取 11 点前入睡", "午休 20 分钟可以缓解疲劳", "今天步数偏少, 可以散步 15 分钟" ], // 风险标记 (是否需要医疗提醒) "risk_flag": false, // 消耗统计 (成本核算用) "token_count": 487 }

注意 confidence 和 token_count 这两个字段 — 它们不是给用户看的, 是给 系统决策 用的. confidence 决定走不走兜底; token_count 用来追踪成本.

好的 Schema 还要加约束 ↓
(6)

Schema 约束: 没有约束就是垃圾进垃圾出

光定义字段名不够. 每个字段都需要约束, 否则 "什么都可能发生":

字段类型必填约束默认值
user_querystring✔1-500 字符—
metrics[]array✔至少 1 项—
history_daysnumber✖1-307
max_tokensnumber✖50-500300
summary_textstring✔50-300 字—
confidencenumber✔0.0-1.0—
tags[]array✖0-5 个标签[]
risk_flagboolean✖—false

没有约束的后果: summary_text 没限制长度 → AI 可能输出 2000 字, 前端排版炸掉. confidence 没返回 → 兜底机制失效, 错误的回答直接展示给用户.

Schema 在团队里怎么用? ↓
(7)

IO Schema 是四方的合同

Schema 不是 PM 一个人写的, 它在整个团队中流转:

PM定义业务字段
+ 约束
→
算法实现模型接口
保证 Schema
→
前端按 Schema 解析
+ 渲染 UI
→
QA按 Schema 写
测试用例

PM 的角色: 你不需要写 API 文档, 但你需要用 "业务语言" 定义清楚每个字段的含义和约束. 算法同学会把它翻译成技术接口.

好的 IO Schema = 减少 80% 的需求返工. 因为大家从第一天就对齐了 "什么进去、什么出来".

面试金句:

"在 AI PRD 里, IO Schema 就像合同: PM 定义业务字段和约束, 算法承诺按 Schema 返回. 这解决了 AI 功能需求对齐的第一个问题 — '什么进去, 什么出来'. 每个字段都要有类型、必填标记和长度限制, 否则就是在给团队挖坑."

总结 + 作业 ↓
(10)

一句话规则 + 课后作业

一句话:

IO Schema = AI 功能的合同.
PM 定义 "什么进去, 什么出来, 什么约束".

没有 Schema 就开工 = 没有合同就签字. 返工概率 100%.

课后作业 (30 分钟):
  1. 选 Vitamin 的 "健康问答" 功能, 写出完整的 Input Schema (至少 5 个字段, 包含类型和必填标记).
  2. 为同一个功能写 Output Schema, 并决定用 JSON 还是 Markdown, 写 50 字说明选择理由.
  3. 找一个你熟悉的非 AI 产品, 想象给它加一个 AI 功能, 写出这个功能的 IO Schema.
  4. 问 ChatGPT: "帮我生成一个健康问答 API 的 JSON Schema". 对比它生成的和你写的, 有什么差异?

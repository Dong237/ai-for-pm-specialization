数据从哪来? 谁能看?

你把用户的健康数据传给了 OpenAI 的 API. 这些数据会被存储吗? 会被用来训练模型吗? 用户知道吗?

法务找我: "用户的健康数据有没有传给第三方?"

我: "呃... 是通过 API 发给 OpenAI 的..."

法务: "用户隐私协议里写了吗? 数据保留多久?"

—— 这些必须写进 PRD.

10 步学完, 你会:
  • 知道 AI 产品涉及的 3 类数据 (训练/检索/日志)
  • 了解数据流向: 用户 → App → API → 模型提供商
  • 会在 PRD 里写数据来源 + 隐私要求栏
  • 用 checklist 自检数据合规性

(1)

AI 产品的数据, 比传统产品复杂得多

传统 App: 用户数据存在自己的数据库. 你完全控制. AI 产品: 用户数据可能经过 3 个地方:

用户设备 (App)
↓
用户健康数据
你的服务器
↓
API 请求 (含用户数据)
模型提供商 (OpenAI / 等)
↓
生成回答 (可能被日志记录)
是否用于模型训练?

核心问题: 用户的健康数据 (体重/经期/心率) 经过了第三方 (模型提供商). 这涉及: 数据出境、第三方共享、隐私合规.

3 类数据, 逐个看 ↓
(2)

AI 产品涉及的 3 类数据

1. 训练数据: 模型是用什么数据训练的? 如果你 fine-tune 了一个模型, 训练数据的来源和授权是什么?

2. 检索数据 (RAG): 如果你用了 RAG (检索增强生成), 检索的知识库里有什么? 是公开数据还是用户数据?

3. 日志数据: 每次 API 调用的输入/输出是否被记录? 记录多久? 谁能访问? 这些日志里包含用户的原始问题和 AI 的回答.

对于 Vitamin 这样的健康 App, 健康数据是 敏感个人信息. 处理标准比一般数据更严格.

PRD 里怎么写? ↓
(3)

PRD 数据栏要回答的 6 个问题

Q1. 数据来源: AI 功能需要哪些数据? 从哪获取? (用户输入 / 数据库 / 第三方)

Q2. 数据传输: 数据是否传给第三方 API? 传输是否加密?

Q3. 数据存储: 日志保留多久? 存在哪里? (自有服务器 / 云)

Q4. 数据访问: 谁能看到这些数据? (开发 / PM / 算法 / 客服)

Q5. 用户控制: 用户能否查看/删除自己的数据?

Q6. 合规要求: 需要符合哪些法规? (个人信息保护法 / GDPR / 行业规范)

逐个展开 ↓
(4)

数据传输: API 调用时传了什么?

关键决策: 传给模型 API 的数据, 是否包含可识别的个人信息?

✓ 推荐做法

  • 脱敏后再传 (去掉姓名/手机号/位置)
  • 只传聚合指标 (平均心率), 不传原始数据
  • 使用匿名用户 ID, 不传真实 user_id
  • API 传输全程 HTTPS 加密
  • 确认 API 提供商不使用数据训练模型

✗ 禁止做法

  • 把用户姓名/手机号写进 Prompt
  • 传原始医疗报告给 API
  • 使用默认 API 设置 (可能允许数据训练)
  • 不加密传输健康数据
  • 不告知用户数据流向
数据保留策略 ↓
(5)

数据保留: 留多久, 谁能看

数据类型保留时长访问权限到期处理
API 调用日志30 天算法 + 后端 (仅脱敏)自动删除
用户反馈数据90 天PM + 算法归档 → 180 天后删除
评测用例长期算法 (需脱敏)年度审计清理
用户健康数据用户删除即删仅用户本人用户主动操作

原则: 数据最小化 — 不需要的数据不收集, 收集了的数据到期就删. 这不只是合规要求, 也是用户信任的基础.

用户权利 ↓
(6)

用户的数据权利: 查看、删除、拒绝

个人信息保护法和 GDPR 赋予用户的基本权利:

查看权: 用户可以查看 AI 功能存了哪些关于自己的数据.

删除权: 用户可以要求删除所有个人数据. 删除后, AI 功能可能需要重新 "认识" 用户.

拒绝权: 用户可以拒绝 AI 功能使用自己的数据. 此时需要提供非 AI 的替代体验.

知情权: 用户必须知道数据被发送到了第三方 API.

写进 PRD: "设置页增加 '我的数据' 入口, 用户可查看 AI 功能使用的数据列表, 一键删除, 删除操作 72 小时内生效."

合规法规 ↓
(7)

关键法规: PM 需要知道的底线

法规适用范围对 AI 产品的要求
个人信息保护法中国收集个人信息需告知 + 同意; 敏感信息需单独同意; 跨境传输需评估
GDPR欧盟数据最小化; Privacy by Design; 用户有权删除; 需做 DPIA
EU AI Act欧盟高风险 AI 需要额外文档 + 人工监督; 健康场景可能被归类为高风险

PM 注意: 你不需要成为法务专家, 但必须知道: (1) 健康数据是敏感数据, 标准更高; (2) 传给第三方 API 需要用户知情同意; (3) PRD 里写清楚数据处理规则, 法务才能审核.

Vitamin AI 功能 — 数据隐私 Checklist

Vitamin 的隐私栏 ↓
(9)

Vitamin PRD 的数据隐私栏

数据来源: 用户主动记录的健康指标 (体重/心率/睡眠/经期/饮食), 存储在自有数据库.

数据传输: 调用 LLM API 时, 传脱敏后的聚合指标, 不传原始数据. HTTPS 加密. API 提供商承诺不用于训练.

数据存储: API 日志保留 30 天; 用户反馈保留 90 天; 评测数据长期保留但脱敏. 到期自动删除.

用户权利: 设置页 "我的数据" 入口, 可查看/删除. 可关闭 AI 功能.

合规声明: 符合个人信息保护法; 健康数据获用户单独同意; 无跨境传输 (使用境内 API 节点).

面试金句:

"AI 产品的数据隐私不是事后打补丁, 是 PRD 里的必写模块. 我会在 PRD 里回答 6 个问题: 数据从哪来、传给谁、存多久、谁能看、用户能不能删、需要什么合规. 特别是健康数据属于敏感个人信息, 需要用户单独同意, 传给 API 前必须脱敏."

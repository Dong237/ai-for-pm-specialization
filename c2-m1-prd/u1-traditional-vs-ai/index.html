(1)

先回忆: 传统 PRD 长什么样?

你一定写过, 或者至少见过这样的 PRD:

传统 PRD 示例: "搜索功能"

功能描述: 用户在搜索框输入关键词, 点击搜索按钮, 系统返回匹配的文章列表.

交互流程: 输入框获焦 → 输入文字 → 点击搜索 → 展示结果列表 → 点击结果进入详情.

验收标准: 搜索响应时间 < 500ms; 结果按相关度排序; 无结果时显示 "暂无结果".

注意一个关键特征: 同样的输入, 永远产生同样的输出. 搜索 "跑步", 今天和明天的结果是一样的 (数据没变的话).

这叫 确定性系统. 传统软件几乎都是确定性的.

但 AI 功能不是这样的 ↓
(2)

AI 功能的 PRD, 传统方法写不了

✗

错觉: "AI 功能也是输入→输出, 一样写就行"

功能描述: 用户输入健康问题, AI 返回建议.

为啥不行: "AI 返回建议" 是什么建议? 每次一样吗? 错了怎么办? 多长合适? 这些全没说.
✗

错觉: "验收标准就是 AI 答对就行"

验收标准: AI 回答正确.

为啥不行: "正确" 怎么定义? 谁来判断? 正确率目标是多少? 99% 还是 80%? 不同场景不一样.
真相

AI 产品是概率性系统:

同样的输入, 可能产生不同的输出.

而且输出可能是错的. 你作为 PM, 必须在 PRD 里写清楚: 什么算 "够好", 错了怎么办.

→ 传统 PRD 不需要回答这些问题, 因为传统软件不会 "答错".

那传统 PRD 的核心结构是什么? 先拆清楚 ↓
(3)

传统 PRD 的 3 大支柱

任何传统 PRD, 不管格式怎么变, 核心就 3 块:

📄

功能描述

用户做什么, 系统做什么. 用文字 + 流程图描述确定性行为.

🔄

交互流程

每一步的 UI 状态: 点击哪里 → 弹出什么 → 跳转到哪.

✅

验收标准

怎么算 "做完了": 响应时间、边界条件、异常处理.

这 3 块围绕一个核心假设: "人操作 → 系统确定性响应". 链条是闭合的, 每个环节都可预测.

比如你写 "用户点击收藏按钮 → 按钮变为已收藏状态 → 收藏列表 +1". 这里没有任何 "可能", 没有 "大概率", 没有 "如果 AI 理解错了". 一切都是确定的.

AI PRD 保留这 3 块, 但还要加上新的模块 ↓
(4)

AI PRD 多出来的 4 个模块

传统 PRD 的 3 大支柱不够用了. AI PRD 在此基础上, 需要 额外写清楚 4 件事:

模块 1

模型 IO 描述

什么数据进去 (Input Schema), 什么数据出来 (Output Schema). 给算法工程师的 "合同".

模块 2

效果指标定义

准确率、接受率、拒答率、延迟. 每个指标的 "及格线" 是多少.

模块 3

错误兜底机制

AI 答错了怎么办? 重试 / 切模型 / 切规则 / 转人工 / 拒答. 5 级兜底链.

模块 4

Prompt 模板管理

Prompt 是产品资产. 版本号、变更日志、A/B 实验、灰度发布.

一句话记: 传统 PRD = 功能 + 交互 + 验收. AI PRD = 功能 + 交互 + 验收 + IO + 指标 + 兜底 + Prompt.

后面的 Unit 2-5 会一个一个拆解这 4 个模块.

为什么要多这 4 个? 核心是思维方式变了 ↓
(5)

从 "确定性思维" 到 "概率性思维"

传统 PM 的思维是 if/else 逻辑. AI PM 的思维是概率分布 + 阈值.

传统思维

确定性: if → then

if (用户点击收藏) { item.saved = true; showToast("已收藏"); }
→

AI 思维

概率性: predict → verify → fallback

result = model.predict(input); if (result.confidence >= 0.85) { show(result); } else { fallback(); // 兜底 }

关键区别: 传统软件没有 confidence (置信度) 这个概念. AI 系统的每个输出都带着一个 "我有多确定" 的分数. 这个分数决定了: 直接展示, 还是走兜底.

具体到 PRD 结构上, 差异有多大? ↓
(6)

结构对比: 传统 PRD vs AI PRD

放在一起看, 差异一目了然:

PRD 章节 传统 PRD AI PRD
需求背景 ✔ 用户痛点 + 目标 ✔ 相同, 但要说明为什么用 AI 而不是规则
功能描述 ✔ 确定性流程 ✔ 概率性流程 + 置信度分支
交互设计 ✔ 每步 UI 状态 ✔ 加载态 + 不确定态 + 兜底态
模型 IO ✖ 不需要 ✔ NEW Input/Output Schema
效果指标 ✖ 只有性能指标 ✔ NEW 准确率/接受率/延迟
错误兜底 ✖ 只有异常处理 ✔ NEW 5 级兜底链
Prompt 管理 ✖ 不存在 ✔ NEW 版本/A/B/灰度
成本预算 ✖ 服务器固定成本 ✔ NEW token × 单价 × 频率
数据隐私 ▬ 简单合规声明 ✔ 升级 数据来源/保留/访问权限

绿色标注的部分就是 AI PRD 独有的. 这 6 个 "新增/升级" 项, 就是接下来 Unit 2 - Unit 7 要一个一个教你写的内容.

没有这些模块会怎样? ↓
(7)

不写这些模块, 会出什么事?

这不是 "锦上添花", 是 "没有会出事":

场景 1: 没写 IO Schema
算法工程师问你: "这个接口的输入是 JSON 还是纯文本? 需要传用户历史吗?" 你答不上来. 返工 3 天.

场景 2: 没写效果指标
上线后老板问: "AI 回答准不准?" 你说 "挺准的". 老板问 "多准? 有数据吗?" 你答不上来.

场景 3: 没写兜底机制
AI 给用户推荐了一个有潜在危险的健康建议. 没有兜底, 直接展示给用户了. 投诉 + 法律风险.

场景 4: 没写成本预算
上线一个月, 财务找你: "这个 AI 功能烧了 12 万 API 费用, 预算只有 3 万." 功能被砍.

这些场景不是假设, 是真实发生在 AI 产品团队里的 "血泪故事". AI PRD 的新增模块, 每一个都是一个 "防坑栏".

来亲手对比看看 ↓
(8)

亲手对比: 传统 PRD vs AI PRD 骨架

切换两个标签, 看看同一个功能的 PRD, 传统版和 AI 版有什么不同:

传统 PRD: Vitamin 每日健康总结

1. 需求背景

用户希望每天看到自己的健康数据总结, 了解当日状态.

2. 功能描述

系统根据用户当日记录的数据, 生成一段健康总结文字, 展示在首页卡片.

3. 交互流程

用户打开首页 → 看到总结卡片 → 点击展开详情 → 可分享.

4. 验收标准

页面加载 < 1s; 总结字数 50-150 字; 无数据时显示引导.

(共 4 个章节 — 传统 PRD 就够了)

AI PRD: Vitamin 每日健康总结

1. 需求背景

用户希望每天看到 AI 生成的个性化健康总结. 选择 AI 而非模板, 因为每个人的数据组合不同.

2. 功能描述

LLM 根据用户当日数据 + 历史趋势, 生成个性化总结. 置信度 < 0.85 时走兜底.

3. 交互流程

加载态(骨架屏) → AI 生成完成 → 展示总结 → 用户可点"不准确"反馈.

4. 验收标准

端到端延迟 < 3s; 总结字数 80-200 字; 用户满意度 > 4.0/5.

5. 模型 IO Schema

Input: {user_id, date, metrics[], history_days:7}. Output: {summary_text, confidence, tags[]}.

6. 效果指标

准确率 > 90%; 接受率 > 85%; 拒答率 < 5%; P95 延迟 < 3s.

7. 兜底机制

置信度低 → 模板兜底; 超时 → 缓存昨日总结; 异常 → "今日数据不足, 明日再看".

8. Prompt 管理

Prompt v1.0; 变更需 PM + 算法双签; A/B: 简洁版 vs 详细版.

9. 成本预算

~500 token/次 × $0.003/1K × 10K DAU = ~$15/天 ≈ $450/月.

10. 数据与隐私

仅用脱敏后的健康指标; 不传原始文本; 日志保留 30 天; 用户可删除.

(共 10 个章节 — 多了 6 个 AI 专属模块)

拿 Vitamin 实际功能来看 ↓
(9)

Vitamin 的 AI 功能, 传统 PRD 差在哪?

Vitamin 有 5 个状态追踪 + AI 健康建议功能. 如果只用传统 PRD 写, 会漏掉什么:

✓ 传统 PRD 能覆盖的

  • 记录页的 UI 交互流程
  • 数据展示的图表样式
  • 设置页的功能列表
  • 推送通知的触发逻辑
  • 基础异常处理 (断网、空数据)

✗ 传统 PRD 覆盖不了的

  • AI 生成总结的质量标准 (多准算合格?)
  • AI 输出错误时的用户体验
  • 模型输入需要传哪些数据字段
  • 每月 API 调用成本上限
  • Prompt 变更的审批和回滚流程
  • 用户健康数据是否传给第三方模型
面试金句:

"传统 PRD 描述确定性系统, AI PRD 描述概率性系统. AI PRD 比传统 PRD 多出 4 大模块: IO Schema、效果指标、兜底机制、Prompt 管理. 另外还要加成本预算栏和数据隐私栏. 这是因为 AI 产品的输出有不确定性, PM 需要在 PRD 里定义 '什么算够好' 和 '不够好时怎么办'."

(10)

一句话规则 + 课后作业

一句话:

写 AI PRD 时, 永远多问一个问题:
"如果 AI 答错了, 怎么办?"

这个问题的答案, 就是传统 PRD 和 AI PRD 的本质差异.

  2. 打开 Vitamin 的产品文档, 列出 3 个需要 AI 能力的功能, 给每个功能写一句: "如果 AI 答错了, 兜底方案是___".

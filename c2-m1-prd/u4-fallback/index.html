Unit 4·
AI 答错了, 怎么办?

AI 不是不会错 — 它一定会错. 问题不是 "会不会错", 而是 "错了有没有兜底".

Vitamin 的 AI 给用户说 "经期可以剧烈运动"...

这是错的! 但已经展示给用户了!

为什么没有任何兜底机制?

—— 因为 PRD 里没写.

10 步学完, 你会:
  • 知道 AI 产品的 5 类兜底机制
  • 能为每个 AI 功能设计兜底链 (cascade)
  • 理解置信度阈值和兜底的关系
  • 给 Vitamin 写出完整的兜底方案
走起 →
(1)

AI 答错不是 bug — 是概率性系统的必然

Module 1 学过: LLM 输出是概率性的. 这意味着 错误不是意外, 而是系统特性.

传统软件: 代码写对了, 就不会出错. AI 产品: 模型再好, 也有概率答错.

PM 的职责: 你不能阻止 AI 犯错 (那是算法的活). 但你必须设计 "犯错后的体验". 用户看到的不应该是一个错误的答案, 而是一个优雅的降级.

没有兜底会怎样? ↓
(2)

没有兜底的 3 个灾难场景

场景 1: 健康建议出错
AI 说 "感冒了可以多喝冰水". 没有置信度检查, 直接展示. 用户照做, 症状加重, 投诉 + 法律风险.

场景 2: 模型超时
LLM API 偶尔响应超过 10 秒. 没有超时兜底, 用户盯着空白页面, 然后卸载 App.

场景 3: 模型宕机
OpenAI API 偶尔会 503. 没有备用方案, 整个 AI 功能不可用, 用户以为 App 坏了.

共同原因: PRD 里没有 "错误兜底" 模块. 开发按 happy path 实现, 没人想过 unhappy path.

5 类兜底, 逐个看 ↓
(3)

5 类兜底, 从轻到重

兜底不是只有 "转人工" 一种. 好的兜底是一条 从轻到重的链条, 逐级升级:

L1

重试 (Retry)

换一个 Prompt 或调一下参数, 再问模型一次. 成本最低, 能解决偶发错误.

适用: 超时、偶发低质量输出. 限制: 最多重试 1-2 次, 避免死循环.

L2

切模型 (Model Switch)

主模型不行, 切到备用模型. 比如主用 GPT-4o, 备用 GPT-4o-mini 或 DeepSeek.

适用: 主模型 API 故障或准确率下降. 限制: 备用模型效果可能差一些.

L3

切规则 (Rule Fallback)

AI 不行, 退回规则引擎 / 模板 / 缓存. 比如展示上一次的总结, 或用模板生成基础版.

适用: AI 完全不可用时. 限制: 规则版的体验不如 AI 版个性化.

L4

转人工 (Human Handoff)

提交给人工审核或客服. 适用于高风险场景 (医疗、法律).

适用: AI 置信度极低 + 用户问题涉及安全. 限制: 人工成本高, 响应慢.

L5

拒答 (Refuse)

最后的防线. 直接告诉用户 "暂时无法回答这个问题", 给出替代建议.

适用: 所有兜底都失败. 关键: 拒答文案必须友好, 提供下一步指引.

这 5 级怎么串起来? ↓
(4)

兜底链: 逐级降级的 Cascade

5 类兜底不是 "选一个用", 而是 按顺序逐级触发:

AI 模型正常回答 (confidence ≥ 0.85)
↓
confidence < 0.85 或超时
↓
L1: 换 Prompt 重试
↓
重试仍不达标
↓
L2: 切备用模型
↓
备用也失败
↓
L3: 规则/模板兜底
↓
涉及高风险
↓
L4: 转人工
↓
人工也无法处理
↓
L5: 友好拒答 + 替代建议

实际中不一定要 5 级全用. 低风险场景可能 L1 + L3 + L5 就够了. 高风险场景才需要完整链条.

置信度阈值怎么定? ↓
(5)

触发兜底的关键: 置信度阈值

什么时候走兜底? 核心依据是模型返回的 confidence 值:

confidence ≥ 0.85: 直接展示 AI 回答. ✔

0.70 ≤ confidence < 0.85: 展示 AI 回答, 但加 "仅供参考" 标签 + 反馈按钮.

0.40 ≤ confidence < 0.70: 不展示 AI 回答, 走 L1 重试. 重试后仍不达标, 走 L3 规则兜底.

confidence < 0.40: 直接走 L3 或 L5, 不尝试重试.

PM 决策: 阈值不是固定的, 需要根据业务风险调整. 健康建议的阈值应该设高 (0.90+); 趣味内容可以设低 (0.70+).

兜底文案怎么写? ↓
(6)

兜底文案 = 用户体验的最后防线

兜底不只是技术方案, 还是 用户体验设计:

✗

坏的兜底文案

"系统错误, 请稍后重试."

冰冷, 没有信息量, 用户不知道发生了什么.
✔

好的兜底文案

"这个问题涉及专业医疗知识, 我还不够有把握回答. 建议你咨询医生, 或查看我们的健康百科 >"

诚实、有温度, 提供了下一步行动.

兜底文案 3 原则: (1) 承认局限, 不假装没出问题; (2) 解释原因, 让用户理解; (3) 给替代方案, 别让用户卡在那里.

写进 PRD 的格式 ↓
(7)

PRD 里的兜底模块怎么写

在 PRD 的兜底机制章节, 用一个表格讲清楚:

触发条件兜底级别动作用户看到什么
confidence < 0.85L1 重试换 Prompt 重试 1 次加载动画 (用户无感知)
重试后仍 < 0.85L3 规则用模板生成基础版总结"基础版总结" + "AI 版本稍后更新" 标签
API 超时 (> 5s)L3 缓存展示昨天的总结"这是昨天的总结, 今日版本正在生成中"
涉及医疗禁忌L5 拒答直接拒答 + 建议就医"建议咨询专业医生" + 医院链接
API 503 宕机L3 缓存展示缓存的最近总结"AI 功能暂时维护中, 先看看上次的总结"

PM 写这个表, 不用写代码. 但这个表必须写清楚, 否则开发会按自己的理解实现 — 通常是最简单的 "报错弹窗".

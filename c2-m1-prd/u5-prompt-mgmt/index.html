Unit 5·
Prompt 是产品代码

你改了一个 Prompt 里的逗号, AI 的输出风格就变了. Prompt 不是随便写写的文本, 它是你产品的 "灵魂代码".

上周改了一下 Prompt, AI 的语气突然变生硬了...

改之前的版本呢? 找不到了!

Prompt 不应该像代码一样有版本管理吗?

—— 没错, 这就是今天要学的.

10 步学完, 你会:
  • 理解为什么 Prompt 是产品资产, 需要版本管理
  • 知道 Prompt 变更的审批流程
  • 学会 A/B 测试 Prompt 的方法
  • 为 Vitamin 设计 Prompt 管理规范
走起 →
(1)

改一个词, 输出大变 — Prompt 的蝴蝶效应

传统代码: 改一行, 编译器会告诉你有没有 bug. Prompt: 改一个词, 没有编译器, 没有报错, 但输出可能全变了.

真实案例: Prompt 从 "你是健康助手" 改成 "你是专业健康顾问", AI 的回答从温暖友好变成了冰冷的医学术语. 用户接受率从 86% 掉到了 52%.

这就是为什么 Prompt 需要像代码一样管理: 版本号、变更日志、评测、回滚.

但很多团队还在随便改 Prompt ↓
(2)

大多数团队的 Prompt 管理现状: 一团糟

✗

Prompt 写在代码里, 改了没人知道

工程师想 "优化一下", 直接改了 Prompt, 没通知 PM.

结果: AI 语气变了, 用户反馈 "AI 怎么变了", PM 不知道发生了什么.
✗

Prompt 放在飞书文档, 版本混乱

"Prompt 最终版.docx", "Prompt 最终版v2.docx", "Prompt 最终版v2-改.docx"

结果: 没人知道线上跑的是哪个版本. 出了问题, 无法回滚.
真相

Prompt 是产品资产, 地位等同于 产品代码.

它需要: 版本号 · 变更日志 · 评审 · A/B · 回滚

每一次 Prompt 变更, 都相当于一次产品发版.

Prompt 的生命周期是什么? ↓
(3)

Prompt 的 4 阶段生命周期

✍

1. 起草

PM 写初版 Prompt, 在 Playground 调试, 直到效果满意.

🔍

2. 评审

PM + 算法双签审批. 跑评测集, 对比新旧版指标.

📊

3. 灰度/A/B

5% 用户先用新版. 观察指标, 无退步则逐步放量.

🚀

4. 全量/回滚

指标达标, 全量发布. 出问题, 一键回滚到上一版.

关键: 这个流程跟传统的 "代码发版" 几乎一样. 只是对象从代码变成了 Prompt. PM 在这里的角色是 "Prompt 的产品经理" — 决定改什么、怎么测、什么时候发.

版本号怎么编? ↓
(4)

Prompt 版本号规范

借鉴软件的语义化版本: 主版本.次版本:

v1.0 → v1.1: 小修改 — 调语气、改措辞、加约束. 效果应该只是微调.

v1.x → v2.0: 大改 — 结构重写、增加新的输入字段、改变输出格式. 效果可能显著变化.

每个版本都需要记录:

字段示例
版本号v1.2
变更日期2025-03-10
变更人PM + 算法
变更内容增加趋势对比, 优化语气
变更原因用户反馈总结缺少趋势分析
评测结果准确率 84% → 91%, 接受率 72% → 86%
状态全量上线 / A/B 中 / 已回滚
A/B 测试怎么做? ↓
(5)

Prompt 的 A/B 测试

不确定哪个 Prompt 更好? A/B 测试:

A 组 (50% 用户): Prompt v1.2 — 当前版本, 温暖友好风格.

B 组 (50% 用户): Prompt v1.3 — 新版本, 增加了结构化输出 (先数据 → 再分析 → 再建议).

跑 7 天, 看: 接受率谁高? 准确率有变化吗? 用户反馈有差异吗?

结果: B 组接受率 +8%, 准确率持平. → B 胜出, 全量发 v1.3.

PM 注意: Prompt A/B 测试跟传统 UI A/B 一样, 但观察的指标不同. 不看点击率, 看 "AI 输出的接受率 + 准确率". 先小流量 (5-10%), 再逐步放量.

灰度发布怎么做? ↓
(6)

灰度发布: 先给 5% 用户试

大版本更新 (v1.x → v2.0) 风险高. 灰度发布 = 逐步放量:

Day 1: 5% 用户用新 Prompt, 观察指标. ✔ 无退步 → 继续.

Day 3: 放量到 20%. ✔ 接受率 +3% → 继续.

Day 5: 放量到 50%. ✔ 稳定 → 继续.

Day 7: 全量 100%. ✔ 新版本正式上线.

任何一步出问题 → 立刻回滚到旧版. 这就是为什么需要版本管理: 回滚不是 "找到之前的飞书文档", 而是 "一键切回 v1.2".

PRD 里怎么写? ↓
(7)

PRD 中的 Prompt 管理模块怎么写

在 PRD 的 Prompt 管理章节, 写清楚这些:

1. 当前 Prompt 版本: v2.0 (附完整 Prompt 文本)

2. 变更审批流程: PM 起草 → 算法 Review → 评测集跑分 → PM+算法双签

3. 发布策略: 小修 (v1.x) 可直接发; 大改 (vX.0) 必须灰度 7 天

4. 回滚机制: 保留最近 5 个版本, 任意版本可一键回滚, 回滚后立即生效

5. A/B 测试规则: 每次有重大改动, 先 A/B 跑 7 天, 以接受率为主指标

注意: PM 不需要写 Prompt 的存储技术方案 (那是工程的事). 但 PM 必须定义 "谁有权改 Prompt" 和 "怎么验证改完是否更好".

    Vitamin 的完整方案 ↓
    (9)

    Vitamin 的 Prompt 管理规范

    ✓ 规范要求

    • 每个 AI 功能有独立的 Prompt, 独立版本号
    • 所有 Prompt 存储在配置中心, 不硬编码
    • 变更需 PM + 算法双签
    • 大版本灰度 7 天, 小修可热更新
    • 保留最近 10 个版本, 支持一键回滚

    ✗ 禁止事项

    • 工程师私自改 Prompt 不通知 PM
    • Prompt 只存在代码里, 不在配置中心
    • 改完 Prompt 不跑评测就上线
    • 没有回滚机制就做大版本更新
    • A/B 测试不到 7 天就下结论
    面试金句:

    "在 AI 产品中, Prompt 是产品代码. 我会在 PRD 里建立 Prompt 管理规范: 版本号、变更日志、PM+算法双签审批、A/B 测试、灰度发布、一键回滚. 每次 Prompt 变更都等同于一次产品发版, 必须有评测数据支持."

    总结 + 作业 ↓
    (10)

    一句话规则 + 课后作业

    一句话:

    Prompt 是产品代码 —
    每次改动都需要版本号、评测、回滚能力.

    改 Prompt ≈ 产品发版. 不评测就上线 = 不测试就发布.

    课后作业 (30 分钟):
    1. 为 Vitamin 的 "健康问答" 功能, 写一个 System Prompt (100-200 字), 标注为 v1.0.
    2. 修改 Prompt 的一个方面 (如语气/结构/约束), 标注为 v1.1, 写清楚变更记录.
    3. 设计一个 A/B 测试方案: v1.0 vs v1.1, 分别给 50% 用户, 跑 7 天, 主指标是什么?
    4. 去 PromptLayer 或 Humanloop 的官网看看, 了解专业的 Prompt 版本管理工具长什么样.

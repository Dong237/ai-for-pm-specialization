①

一个生活例子: 搜猫片

你在手机相册里搜 "猫":

相册里一共有 20 张猫的照片.

搜索结果返回了 10 张, 其中 8 张是猫, 2 张是狗.

Precision (精确率) = 找出来的里面有多少是对的 = 8/10 = 80%

Recall (召回率) = 该找出来的找到了多少 = 8/20 = 40%

Precision 高 = 结果可靠. Recall 高 = 不漏东西.

这两个数字来自一个 2×2 的表 ↓

口诀: TP = 找对了. FP = 找错了. FN = 漏了. TN = 正确没找.

Precision 的公式 ↓
③

Precision 详解

公式

Precision = TP / (TP + FP)

找出来的里面, 有多少是对的.

高 Precision = 结果可靠, 不瞎报. 低 Precision = 找出来的一半是噪音.

什么时候关注 Precision?

→ False Positive 代价高的场景. 比如: 医疗诊断 (误诊=恐慌), 内容审核 (误封=用户流失), 推荐系统 (推错=用户不信任).

Recall ↓
④

Recall 详解

公式

Recall = TP / (TP + FN)

该找出来的, 找到了多少.

高 Recall = 不漏东西. 低 Recall = 有重要信息被遗漏.

什么时候关注 Recall?

→ False Negative 代价高的场景. 比如: 医疗信息检索 (漏掉关键文档=给错建议), 安全检测 (漏掉攻击=被攻破), 癌症筛查 (漏诊=延误治疗).

Vitamin 应该更关注 Recall — 漏掉一条重要的经期注意事项, 比多返回一条无关信息更危险.

P 和 R 的矛盾 ↓
⑤

P 和 R 的跷跷板

提高 Precision 通常会降低 Recall, 反之亦然.

直觉:

🔸 想提高 Precision (更精准) → 只返回最确定的结果 → 很多边界上的正确结果被丢弃 → Recall 下降

🔸 想提高 Recall (不漏) → 把更多可能相关的结果都返回 → 混入更多噪音 → Precision 下降

类比: 安检. 灵敏度调高 (Recall 高) → 每个人都报警 → 误报率高 (Precision 低). 灵敏度调低 (Precision 高) → 偶尔漏掉 → Recall 低.

怎么平衡? F1 ↓
⑥

F1 Score: 一个数字总结两个

公式

F1 = 2 × P × R / (P + R)

调和平均数. 如果 P 或 R 任一个极低, F1 也会很低.

F1 不是简单平均 — 它惩罚极端不平衡. P=90% R=10% → F1=18%, 不是 50%.

什么时候用 F1?

当 Precision 和 Recall 同等重要时, 用 F1 做单一衡量. 实际中, 很多场景会偏重其中一个 (医疗偏 Recall, 推荐偏 Precision).

在 LLM / RAG 场景怎么用? ↓
⑦

在 RAG 场景的应用

Vitamin 用 RAG 检索医学知识库. P/R/F1 怎么算?

RAG Precision = 检索到的文档中, 真正相关的比例

→ "搜出 10 篇, 6 篇相关" = Precision 60%

RAG Recall = 所有相关文档中, 被检索到的比例

→ "知识库有 8 篇相关, 搜到了 6 篇" = Recall 75%

F1 = 2 × 0.6 × 0.75 / (0.6 + 0.75) = 66.7%

DeepEval 中的 RAG 指标:

🔸 Contextual Precision — 检索结果排序对不对 (相关的排前面)

🔸 Contextual Recall — 该检索的都检索到了吗

🔸 Faithfulness — 生成的答案忠于检索内容吗 (不瞎编)

Vitamin 的目标值 ↓
⑨

Vitamin 知识库评测目标

Vitamin RAG 评测目标:

🔸 Recall > 80% — 不能漏掉重要的医学信息. 医疗场景 Recall 优先.

🔸 Precision > 60% — 别塞太多无关内容 (否则 LLM 会被噪音干扰).

🔸 F1 > 65% — 综合指标的及格线.

🔸 Faithfulness > 90% — 生成的答案必须忠于检索到的内容, 不能瞎编.

(1)

"Bot 回答得不对" -- 然后呢?

你问 Bot "经期能跑步吗", 它说"可以每天跑 10 公里". 显然不对. 但问题出在哪?

可能的问题节点:

  • 意图识别错了 (把运动问题识别成闲聊)?
  • 知识库检索错了 (没检索到经期运动的内容)?
  • LLM 生成错了 (检索到了但没遵守)?
  • 插件返回错了 (天气数据格式不对)?

不打开调试面板, 你只看到最终结果, 不知道哪一步出了错.

调试面板长什么样? ↓
(2)

Coze 调试面板: 透视每一步

在 Bot 编辑页右栏, 发一条消息后, 点"查看详情" -- 你能看到每个工作流节点的输入和输出.

调试面板显示:

  • 每个节点的名称和类型
  • 每个节点接收到的输入数据
  • 每个节点产出的输出数据
  • 每个节点的执行时间 (延迟)
  • 执行状态: 成功 / 失败 / 超时

调试出问题了怎么修? ↓
(4)

常见调试 3 板斧

板斧 1

改 Prompt

如果意图识别不准, 改 LLM 节点的 Prompt -- 加更多示例, 或者加 Few-shot.
板斧 2

调知识库

如果检索结果不相关, 检查知识库内容质量, 或者调整切片大小和检索 Top-K 参数.
板斧 3

换模型

如果生成质量差, 试试换一个更强的模型. 或者降低 Temperature 让回答更稳定.

调试循环: 改一处 -> 测 5 个 case -> 看结果 -> 再改. 通常需要 3-5 轮才能调到满意.

调试完了, 怎么发布? ↓
(5)

发布: 从调试到上线

发布步骤:

  1. 确认调试效果满意
  2. 点右上角"发布"按钮
  3. 填写发布说明 (非必须, 但建议写)
  4. 选择发布渠道
  5. 点击"发布" -- 审核通常 1 分钟内完成

发布后要注意什么? ↓
(7)

版本管理 + 灰度发布

版本管理:

  • 每次发布自动生成版本号
  • 可以回滚到任何历史版本
  • 发布记录里写清楚"这次改了什么" -- 跟代码 commit 一样

灰度发布思路:

  • 先发到 Web SDK, 自己测 + 找 3-5 个朋友测
  • 收集反馈, 改 Prompt / 知识库
  • 测试满意后, 再发到微信公众号面向更大用户群

这就是产品上线的标准流程: 内测 -> 小流量 -> 全量.

发布后的数据怎么看? ↓
(8)

发布后: 看数据

Coze 提供的使用数据:

  • 对话次数 -- 多少人在用你的 Bot
  • 消息数 -- 每个用户平均聊几轮
  • 用户留存 -- 明天还来的人有多少
  • 用户反馈 -- 点赞 / 点踩 比例

PM 视角: 这些数据就是 M2.3 学的 AI 产品指标! 对话次数 = 使用量, 消息数 = 粘性, 点踩率 = 质量信号. 现在你有真实数据来验证了.

Vitamin 视角 ↓
(9)

Vitamin 调试 + 发布 Checklist

调试 Checklist (发布前):

  • 5 种身体状态各测 3 个问题, 确认回答质量
  • 边界 case: 问医疗诊断, 确认拒答
  • 边界 case: 问无关话题, 确认不跑偏
  • 知识库检索: 确认来源标注正确
  • 插件调用: 确认天气/计算结果正确

发布 Checklist:

  • 先发 Web SDK, 自己测 + 3 个朋友测
  • 收集 10 条真实反馈
  • 根据反馈改 2-3 处 Prompt
  • 再次测试确认
  • 发到微信公众号 (或保留 Web 链接用于面试)

一句话规则 + 今晚作业

一句话

调试 = 逐步检查每个节点. 发布 = 先内测再灰度再全量.

出了问题, 调试面板帮你定位是哪个节点的锅. 别猜, 看数据.

课后作业 (今晚 30 分钟):
  1. 打开你的 Coze Bot, 用右栏调试面板测试 5 个问题. 对每个问题, 查看每个节点的输入输出.
  2. 找到 1 个回答不满意的 case, 用"3 板斧"定位并修复问题.
  3. 把 Bot 发布到 Web SDK, 获取一个分享链接.
  4. 把链接发给 1 个朋友, 让 ta 问 3 个问题, 记录反馈.

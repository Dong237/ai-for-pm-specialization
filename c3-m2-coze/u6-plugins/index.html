插件: 给 Bot 装上手脚

Bot 能聊天、能查知识库, 但还不能"做事". 今天给它接上外部工具.

用户问"今天适合运动吗", Bot 不知道天气...

用户想算 BMI, Bot 算数不靠谱...

-- 插件来解决.

10 步学完, 你会:
  • 浏览 Coze 官方插件商店并安装插件
  • 理解插件 = API 工具的封装
  • 知道如何创建自定义插件
  • 规划 Vitamin Bot 需要哪些插件
走起 →
(1)

LLM 的"手脚"问题

LLM 擅长"想" (文本生成), 但不擅长"做" (获取实时数据、精确计算、操作外部系统). 插件就是它的手脚.

M1.1 就讲过: LLM 不能算 BMI (精确计算), 不能查今天天气 (实时数据). 但接上插件后, 这些都能做了.

插件 = 让 LLM 调用外部工具的标准接口.

插件 API

调用流程:

  1. 用户提问 -> Bot 把问题传给 LLM
  2. LLM 判断: "这个问题需要查天气" -> 调用天气插件
  3. 插件返回天气数据 -> LLM 结合数据生成回复
  4. 回复返回给用户

LLM 自己决定什么时候调用什么插件 -- 这叫 Function Calling.

自定义插件怎么做? ↓
(5)

创建自定义插件: 4 步

步骤:

  1. 新建插件: 在 Coze 工作空间点"插件" -> "创建插件"
  2. 添加工具: 每个工具 = 一个 API 接口. 填名称、描述、HTTP 方法
  3. 配参数: 定义输入参数 (如 city: string) 和输出参数 (如 temperature: number)
  4. 调试测试: 用调试控制台发请求, 确认 API 能正常返回

PM 注意: 你不需要自己写 API. 只需要知道 API 的 URL、参数格式、返回格式 -- 这些信息找研发要就行. PM 的工作是选择哪些 API 该接、怎么跟产品流程结合.

插件在工作流中怎么用? ↓
(6)

插件 + 工作流 = 完整能力

在工作流中使用插件:

  • 拖一个"插件节点"到工作流画布
  • 选择要调用的插件和工具
  • 配置输入参数 (可以引用上一个节点的输出)
  • 插件的输出可以传给下一个节点 (如 LLM 节点)

Vitamin 示例: 意图 = exercise 时, 工作流走: 意图识别 -> 天气插件 -> 知识库检索 -> LLM 生成. LLM 同时拿到天气数据 + 运动知识, 生成"今天 32 度, 建议室内运动, 经期建议瑜伽".

插件安全要注意什么? ↓
(7)

插件安全: 3 条红线

红线 1

API Key 不能暴露

自定义插件的 API Key 要放在 Coze 的安全配置里, 不能硬编码在插件参数中.
红线 2

用户数据不能泄露

插件返回的数据可能包含敏感信息. 确保 LLM 不会在回复中"复述"不该给用户看的字段.
红线 3

权限要最小化

插件只给它需要的权限. 天气插件不需要写权限, 日历插件才需要. 原则: 能读不写.
插件的性能影响? ↓
(8)

性能考量: 插件会拖慢回复

关键认知: 每调用一个插件, 回复时间 +0.5~2 秒. 如果一个工作流调了 3 个插件, 用户可能要等 5 秒以上.

优化策略:

  • 只在必要时调用插件 (条件分支控制)
  • 能缓存的数据就缓存 (天气不用每次都查)
  • 并行调用不依赖的插件 (工作流支持并行分支)
Vitamin 的插件规划 ↓
(9)

Vitamin Bot 插件规划

插件用途触发条件
天气查询运动建议需要天气数据意图 = exercise
计算器BMI / 热量计算用户提供体重身高
网页搜索知识库没有的最新信息知识库检索为空时
日历提醒经期/服药提醒用户要求设置提醒

面试金句: "Vitamin Bot 的插件调用是按需触发, 不是每次都调. 通过工作流的条件分支, 只有运动类问题才查天气, 只有计算类需求才调计算器. 这既保证了功能完整, 又控制了响应时间和成本."
